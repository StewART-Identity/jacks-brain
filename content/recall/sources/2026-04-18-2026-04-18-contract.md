Architecture and conventions document for UNT IAM provisioning scripts. Intended as context for any human or AI assistant integrating, migrating, or creating scripts within the [[recall/entities/iam-modules]] infrastructure. All patterns described have been validated against production eDirectory, Active Directory, Entra ID, and Oracle database infrastructure serving 72,000+ users.

## Repository Layout

The IAM scripting infrastructure uses two types of repositories with a symlink relationship:

**Standalone module repo** тАФ `iam-modules/` (hyphen in repo name):
- Contains the shared Python library: `config.py`, `connections.py`, `edir_client.py`, `ad_client.py`, `gc_client.py`, `graph_client.py`, `logging_utils.py`, `cli_utils.py`, `idm_utils.py`
- Single `.env` file as the source of truth for all credentials
- `__init__.py` exposes `load_config`, `setup_logging`, `set_operation`, `run`, and constants

**Consuming repos** тАФ symlink `iam_modules` (underscore) тЖТ `../iam-modules/`:
- `iam-scripts-provisioning/` тАФ eDirectory and EIS provisioning scripts; `lib/hrpd.py`, `lib/lspd.py` for database logic
- `iam-scripts-utilities/` тАФ batch, cron, group management, mail, user lifecycle
- `iam-scripts-alma-v2/` тАФ batch account lifecycle scripts
- `iam-scripts-drip/` тАФ LDIF record drip import scripts

The symlink **must use an underscore** (`iam_modules`) because Python cannot import names containing hyphens.

## Key Architectural Rules

1. Scripts are always **exactly one directory level below the repo root** тАФ no nested subdirectories.
2. All authentication, configuration, and connection management goes through `iam_modules`. No script defines its own connection setup, TLS configuration, logging, or `.env` loading.
3. **Business logic stays in `lib/`** тАФ SQL queries, domain rules, and data transformation belong in the consuming repo's `lib/` directory, not in `iam_modules`. The module gives a connection; `lib/` knows what to do with it.
4. The `.env` file lives in the `iam-modules` repo directory. `load_config()` uses `os.path.realpath()` to resolve through symlinks.

## sys.path Preamble

Every script that imports from `iam_modules` must include this exact preamble before any `from iam_modules import ...` statements:

```python
import os
import sys
from pathlib import Path

_script_dir = Path(os.path.realpath(__file__)).parent
_repo_root = _script_dir.parent
if (_repo_root / 'iam_modules').exists():
    sys.path.insert(0, str(_repo_root))
```

## Mandatory Conventions

See [[recall/concepts/dry-run-by-default]] for the `--commit` flag pattern. Additional mandatory conventions:

- **Argparse for all scripts** тАФ provides `--help` automatically; include description and dry-run epilog
- **Shebang**: `#!/usr/local/iamadm/miniconda3/envs/scripts/bin/python`
- **Logging**: always use `setup_logging('script_name')` from `iam_modules` тАФ writes to `/var/log/iam/script_name_YYYYMMDD.log`
- **Script naming taxonomy**: `send_`, `invoke_`, `sync_`, `get_`, `repair_`, `restore_`, `clear_`
- **Ctrl+C handling**: automatic on import; scripts call `set_operation()` to describe current phase; entry point is `run(main)`, never `sys.exit(main())`

## Connection Functions Reference

All connections use `raise_exceptions=True` тАФ failed LDAP operations raise `LDAPException` rather than returning a result code. Scripts must use `try/except`, not `conn.result` checks.

| Function | Target | Notes |
|----------|--------|-------|
| `connect_to_edir(config, env, logger)` | eDirectory | `env` = `'test'` or `'prod'`; TLSv1.2 pinned |
| `connect_to_idtree(config, logger)` | eDirectory prod | Legacy alias тАФ delegates to `connect_to_edir(..., 'prod', ...)` |
| `connect_to_ad(config, domain, logger)` | Active Directory | `domain` = `'ad'`, `'hsc'`, `'students'`, or `'unt'` |
| `connect_to_gc(config, logger)` | AD Global Catalog | Port 3269; forest-wide read-only |
| `connect_to_hrdb(config, logger)` | Oracle HRPD | PeopleSoft HR database |
| `connect_to_lsdb(config, logger)` | Oracle LSPD | PeopleSoft LS database |

## Client Classes (Higher-Level API)

For scripts needing group membership management, user resolution, or dry-run-aware attribute modification:

- **`EDirectoryClient`** тАФ `find_user_by_uid()`, `find_user_by_workforce_id()`, `find_user_by_email()`, `modify_attribute(..., dry_run=)`. Supports explicit `environment='prod'` or legacy config dict.
- **`ADDomainControllerClient`** тАФ `add_group_member()`, `remove_group_member()` with `dry_run=` support
- **`ADGlobalCatalogClient`** тАФ `find_group()`, `find_user_by_upn()`, `find_user_by_employee_id()`, `get_group_members()`
- **`GraphAPIClient`** тАФ Entra ID / Microsoft Graph: `get_group_by_name()`, `get_group_members()`, `add_group_member()` with `dry_run=` support

## Configuration Reference

`load_config(__file__)` returns a dict keyed by service:

| Key | Source Env Vars |
|-----|----------------|
| `config['edir_prod']` / `config['edir']` | `EDIR_PROD_*` / `EDIR_*` (legacy alias) |
| `config['edir_test']` | `EDIR_TEST_*` |
| `config['ad_unt']`, `config['ad_hsc']`, `config['ad_students']`, `config['ad_ad']` | `AD_UNT_*`, etc. |
| `config['gc']` | `AD_GC_*` |
| `config['entra']` | `ENTRA_*` (tenant_id, client_id, client_secret) |
| `config['duo']` | `DUO_*` (api_hostname, client_id, client_secret) |
| `config['hr_db']` | `HR_DB_*` |
| `config['ls_db']` | `LS_DB_*` |
| `config['smtp']` | `SMTP_*` |
| `config['logging']` | `LOG_*` |

Application-specific env vars (`BATCH_*`, `DEACTIVATE_*`, `VPN_*`, etc.) are **not** loaded by `load_config()` тАФ scripts read them via `os.getenv()` after calling `load_config()` (which calls `load_dotenv()`).

## Migration Patterns

Six named patterns for converting legacy scripts:

- **Pattern A** тАФ Replace inline eDirectory TLS/connection setup with `connect_to_edir()`
- **Pattern B** тАФ Replace inline logging setup with `setup_logging()`
- **Pattern C** тАФ Replace inline AD connection setup with `connect_to_ad()`
- **Pattern D** тАФ Replace custom `LDAPConnectionManager` class with a dict of `connect_to_ad()` calls
- **Pattern E** тАФ Replace hardcoded SMTP constants with `config['smtp']` values
- **Pattern F** тАФ Replace `cx_Oracle` connection code with `connect_to_hrdb()` / `connect_to_lsdb()`

## Service Account Permissions

Each directory maintains least-privilege service accounts. See [[recall/concepts/iam-scripting-architecture]] for the layered account model.

**AD service accounts** (one per domain, no admin group membership required):

| Domain | Account |
|--------|---------|
| Forest root + GC | `CN=IAMScriptAD,OU=IAM,OU=Services,DC=ad,DC=unt,DC=edu` |
| UNT | `CN=IAMScriptUNT,OU=IAM,OU=Services,DC=unt,DC=ad,DC=unt,DC=edu` |
| HSC | `CN=IAMScriptHSC,OU=IAM,OU=Services,DC=hsc,DC=ad,DC=unt,DC=edu` |
| STUDENTS | `CN=IAMScriptSTU,OU=IAM,OU=Services,DC=students,DC=ad,DC=unt,DC=edu` |

Key AD attributes written: `member` (group), `description` (user), `employeeNumber` (user), `extensionAttribute15`, `gidNumber`, `userAccountControl`.

**eDirectory service accounts**: `cn=IAMScript,ou=IAMApplications,o=unt` (general) and `cn=IAMDrip,ou=IAMApplications,o=unt` (LDIF imports, requires write to arbitrary attributes on `ou=people,o=unt`).

Key eDirectory attributes written: `loginDisabled`, `groupMembership`, `mail`, `prefmail`, `DirXML-Associations`, `untAccountHistory`, `employeeMail`, `studentMail`, `msxmail`.

## Environment Details

- **Server**: `iam-script-gab` (production IAM scripting host)
- **Python**: Miniconda3 at `/usr/local/iamadm/miniconda3/envs/scripts/`
- **User**: `iamadm` (group: `iamteam`)
- **eDirectory**: `idmpidv01.untsystem.edu:637` (SSL, TLSv1.2 only)
- **AD domains**: `ad.unt.edu`, `hsc.ad.unt.edu`, `students.ad.unt.edu`, `unt.ad.unt.edu`
- **Global Catalog**: Port 3269, `dcpd-ad-pri-01.ad.unt.edu`
- **Oracle HR (HRPD)**: `oradb404.its.unt.edu:1521/hrpd.its.unt.edu`
- **Oracle LS (LSPD)**: `oradb405.its.unt.edu:1521/lspd.its.unt.edu`
