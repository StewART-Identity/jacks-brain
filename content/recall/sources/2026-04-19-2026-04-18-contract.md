Architecture and conventions document for [[recall/entities/unt-system]] IAM provisioning scripts. Serves as context for any human or AI assistant integrating, migrating, or creating scripts within the [[recall/entities/iam-modules]] infrastructure. Validated against production eDirectory, Active Directory, Entra ID, and Oracle database infrastructure serving 72,000+ users.

> The original ingest (2026-04-18) is at [[recall/sources/2026-04-18-2026-04-18-contract]]. This entry records a second read on 2026-04-19. It focuses on three areas that the first read underemphasized: the explicit anti-pattern codex, the per-script attribute delegation matrix as a governance mechanism, and the env var naming migration hazards.

## The Anti-Pattern Codex

The CONTRACT's "What NOT to Do" section is unusual: it is a 15-rule codified anti-pattern list embedded in an architecture document. Most architecture documents describe what to do; this one also explicitly names what to avoid, and why. This is as much a code review checklist as a design reference.

Key anti-patterns that define the boundaries of [[recall/concepts/iam-scripting-architecture]]:

| Anti-Pattern | Correct Approach |
|-------------|-----------------|
| Define TLS config in a script | TLS lives only in `connections.py` and `edir_client.py` |
| Call `load_dotenv()` directly | Use `load_config(__file__)` |
| Hardcode server names, ports, or credentials | All come from `.env` via config |
| Default to live mode | Default is always dry-run; [[recall/concepts/dry-run-by-default]] |
| Create subdirectories inside category directories | Scripts are one level deep, always |
| Use `--dry-run` as a flag | Flag is `--commit`; absence means dry-run |
| Import from `iam-modules` (hyphen) | Import is always `from iam_modules` (underscore) |
| Use `cx_Oracle` | Use `oracledb` (the official successor library) |
| Put SQL in `iam_modules` | SQL and business logic go in `lib/` in consuming repo |
| Put database business logic in scripts | Goes in `lib/hrpd.py` or `lib/lspd.py` for reuse |
| Allow Python tracebacks on Ctrl+C | Use `run(main)` + `set_operation()`; see [[recall/concepts/graceful-interrupt-handling]] |
| Use `sys.exit(main())` as entry point | Use `run(main)` for `KeyboardInterrupt` protection |
| Put app-specific env vars in `iam_modules` config | `BATCH_*`, `DEACTIVATE_*`, `VPN_*` etc. stay in scripts, read via `os.getenv()` |
| Use old env var names | `*_SERVER` → `*_HOSTNAME`; `HRDB_*` → `HR_DB_*`; `LSDB_*` → `LS_DB_*`; `SMTP_HOST` → `SMTP_HOSTNAME` |
| Check `conn.result['result']` after LDAP writes | All connections use `raise_exceptions=True`; use `try/except LDAPException` |

## Env Var Naming Migration Hazards

On re-read, a notable operational risk: the CONTRACT documents renamed env var keys that could silently break scripts not yet migrated. Old names that are no longer canonical:

- `*_SERVER` → `*_HOSTNAME` (e.g., `EDIR_SERVER` → `EDIR_HOSTNAME`)
- `HRDB_*` → `HR_DB_*`
- `LSDB_*` → `LS_DB_*`
- `HR_DB_CONNECTSTRING` → `HR_DB_CONNECT_STRING`
- `SMTP_HOST` → `SMTP_HOSTNAME`

Scripts using the old names will read `None` from `os.getenv()` without error, producing silent connection failures. The `.env.example` file in the `iam-modules` repo is the canonical reference.

## Service Account Attribute Delegation Matrix

The CONTRACT's service account section is structurally distinct from typical permission documentation: it cross-references every LDAP attribute written with the specific script that writes it. This functions as a **living permission audit trail** — when a new script writes to a new attribute, the CONTRACT requires updating the table.

**STUDENTS domain exception**: The `deactivate_terminateduser` and `reactivate_terminateduser` scripts skip the STUDENTS domain by default. The `--students` flag must be passed explicitly to include it. This is a deliberate risk management decision: the STUDENTS domain has a different population (student accounts) from the main UNT and HSC domains, and inadvertent deactivation at scale has different consequences.

**IAMDrip exception**: The `cn=IAMDrip` service account (used by `drip_ldifrecords.py`) requires write access to **arbitrary attributes** on user objects in `ou=people,o=unt` — the only account in the infrastructure with this scope. All other accounts have attribute-specific delegations.

## Application-Specific Env Var Ownership

A clear governance boundary: `load_config()` in [[recall/entities/iam-modules]] owns the infrastructure env vars (directory credentials, database DSNs, SMTP settings, log config). It explicitly does NOT load application-specific variables like `BATCH_*`, `DEACTIVATE_*`, `REACTIVATE_*`, or `VPN_*`.

Because `load_config()` calls `load_dotenv()` internally, those application-specific variables become accessible via `os.getenv()` after any `load_config()` call — but they are owned by the individual scripts, not the module. This keeps `iam_modules` from accumulating knowledge of each consuming repo's business logic.

## Migration Checklist Format

The CONTRACT includes a 17-step migration checklist for converting legacy scripts. This is an operational runbook format embedded in the architecture document:

1. Add sys.path preamble (copy exactly)
2. Replace `.env` loading with `load_config(__file__)`
3. Replace logging setup with `setup_logging('script_name')`
4. Replace LDAP connection code with appropriate `connect_to_*()` function
5. Replace Oracle connection code with `connect_to_hrdb()` / `connect_to_lsdb()`
6. Replace hardcoded SMTP/email/log constants with `iam_modules` imports
7. Replace `--dry-run` flag with `--commit`; default must be dry-run
8. Ensure `argparse` is used
9. Ensure shebang is `#!/usr/local/iamadm/miniconda3/envs/scripts/bin/python`
10. Remove all `import ssl`, direct `Tls()` construction, and `load_dotenv()` calls
11. Remove `import cx_Oracle`; replace with `from iam_modules.connections import ...`
12. Remove any in-script `connect_to_idtree()` or `connect_to_edir()` definitions
13. Remove any `LDAPConnectionManager` class definitions
14. Add `set_operation()` calls at each processing phase
15. Replace `conn.result['result']` checks with `try/except LDAPException`
16. Use `run(main)` as entry point (not `sys.exit(main())`)
17. Test with `python script_name.py --verbose` (dry-run) before committing

## Read-Only Scripts Catalog

The CONTRACT explicitly catalogs scripts that connect to directories but perform **no writes**. This is useful for permission scoping: these scripts need only search/read access, not write delegation.

**AD read-only**: `compare_adtoentragroup.py`, `count_groupmembers.py`, `get_aduser.py`, `get_groupmember.py`, `get_uachieveinfo.py`, `start_idmsyncuserevent.py`, `sync_idmusermailattrs.py`

**eDirectory read-only**: `export_vpndata.py`, `find_nonuntemailaddress.py`, `test_logindisabled.py`, `update_uachieveusers.py`, `get_mailaddress.py`, `get_usermailattrs.py`, `get_ediruser.py`, `get_nonstudents.py`, `get_userattributes.py`, `search_euid.py`, `search_untdeptattribute.py`
