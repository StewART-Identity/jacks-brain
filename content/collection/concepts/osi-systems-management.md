**OSI Systems Management** is the standardized framework for managing networked systems within the OSI reference model. It defines a manager/agent architecture, a managed object model, a Management Information Base (MIB), and application-layer protocols for carrying management operations. The primary application protocol is **CMIP** (Common Management Information Protocol, X.711), carried over the **CMIS** (Common Management Information Service, X.710).

X.530 uses OSI Systems Management as the management substrate for [[collection/concepts/x500-directory-management|X.500 Directory administration]] — DSA operational state, performance counters, fault notifications, and security audit trails are all managed through this framework.

## Manager/Agent Architecture

OSI systems management uses a **manager/agent** model:

- **Manager** — The managing system. Issues management operations (Get, Set, Action, Create, Delete, Event Report) to agents. Receives notifications (event reports) from agents.
- **Agent** — The managed system. Controls one or more managed object instances. Executes operations on behalf of the manager and generates event reports in response to state changes.

A single system may act as both manager and agent simultaneously. The manager/agent relationship is many-to-many: one manager may control many agents; one agent may serve many managers.

This contrasts with [[collection/concepts/ldap|LDAP]]'s client/server model, where management is not standardized — LDAP servers expose no protocol-level management objects.

## Managed Objects

The fundamental unit of management is the **managed object** — an abstraction of a manageable resource. A managed object is defined by:

- **Object Class** — Specifies the allowable attributes, notifications, and actions for objects of that class. Object classes form an inheritance hierarchy.
- **Attributes** — Properties of the managed object that can be read or set (e.g., operational state, counters, policy settings).
- **Notifications** — Event reports generated by the agent when state changes occur (e.g., state change, operational violation, security violation).
- **Actions** — Operations the manager can invoke on the object (e.g., reset a counter, initiate a shadow update).
- **Packages** — Named groups of attributes, notifications, and actions, optionally conditional on implementation choices.

Managed objects are defined using the **Guidelines for Definition of Managed Objects (GDMO)** notation (X.722), a formal language for specifying managed object classes, packages, name bindings, and notification parameters. GDMO definitions are paired with [[collection/concepts/asn1|ASN.1]] type definitions for attribute syntaxes and parameter encodings.

## Management Information Base (MIB)

The **Management Information Base (MIB)** is the collection of managed object instances in a managed system. The MIB is organized as a hierarchical naming tree — a separate namespace from the [[collection/concepts/directory-information-tree|DIT]] — in which each managed object instance is named by a sequence of arcs called a **name binding**. Name bindings specify which managed object classes are subordinate to which others (e.g., DSA objects are subordinate to DMD objects in [[collection/concepts/x500-directory-management|X.500 directory management]]).

## CMIP Operations

CMIP defines the following management operations:

| Operation | Direction | Effect |
|-----------|-----------|--------|
| **Get** | Manager → Agent | Read attribute values |
| **Set** | Manager → Agent | Write attribute values |
| **Action** | Manager → Agent | Invoke a named action on a managed object |
| **Create** | Manager → Agent | Instantiate a new managed object |
| **Delete** | Manager → Agent | Remove a managed object instance |
| **Event Report** | Agent → Manager | Report an event (notification) |

CMIP runs over the OSI application layer using [[collection/concepts/acse|ACSE]] for association establishment and [[collection/concepts/rose-remote-operations|ROSE]] for the request/response framework — the same underlying services used by X.500's own DAP and DSP protocols.

## Standard Notification Types

X.721 defines reusable notification types that managed object packages can reference:

| Notification | Meaning |
|-------------|---------|
| `stateChange` | Managed object changed operational or administrative state |
| `operationalViolation` | Protocol violation or operational error detected |
| `processingErrorAlarm` | Internal processing error (critical extension unavailable, resource exhausted) |
| `securityServiceOrMechanismViolation` | Authentication failure or access control failure |
| `communicationsAlarm` | Abnormal communication termination |
| `communicationsInformation` | Informational PDU trace (request/response logging) |

[[collection/concepts/x500-directory-management|X.500 directory management]] uses all six of these notification types: state change for DSA start/stop, operational violation for name/service/attribute errors, processing error alarm for critical extension and loop detection faults, security violation for authentication and access control failures, communications alarm for abnormal association termination, and communications information for PDU-level audit trails.

## TMN Integration

The **Telecommunications Management Network (TMN)** framework, defined in ITU-T M.3010, provides an organizational model for how OSI management is deployed across networks. TMN describes a five-layer hierarchy (Network Element → Network Element Management → Network Management → Service Management → Business Management) that X.530 adapts directly for [[collection/concepts/x500-directory-management|Directory management]].

The TMN alignment means X.500 management is positioned as part of a broader integrated management framework — the same manager workstations and CMIP infrastructure used to manage telecom switches could, in principle, manage X.500 directory components. This was architecturally coherent with the OSI-world vision but was never widely realized in LDAP-based deployments, which have no equivalent management layer standard.

## Contrast with LDAP Management

[[collection/concepts/ldap|LDAP]] defines no management protocol. Each LDAP server vendor exposes management through proprietary mechanisms: command-line tools, web consoles, SNMP MIBs, or vendor-specific extended operations. No IETF standard defines managed objects, event notifications, or manager/agent protocol for LDAP servers. See [[collection/synthesis/x500-directory-management-vs-ldap-administration|X.500 Management vs. LDAP Administration]] for the full comparison.
