**Recommendation ITU-T X.509 (2019) – Technical Corrigendum 2** (approved October 2023, [[collection/entities/itu-t|ITU-T]] Study Group 17) corrects two defect reports — DRs 434 and 435 — against the 9th edition of [[collection/concepts/x509-pki|X.509]]. It is co-published as ISO/IEC 9594-8:2020/Cor.2:2023. DR 434 retrofits complete LDAP integration into the Privilege Management Infrastructure (PMI) attribute types, object classes, and matching rules — adding a new clause 19.4 with seven SYNTAX-NAME definitions. DR 435 introduces the `noRevAvail` certificate extension for environments where revocation is unnecessary, such as short-lived certificate deployments.

## Defect Report 434: PMI LDAP Syntax Definitions

DR 434 addresses incomplete LDAP integration in the X.509 PMI (Privilege Management Infrastructure) — the attribute certificate framework added in the 4th edition (2000). Many of the PMI attribute types and object classes lacked `LDAP-SYNTAX`, `LDAP-NAME`, and `LDAP-DESC` fields in their [[collection/concepts/asn1|ASN.1]] definitions, making them non-interoperable with [[collection/concepts/ldap|LDAP]] directories.

### Corrected Matching Rules

The `dualStringMatch` matching rule (§16.8.2) and both attribute certificate matching rules (§19.3.1–19.3.2) are updated to include `LDAP-SYNTAX` and `LDAP-NAME` fields:

- **`dualStringMatch`** — syntax: `ldapDualStringSyntax.&id`, LDAP-NAME: `{"permission"}`
- **`attributeCertificateExactMatch`** — syntax: `attCertExactAssertion.&id`, LDAP-NAME: `{"attributeCertificateExactMatch"}`
- **`attributeCertificateMatch`** — syntax: `attCertAssertion.&id`, LDAP-NAME: `{"attributeCertificateMatch"}`

### New Clause 19.4: PMI Directory Syntax Definitions

The most structurally significant addition is a new clause **19.4** — "PMI directory syntax definitions" — containing seven SYNTAX-NAME information objects that map X.509 PMI data types to LDAP syntaxes. Values in all seven syntaxes are encoded using DER (see also RFC 4522, binary encoding option):

| Syntax name | Section | LDAP-DESC | X.509 type mapped |
|---|---|---|---|
| `ldapRoleSyntax` | 19.4.1 | "LDAP RoleSyntax" | `RoleSyntax` |
| `ldapDualStringSyntax` | 19.4.2 | "LDAP DualStringSyntax" | `DualStringSyntax` |
| `x509AttributeCertificate` | 19.4.3 | "X.509 AttributeCertificate" | `AttributeCertificate` |
| `ldapAttCertPath` | 19.4.4 | "LDAP AttCertPath" | `AttCertPath` |
| `ldapPolicySyntax` | 19.4.5 | "LDAP Policy syntax" | `PolicySyntax` |
| `attCertExactAssertion` | 19.4.6 | "Attribute Certificate Exact Match" | `AttributeCertificateExactAssertion` |
| `attCertAssertion` | 19.4.7 | "Attribute Certificate Match" | `AttributeCertificateAssertion` |

The `attCertExactAssertion` and `attCertAssertion` syntaxes use the Generic String Encoding Rules (GSER) specified in IETF RFC 3641 — newly added as a normative reference. RFC 4522 (LDAP binary encoding option) is added to the bibliography.

### ASN.1 Module Changes

The Annex A `AttributeCertificateDefinitions` module receives:
- Seven new OID assignments (`id-asx 13` through `id-asx 19`) for the new syntaxes
- `id-asx` added to the IMPORTS from the `usefulDefinitions` module

## Defect Report 435: noRevAvail Extension

DR 435 introduces a new certificate extension — **`noRevAvail`** — for environments where revocation status information is intentionally not provided.

### New Clause 9.6.2.7

```asn1
noRevAvail EXTENSION ::= {
    SYNTAX      NULL
    IDENTIFIED BY id-ce-noRevAvail }
```

**Applicability:**
- May be present in end-entity public-key certificates issued by CAs
- May be present in attribute certificates issued by AAs
- **SHALL NOT** be present in CA or AA certificates
- **SHALL always be flagged as non-critical**

**Effect on relying parties:**
- If present in a public-key certificate, a relying party need not seek revocation status information
- If present in an attribute certificate, a privilege verifier need not seek revocation status information

This extension addresses use cases such as certificate issuance with very short validity periods (hours to days), where the certificate's lifetime is shorter than the revocation propagation delay — making CRL or OCSP checks effectively unnecessary. In these environments, the CA signals this intent explicitly rather than leaving relying parties to discover the absence of CRL distribution points.

The correction also updates §17.2.2.7 to clarify the scope of `noRevAvail` in the attribute certificate context: it may appear in attribute certificates issued to PMI end entities (including SOA-issued certificates), but not in public-key certificates or attribute certificates issued to AAs.

## Relationship to Existing Wiki Content

This corrigendum continues the standards maintenance pattern documented in [[collection/synthesis/x500-standards-maintenance-and-ldap-inheritance|X.500 standards maintenance and LDAP inheritance]]. The category of correction — filling in LDAP mapping fields that were omitted from PMI definitions — parallels the "prose → machine-checkable ASN.1" fixes that recurred throughout the DIG v15 defect series. The PMI LDAP syntax additions make the [[collection/concepts/x509-pki|X.509 PKI]] attribute certificate framework fully interoperable with LDAP directories at the normative level. The `noRevAvail` extension reflects an evolution in PKI deployment practice toward short-lived certificates — increasingly common in modern certificate issuance infrastructure where revocation is replaced by natural expiry.
