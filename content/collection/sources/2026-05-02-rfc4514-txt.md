RFC 4514 (June 2006) defines the string representation of X.500 [[collection/concepts/distinguished-name|Distinguished Names]] (DNs) used in the Lightweight Directory Access Protocol. It is edited by [[collection/entities/kurt-zeilenga|K. Zeilenga]] ([[collection/entities/openldap-foundation|OpenLDAP Foundation]]) and supersedes [[collection/sources/2026-05-02-rfc2253-txt|RFC 2253]] (1997) as part of the [[collection/synthesis/ldap-technical-specification-architecture|RFC 4510 modular reorganization]]. The acknowledgements credit the original RFC 2253 authors: [[collection/entities/mark-wahl|M. Wahl]], [[collection/entities/tim-howes|T. Howes]], and [[collection/entities/steve-kille|S. Kille]].

## Purpose

Distinguished Names are the primary keys to entries in X.500 and LDAP directories. At the X.500 Directory Access Protocol (DAP) level, DNs are ASN.1-encoded using BER. LDAP transfers DNs as UTF-8 strings. RFC 4514 defines the canonical string format for this transfer. The primary goal is ease of encoding and decoding; human readability is secondary. The RFC explicitly notes it does **not** define a canonical string representation — DN equality comparison is done via the `distinguishedNameMatch` matching rule defined in RFC 4517, not by string comparison.

## DN Structure (ASN.1 Background)

From X.501, the ASN.1 structure is:

```
DistinguishedName         ::= RDNSequence
RDNSequence               ::= SEQUENCE OF RelativeDistinguishedName
RelativeDistinguishedName ::= SET SIZE (1..MAX) OF AttributeTypeAndValue
AttributeTypeAndValue     ::= SEQUENCE { type AttributeType, value AttributeValue }
```

An empty RDNSequence produces the empty string. Otherwise, the RDN components are encoded in reverse order (last to first, i.e., most general to most specific), then joined with commas.

## Encoding Algorithm (ASN.1 → String)

RFC 4514 specifies the RECOMMENDED algorithm, but permits alternative algorithms as long as they produce strings conforming to the Section 3 grammar.

### RDNSequence

Components are encoded from **last to first**, separated by commas. This means `UID=jsmith,DC=example,DC=net` reads left-to-right from most-specific (UID) to most-general (DC=net).

### RelativeDistinguishedName

For multi-valued RDNs, each AttributeTypeAndValue is encoded in any order, separated by `+`. Example: `OU=Sales+CN=J. Smith`.

### AttributeTypeAndValue

Encoded as `type=value`. The attribute type uses its registered short name (descriptor) if known:

| Short Name | X.500 AttributeType | OID |
|------------|---------------------|-----|
| CN | commonName | 2.5.4.3 |
| L | localityName | 2.5.4.7 |
| ST | stateOrProvinceName | 2.5.4.8 |
| O | organizationName | 2.5.4.10 |
| OU | organizationalUnitName | 2.5.4.11 |
| C | countryName | 2.5.4.6 |
| STREET | streetAddress | 2.5.4.9 |
| DC | domainComponent | 0.9.2342.19200300.100.1.25 |
| UID | userId | 0.9.2342.19200300.100.1.1 |

If no short name is registered, the OID dotted-decimal form is used (e.g., `1.3.6.1.4.1.1466.0`).

### AttributeValue Encoding

Two encoding modes:

1. **Hex BER form** (`#hex`): Used when the attribute type is in dotted-decimal form, when no LDAP-specific string encoding exists, or when the string encoding is not restricted to UTF-8 Unicode. Also available by choice when a reversible representation is needed (e.g., for applications that must reconstruct DER form).

2. **UTF-8 string form**: Otherwise, the value is UTF-8 encoded and the following characters must be escaped:
   - Leading space or `#`
   - Trailing space
   - Null character (U+0000)
   - Any of: `"`, `+`, `,`, `;`, `<`, `>`, `\`

   Escaping: backslash followed by two hex digits (`\XX`) for any character; backslash prefix alone for the printable special characters (`' '`, `"`, `#`, `+`, `,`, `;`, `<`, `=`, `>`, `\`).

## Parsing Grammar (String → ASN.1)

RFC 4514 specifies a formal grammar in ABNF (RFC 4234):

```abnf
distinguishedName      = [ relativeDistinguishedName
                           *( COMMA relativeDistinguishedName ) ]
relativeDistinguishedName = attributeTypeAndValue
                            *( PLUS attributeTypeAndValue )
attributeTypeAndValue  = attributeType EQUALS attributeValue
attributeType          = descr / numericoid
attributeValue         = string / hexstring
```

The `string` production handles escaped characters, leading/trailing special handling, and multi-byte UTF-8 sequences. The `hexstring` production handles the `#hex` form.

## Examples

```
UID=jsmith,DC=example,DC=net
OU=Sales+CN=J. Smith,DC=example,DC=net
CN=James \"Jim\" Smith\, III,DC=example,DC=net
CN=Before\0dAfter,DC=example,DC=net
1.3.6.1.4.1.1466.0=#04024869
CN=Lu\C4\8Di\C4\87
```

The last example encodes a five-letter name containing non-ASCII Unicode characters (Latin Small Letter C with Caron and Latin Small Letter C with Acute) as escaped UTF-8 byte sequences.

## Security Considerations

### Information Disclosure

DNs often contain descriptive information about real-world entities: full names, email addresses, physical locations, organizational affiliations. Privacy laws in many jurisdictions restrict disclosure of such information. RFC 4514 encourages use of Directory Information Tree (DIT) structural rules, access controls, and LDAP authentication/data security services to limit exposure of sensitive naming attributes.

### Non-Reversibility

The string encoding of a DN is **not a reversible transformation to DER**. A `CN=Sam` string is ambiguous between TeletexString and PrintableString underlying encodings, both of which have different DER forms. Applications requiring DER reconstruction (notably X.509 certificate verification) MUST NOT use the string representation as an intermediate step — they should use the `#hex` BER encoding form instead. This is an important interoperability concern wherever LDAP and PKI systems share DN handling.

## Changes Since RFC 2253

RFC 4514 introduces several substantive changes over [[collection/sources/2026-05-02-rfc2253-txt|RFC 2253]]:

- Removed the IESG Note (addressed by the 2006 authentication standardization)
- Clarified Section 2 as a RECOMMENDED algorithm, not the only permissible one
- Expanded short name usage: any IANA-registered attribute short name may appear, not just those in a fixed table
- Updated ABNF (now using RFC 4234): allows single-character type names (`L`), more restrictive OID productions, permits escaping `=` as `\=`, permits `\` + space for escaped space, requires hex escaping of null (U+0000), removes LDAPv2-only constructs
- Dropped LDAPv2 compatibility requirements (LDAPv2 is now Historic)
- Added presentation guidance (Appendix A) for human interfaces displaying DN strings
- Added discussion of the non-reversibility security concern
