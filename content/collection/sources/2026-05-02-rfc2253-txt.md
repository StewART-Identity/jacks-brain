RFC 2253 (December 1997) defines the string format for representing X.500 [[collection/concepts/distinguished-name|Distinguished Names]] in the Lightweight Directory Access Protocol. It obsoletes RFC 1779 and was itself superseded by RFC 4514 (June 2006) as part of the LDAPbis revision. Authors: [[collection/entities/mark-wahl|M. Wahl]] (Critical Angle Inc.), [[collection/entities/steve-kille|S. Kille]] ([[collection/entities/isode-ltd|Isode Ltd.]]), and [[collection/entities/tim-howes|T. Howes]] ([[collection/entities/netscape-communications|Netscape Communications Corp.]]).

## Purpose

Distinguished Names are the primary keys to entries in X.500 and LDAP directories. At the protocol level, DNs are encoded in ASN.1 BER. RFC 2253 specifies a parallel human-readable string encoding for use in the LDAP protocol, where DNs are transferred as UTF-8 strings rather than BER. The primary goal is ease of encoding and decoding; human readability is a secondary goal.

## DN Structure (Background)

From X.501, the ASN.1 structure is:

```
DistinguishedName ::= RDNSequence
RDNSequence       ::= SEQUENCE OF RelativeDistinguishedName
RelativeDistinguishedName ::= SET SIZE (1..MAX) OF AttributeTypeAndValue
AttributeTypeAndValue ::= SEQUENCE { type AttributeType, value AttributeValue }
```

A DN is a sequence of Relative Distinguished Names (RDNs), each containing one or more AttributeTypeAndValue (AVA) pairs. Single-valued RDNs are the common case; multi-valued RDNs are allowed and encoded with `+` as the separator.

## Conversion Algorithm

### RDNSequence to String

Components are encoded from **last to first** ‚Äî from the most general toward the most specific ‚Äî separated by commas. An empty RDNSequence produces an empty string. This reverse ordering means a DN like `CN=Steve Kille,O=Isode Limited,C=GB` reads from most-specific (CN) to most-general (C), which is the familiar human-readable left-to-right order.

### AttributeTypeAndValue Encoding

Each AVA is encoded as `type=value`. If the attribute type appears in the published LDAP attribute table, the short name is used; otherwise the OID is used in dotted-decimal notation. Common short names:

| String | X.500 AttributeType |
|--------|---------------------|
| CN | commonName |
| L | localityName |
| ST | stateOrProvinceName |
| O | organizationName |
| OU | organizationalUnitName |
| C | countryName |
| STREET | streetAddress |
| DC | domainComponent |
| UID | userid |

### AttributeValue Escaping

Values with no string representation are encoded as `#` followed by the hexadecimal BER encoding of the value. Values with string representations are first converted to UTF-8, then the following characters are escaped with a backslash prefix:

- `,` `+` `"` `\` `<` `>` `;` ‚Äî anywhere in the value
- Space or `#` ‚Äî at the beginning of the value
- Space ‚Äî at the end of the value

Characters that cannot be represented in printable ASCII use `\XX` hex notation (e.g., the UTF-8 byte sequence for a non-ASCII Unicode character is escaped byte-by-byte).

## Examples

```
CN=Steve Kille,O=Isode Limited,C=GB
OU=Sales+CN=J. Smith,O=Widget Inc.,C=US
CN=L. Eagle,O=Sue\, Grabbit and Runn,C=GB
CN=Before\0DAfter,O=Test,C=GB
1.3.6.1.4.1.1466.0=#04024869,O=Test,C=GB
SN=Lu\C4\8Di\C4\87
```

## BNF Grammar

RFC 2253 provides a formal ABNF grammar (based on RFC 822) for parsing DN strings. It includes legacy forms from RFC 1779 ‚Äî quoted values, semicolon separators, whitespace around delimiters, `oid.`/`OID.` prefixes ‚Äî that implementations MUST accept for LDAPv2 compatibility but MUST NOT generate.

## Security Considerations

The LDAP string representation of a DN is **not a reversible transformation** from ASN.1/DER. Round-tripping a DN through LDAP string encoding does not necessarily reproduce the original BER/DER form ‚Äî for example, `CN=Sam` is ambiguous between a TeletexString and a PrintableString value, both of which have different DER encodings. Applications that require the DER form (notably X.509 certificate verification) SHOULD use the `#hex` encoding rather than the string form for attribute values.

## IESG Note

RFC 2253 carries the same IESG caveat as its companion December 1997 LDAPv3 RFCs: update operations were discouraged pending standardization of mandatory authentication mechanisms. That gap was closed in 2006 by [[collection/sources/2026-05-02-rfc4513-txt|RFC 4513]].
