RFC 4532 (Standards Track, June 2006), authored by [[collection/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[collection/entities/openldap-foundation|OpenLDAP Foundation]], defines the LDAP **"Who am I?"** extended operation — a mechanism for [[collection/concepts/ldap|LDAP]] clients to obtain the authorization identity the server has associated with the current session. Named after the Unix `whoami(1)` command, which displays the effective user ID.

## Purpose and Background

The RFC is intended to replace the Authorization Identity Controls mechanism of RFC 3829 by [[collection/entities/rob-weltman|Rob Weltman]], [[collection/entities/mark-smith|Mark Smith]], and [[collection/entities/mark-wahl|Mark Wahl]], which used Bind request and response controls to communicate the authorization identity. The core problem with that approach: Bind controls are not protected by the security layers the Bind operation itself establishes. A response control returned as part of the Bind response is exposed before the Bind-established session security is active.

The "Who am I?" operation is sent as an `ExtendedRequest` *after* a successful Bind, inside the established security context — protected by whatever TLS or SASL layers the Bind created. This addresses the timing gap that made RFC 3829's mechanism vulnerable.

An additional motivation: operations outside the Bind request/response cycle are simpler to compose. The RFC explicitly notes that the "Who am I?" operation can be augmented with the [[collection/concepts/ldap-proxy-authorization|Proxy Authorization Control]] ([[collection/sources/2026-05-02-rfc4370-txt|RFC 4370]]) to probe the identity the server would assign to an assumed proxy identity — something not easily expressed via a Bind control.

## Wire Format

The operation is identified by OID `1.3.6.1.4.1.4203.1.11.3` (the `whoamiOID`), assigned by the [[collection/entities/openldap-foundation|OpenLDAP Foundation]] from its IANA-assigned private enterprise arc (`1.3.6.1.4.1.4203`). This is the same `.11` branch used for other Zeilenga-authored LDAP protocol extensions: `passwdModifyOID` at `.11.1` ([[collection/sources/2026-05-02-rfc3062-txt|RFC 3062]]).

```
whoamiOID ::= "1.3.6.1.4.1.4203.1.11.3"
```

**Request**: An `ExtendedRequest` with `requestName = whoamiOID` and no `requestValue`. Example encoding (hex):

```
30 1e 02 01 02 77 19 80 17 31 2e 33 2e 36 2e 31
2e 34 2e 31 2e 34 32 30 33 2e 31 2e 31 31 2e 33
```

**Response**: An `ExtendedResponse` with no `responseName`. The `response` field:
- **Absent or empty** — client is treated as anonymous
- **`dn:<distinguished-name>`** — a [[collection/concepts/distinguished-name|directory entry DN]] is the primary authorization identity
- **`u:<userid>`** — a non-DN user identifier is the primary authorization identity

Example response returning authzId `u:xxyyz@EXAMPLE.NET` (hex):

```
30 21 02 01 02 78 1c 0a 01 00 04 00 04 00 8b 13
75 3a 78 78 79 79 7a 40 45 58 41 4d 50 4c 45 2e 4e 45 54
```

## Operational Semantics

Servers advertise support by including `whoamiOID` in the `supportedExtension` attribute of the root DSE. The RFC specifies servers SHOULD advertise the extension only when willing and able to perform it.

Result handling:
- **Success**: Returns a whoami response with result code `success` and the authzId (or empty if anonymous)
- **Unwilling/unable**: Returns an appropriate non-success result code (`operationsError`, `protocolError`, `confidentialityRequired`, `insufficientAccessRights`, `busy`, `unavailable`, `unwillingToPerform`, or other) with absent response field

**Critical constraint**: Clients MUST NOT invoke the operation while any Bind is in progress — including between two requests in a multi-stage SASL Bind. The server returns `operationsError` if this prohibition is violated.

## Interaction with the Proxied Authorization Control

When the [[collection/concepts/ldap-proxy-authorization|Proxy Authorization Control]] ([[collection/sources/2026-05-02-rfc4370-txt|RFC 4370]]) is attached to the "Who am I?" operation, the semantics shift: instead of returning the session's current authorization identity, the operation returns the authzId the server associates with the *asserted* proxy identity. This enables middleware and proxy servers to verify the server's identity-mapping behavior before operating under an assumed identity. If the client is not authorized to assume the asserted identity, the server returns `authorizationDenied` (result code 123).

## Security Considerations

Authorization identities may be sensitive information; security layers should be established to protect them. The RFC notes that the "Who am I?" mechanism is specifically designed to allow security layers established by Bind to protect the authorization identity — the key improvement over RFC 3829's Bind-control approach.

Servers may place access control or other restrictions on use of this operation.

## OID Registration

OID `1.3.6.1.4.1.4203.1.11.3` was registered with IANA as an LDAP Protocol Mechanism per [[collection/sources/2026-05-02-rfc4510-txt|RFC 4520]]. The OpenLDAP Foundation holds delegated authority for the `1.3.6.1.4.1.4203` arc via its IANA-assigned private enterprise allocation.
