RFC 4527 (Standards Track, June 2006) defines the [[collection/concepts/ldap-read-entry-controls|LDAP Read Entry Controls]] — a pair of controls that allow a client to read the target entry of an update operation as an atomic part of that operation. Authored by [[collection/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[collection/entities/openldap-foundation|OpenLDAP Foundation]], the RFC extends [[collection/concepts/ldap|LDAP]] [[collection/sources/2026-05-02-rfc4510-txt|RFC 4510]] with functionality modeled after similar capabilities in the X.500 Directory Access Protocol (DAP).

## The Two Controls

### Pre-Read Control (OID 1.3.6.1.1.13.1)

The Pre-Read request control attaches to an update request (Modify, Delete, ModifyDN) and asks the server to return the entry's state *before* the update is applied. Its `controlValue` is a BER-encoded `AttributeSelection` (defined in [[collection/sources/2026-05-02-rfc4511-txt|RFC 4511]], extended by RFC 3673), specifying which attributes to include. The corresponding response control returns a BER-encoded `SearchResultEntry`.

Applicable to: `modifyRequest`, `delRequest`, `modDNRequest`.

Typical use cases:
- Capture replaced or deleted attribute values from a Modify operation
- Obtain a copy of the entry before it is deleted

### Post-Read Control (OID 1.3.6.1.1.13.2)

The Post-Read request control attaches to an update request (Add, Modify, ModifyDN) and asks the server to return the entry's state *after* the update is applied. Same `controlValue` format as Pre-Read.

Applicable to: `addRequest`, `modifyRequest`, `modDNRequest`.

Typical use cases:
- Read server-assigned operational attributes (e.g., [[collection/concepts/ldap-entry-uuid|`entryUUID`]] per [[collection/sources/2026-05-02-rfc4530-txt|RFC 4530]], `modifyTimestamp` per [[collection/sources/2026-05-02-rfc4512-txt|RFC 4512]]) that are set or updated by the server as part of the write
- Confirm the resulting state of the entry after a modification

## Atomicity

The RFC is explicit: the normal processing of the update operation and the processing of either control MUST be performed as **one atomic action** isolated from other update operations. This means no other client can observe the entry in an intermediate state between the read and the write. If the update fails, no read control response is returned.

This atomicity guarantee is what makes the controls useful for audit and compare-and-swap workflows — a client can capture the pre- and post-state of a modification in a single round trip, without exposure to concurrent modifications between a separate read and write.

## Combination with Other Controls

The Pre-Read and Post-Read controls can be combined with each other (requesting both pre- and post-state in a single operation) and with other controls. The RFC specifically identifies:

- **[[collection/concepts/ldap-assertion-control|Assertion Control]]** ([[collection/sources/2026-05-02-rfc4528-txt|RFC 4528]]): combining Pre-Read + Assertion + a modify gives a conditional read-modify pattern: "apply this change only if this condition holds, and return both the original and resulting entry"
- **manageDsaIT control** (RFC 3296): semantics of each control in the combination apply independently

This composability with the [[collection/concepts/ldap-assertion-control|Assertion Control]] is the foundation for atomic compare-and-swap on directory entries: the Assertion Control gates execution on a precondition, while the Pre-Read and Post-Read controls capture the entry state as evidence of the transition.

## Discovery

Servers implementing these controls SHOULD publish their OIDs in the `supportedControl` attribute of the root DSE (per [[collection/sources/2026-05-02-rfc4512-txt|RFC 4512]]):

| Control | OID |
|---|---|
| Pre-Read | `1.3.6.1.1.13.1` |
| Post-Read | `1.3.6.1.1.13.2` |

Both OIDs are in the IANA-assigned LDAP arc (`1.3.6.1.1`), not in the [[collection/entities/openldap-foundation|OpenLDAP Foundation]]'s private enterprise arc — consistent with the pattern Zeilenga followed for [[collection/sources/2026-05-02-rfc4528-txt|RFC 4528]]'s Assertion Control OID.

## Security Considerations

The controls extend update operations with read capabilities. Servers MUST enforce that the client is authorized both to perform the update and to read the information returned by the control. The authorization check for the read is subject to access controls just as a Search operation would be — the control does not bypass access control to return attributes the client could not otherwise read.

## Provenance

The Pre-Read and Post-Read controls are modeled after similar capabilities in the X.500 DAP (X.511). The RFC is part of the broader 2006 LDAP standards effort coordinated alongside the [[collection/sources/2026-05-02-rfc4510-txt|RFC 4510]] technical specification series, though it is not one of the nine core documents. It is one of several Zeilenga-authored companion RFCs that extended the [[collection/concepts/ldap-controls|controls mechanism]] defined in [[collection/sources/2026-05-02-rfc4511-txt|RFC 4511]] §4.1.11.

For cross-cutting analysis of how this RFC fits into the LDAP extensibility landscape alongside RFC 4528, RFC 4529, and the controls mechanism broadly, see [[collection/synthesis/ldap-protocol-extensibility|LDAPv3 Extensibility: Controls, Features, and Companion RFCs]].
