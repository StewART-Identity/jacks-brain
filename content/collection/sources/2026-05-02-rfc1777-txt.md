RFC 1777 (Standards Track, March 1995) is the LDAPv2 specification — the second version of the Lightweight Directory Access Protocol. Authored by [[collection/entities/wengyik-yeong|Wengyik Yeong]] (Performance Systems International / PSI Inc.), [[collection/entities/tim-howes|Tim Howes]] (University of Michigan), and [[collection/entities/steve-kille|Steve Kille]] (ISODE Consortium), it defines a lightweight alternative to X.500's Directory Access Protocol (DAP) for reading and writing X.500 directories directly over TCP. It obsoletes RFC 1487 (LDAPv1, December 1993) and was itself obsoleted in December 1997 by [[collection/sources/2026-05-02-rfc2251-txt|RFC 2251]], which defined LDAPv3.

## Purpose and Historical Context

X.500 directory technology in the early 1990s carried high "cost of entry" — full DAP required implementing the OSI session/presentation stack before any directory access was possible. Earlier efforts to reduce this cost (the Directory Assistance Service, RFC 1202; DIXIE, RFC 1249) succeeded but were tied to specific implementations. RFC 1777 defines an implementation-independent protocol that maps directly onto TCP, bypassing the OSI stack entirely while preserving access to X.500 directories.

The three-generation sequence RFC 1487 (1993) → RFC 1777 (1995) → [[collection/sources/2026-05-02-rfc2251-txt|RFC 2251]] (1997) represents the complete LDAPv1→v2→v3 evolution. The architectural evolution across these specifications is analyzed in [[collection/synthesis/ldap-technical-specification-architecture|LDAPv3 Specification Architecture]].

## Protocol Model

LDAP follows a client-server model: clients transmit protocol requests; servers perform directory operations and return responses. Operations are asynchronous — multiple requests and responses may be in flight simultaneously, correlated by `messageID`. RFC 1777 emphasizes minimizing client complexity to encourage widespread deployment.

A notable constraint: **servers handle referrals internally**. If a server cannot service a request, it should locate the appropriate server and return only final results. RFC 1777 makes no provision for returning referrals to clients — this was added in LDAPv3 ([[collection/sources/2026-05-02-rfc2251-txt|RFC 2251]]).

## Transport

LDAP runs over TCP; servers listen on **port 389**. `LDAPMessage` PDUs are mapped directly onto the TCP bytestream. An alternative Connection Oriented Transport Service (COTS) mapping is also defined.

## Protocol Operations

All exchanges use the `LDAPMessage` envelope:

```
LDAPMessage ::= SEQUENCE {
    messageID  MessageID,
    protocolOp CHOICE { ... }
}
MessageID ::= INTEGER (0 .. maxInt)  -- maxInt = 65535
```

Note: `maxInt` is 65535 in LDAPv2; LDAPv3 raised it to 2147483647.

RFC 1777 defines nine operations:

| Operation | Purpose |
|---|---|
| **Bind** | Initiate session and authenticate (simple cleartext or Kerberos v4) |
| **Unbind** | Terminate session (no response) |
| **Search** | Query entries matching a filter within base/one/subtree scope |
| **Modify** | Add/delete/replace attribute values atomically |
| **Add** | Create a new entry |
| **Delete** | Remove a leaf entry |
| **ModifyRDN** | Change the last RDN component of an entry's name |
| **Compare** | Test whether an entry has a specific attribute value |
| **Abandon** | Request cancellation of an in-progress operation |

**Absent from LDAPv2**: the Extended operation (LDAPv3's open-ended extensibility mechanism) and the Controls mechanism (also new in LDAPv3). All LDAPv2 operations have fixed semantics with no per-operation extension capability.

### Bind and Authentication

LDAPv2's Bind operation supports three authentication choices:

- **`simple`** — cleartext password. Used for both name/password authentication and anonymous binds (zero-length name and password).
- **`krbv42LDAP`** — Kerberos version 4 ticket for the LDAP server itself (suggested service name: `ldapserver`).
- **`krbv42DSA`** — Kerberos version 4 ticket for the underlying X.500 DSA (suggested service name: `x500dsa`).

Two Kerberos options exist because LDAP servers and backing DSAs may be distinct systems. The security section acknowledges the limitations plainly: "This version of the protocol provides facilities only for simple authentication using a cleartext password, and for kerberos version 4 authentication." No [[collection/concepts/sasl|SASL]] framework, no [[collection/concepts/ldap-tls|TLS]], no transport-level security — these were all LDAPv3 additions. The security implications of this model are examined in [[collection/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture]].

### Search

The Search request specifies `baseObject`, `scope` (baseObject/singleLevel/wholeSubtree), `derefAliases`, `sizeLimit`, `timeLimit`, `attrsOnly`, a `filter`, and an `attributes` list. The `Filter` type supports: and, or, not, equalityMatch, substrings, greaterOrEqual, lessOrEqual, present, and approxMatch. The [[collection/concepts/ldap-search-filters|search filter string syntax]] for human use was defined separately (RFC 1960, later RFC 2254 for LDAPv3); RFC 1777 specifies only the BER wire encoding.

### ModifyRDN vs. LDAPv3 ModifyDN

RFC 1777's `ModifyRDN` changes only the last RDN component of an entry — it cannot move entries to a new parent DN. LDAPv3 expanded this to `ModifyDN` with an optional `newSuperior` field enabling full entry relocation within the DIT.

## BER Encoding

Protocol elements use ASN.1 Basic Encoding Rules (BER), with two restrictions to reduce decoder complexity:
1. Only definite-length encoding
2. Bitstrings, octet strings, and character string types use primitive form only

## Result Codes

`LDAPResult` carries `resultCode`, `matchedDN`, and `errorMessage`. The LDAPv2 result code set is a subset of LDAPv3's — notably absent are codes for referrals (`referral`, `adminLimitExceeded`, `unavailableCriticalExtension`) and SASL (`saslBindInProgress`), since those features did not exist in v2.
