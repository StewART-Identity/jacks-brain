RFC 3062 (Standards Track, February 2001) defines an LDAPv3 Extended Operation for modifying user passwords. Authored by [[collection/entities/kurt-zeilenga|Kurt D. Zeilenga]] of the [[collection/entities/openldap-foundation|OpenLDAP Foundation]], the operation is identified by OID `1.3.6.1.4.1.4203.1.11.1` (`passwdModifyOID`) — from the OpenLDAP Foundation's IANA-assigned private enterprise arc.

## Motivation

Standard LDAP update operations (e.g., Modify) act on attributes of a directory entry. This works for password changes when the user has a directory entry containing a `userPassword` attribute. But the integration of [[collection/concepts/sasl|SASL]] (RFC 2222, RFC 2829) introduced authentication identities that are not LDAP Distinguished Names — a user might authenticate via DIGEST-MD5 using a plain username rather than a DN. Such a user may have no directory entry at all, or their password may be managed by the SASL service provider rather than stored in the directory.

In this model, standard Modify operations cannot update passwords: there is no entry to modify, or the password is not stored as a directory attribute. RFC 3062 addresses this gap with an extended operation that works regardless of:
- Whether the user is represented by a directory entry
- Whether the user is identified by a DN or a non-DN identity
- Where and how the server stores the user's password

## Protocol

The operation is an LDAPv3 Extended Operation (RFC 2251 Section 4.12):

- **OID**: `1.3.6.1.4.1.4203.1.11.1` (`passwdModifyOID`)
- **Request**: `ExtendedRequest` with `requestName = passwdModifyOID`
- **Response**: `ExtendedResponse` with no `responseName`

### Request Structure (ASN.1)

```asn1
PasswdModifyRequestValue ::= SEQUENCE {
  userIdentity  [0] OCTET STRING OPTIONAL,
  oldPasswd     [1] OCTET STRING OPTIONAL,
  newPasswd     [2] OCTET STRING OPTIONAL
}
```

- **`userIdentity`**: The user whose password is being changed. May be a DN or a non-DN identity string. If absent, the operation acts on the password of the currently authenticated session user.
- **`oldPasswd`**: The user's current password. If present and incorrect, the server MUST reject the request without changing the password.
- **`newPasswd`**: The desired new password. If absent, the server generates one.

### Response Structure (ASN.1)

```asn1
PasswdModifyResponseValue ::= SEQUENCE {
  genPasswd  [0] OCTET STRING OPTIONAL
}
```

- **`genPasswd`**: Present only when the server generated a password on behalf of the client. The server MUST return the generated password in this field upon success.
- If a non-success result code is returned, the response field MUST be absent.

## Operation Requirements

- The server SHALL only return success upon successfully changing the password; it SHALL leave the password unmodified otherwise.
- If `oldPasswd` is present and cannot be verified or is incorrect, the server SHALL NOT change the password.
- If `oldPasswd` is absent, the server MAY use other policy to decide whether to change the password.
- If `newPasswd` is absent, the server SHALL either generate a password or return a non-success result code.
- The server SHALL NOT generate a password if the client provided `newPasswd`.
- The operation SHALL NOT be used anonymously.
- Servers SHOULD advertise support via the `supportedExtension` attribute in the root DSE.
- Clients SHOULD verify the server supports this extension (via `supportedExtension`) before attempting it.
- Possible non-success result codes include: `adminLimitExceeded`, `busy`, `confidentialityRequired`, `operationsError`, `unavailable`, `unwillingToPerform`.

## Security Requirements

RFC 3062 imposes a hard security precondition: this operation MUST be used with confidentiality protection such as [[collection/concepts/ldap-tls|StartTLS]] (RFC 2830). The NULL cipher suite MUST NOT be used. The operation itself provides no privacy or integrity protection — it relies entirely on the transport-layer security established before the operation is invoked. Without confidentiality, both the old and new passwords are transmitted in cleartext.
