**Recommendation ITU-T X.509 (2019) – Technical Corrigendum 1** (approved October 2021, ITU-T Study Group 17) corrects two defect reports — DRs 431 and 432 — against the 9th edition of [[collection/concepts/x509-pki|X.509]]. It is co-published as ISO/IEC 9594-8:2020/Cor.1:2021. Both corrections target the [[collection/concepts/asn1|ASN.1]] specification of the cryptographic algorithm framework: DR 431 replaces the informal `algorithm` component description with a rigorous information object class definition; DR 432 renames four SHA/RSA algorithm identifier constants and fixes their RFC cross-references.

## Defect Report 431: ALGORITHM Information Object Class

DR 431 replaces the first part of **clause 6.2.2** with a formal ASN.1 CLASS definition for cryptographic algorithms. The original text described the algorithm component informally; the correction introduces the `ALGORITHM` information object class and three parameterized data types.

### The ALGORITHM Class

```asn1
ALGORITHM ::= CLASS {
    &Type       OPTIONAL,
    &DynParms   OPTIONAL,
    &id         OBJECT IDENTIFIER UNIQUE }
WITH SYNTAX {
    [PARMS      &Type]
    [DYN-PARMS  &DynParms]
    IDENTIFIED BY &id }
```

The three fields serve distinct roles:

- **`&Type`** — fixed parameters required to fully specify the algorithm (e.g., NULL for RSA-based algorithms that take no parameters). Absent for algorithms that need no fixed parameters.
- **`&DynParms`** — dynamic parameters exchanged between communicating entities at invocation time. Absent when no dynamic parameters are needed.
- **`&id`** — the OID that uniquely identifies the algorithm.

### Three Parameterized Data Types

The correction defines three parameterized types built on the `ALGORITHM` class, each serving a distinct use case:

**`AlgorithmWithInvoke`** — used when the algorithm type is signalled *together with* its invocation (both static and dynamic parameters may be present):

```asn1
AlgorithmWithInvoke{ALGORITHM:SupportedAlgorithms} ::= SEQUENCE {
    algorithm   ALGORITHM.&id({SupportedAlgorithms}),
    parameters  [0] ALGORITHM.&Type({SupportedAlgorithms}{@algorithm}) OPTIONAL,
    dynamParms  [1] ALGORITHM.&DynParms({SupportedAlgorithms}{@algorithm}) OPTIONAL,
    ... }
```

**`AlgorithmIdentifier`** — used when the algorithm type is signalled *without* a corresponding invocation (no dynamic parameters exchanged):

```asn1
AlgorithmIdentifier{ALGORITHM:SupportedAlgorithms} ::= SEQUENCE {
    algorithm   ALGORITHM.&id({SupportedAlgorithms}),
    parameters  ALGORITHM.&Type({SupportedAlgorithms}{@algorithm}) OPTIONAL,
    ... }
```

**`AlgoInvoke`** — used when the algorithm has *already been determined* and only the dynamic invocation information is needed:

```asn1
AlgoInvoke{ALGORITHM:SupportedAlgorithms} ::=
    ALGORITHM.&DynParms({SupportedAlgorithms})
```

The `AlgorithmIdentifier` type is the one most commonly encountered by practitioners: it appears in certificate `signatureAlgorithm` fields, `subjectPublicKeyInfo.algorithm`, CRL `signature` fields, and TLS handshake algorithm negotiation. The formalization of its relationship to the `ALGORITHM` object class makes the constraint system (which OIDs are valid, which parameter types apply to each OID) machine-checkable rather than prose-described.

## Defect Report 432: SHA/RSA Algorithm Identifier Renaming

DR 432 corrects four algorithm identifier constants in **Annex B**. The original names used an inconsistent suffix convention (`...Algorithm`) and cited incorrect IETF RFCs. The correction renames them and updates all RFC cross-references:

| Old name | New name | Old RFC ref | New RFC ref |
|---|---|---|---|
| `sha224WithRSAEncryptionAlgorithm` | `sha224RSA` | RFC 5754 | RFC 4055 |
| `sha256WithRSAEncryptionAlgorithm` | `sha256RSA` | RFC 7427 | RFC 4055 |
| `sha384WithRSAEncryptionAlgorithm` | `sha384RSA` | RFC 7427 | RFC 4055 |
| `sha512WithRSAEncryptionAlgorithm` | `sha512RSA` | RFC 7427 | RFC 4055 |

All four retain their original OID values (`sha224WithRSAEncryption`, `sha256WithRSAEncryption`, `sha384WithRSAEncryption`, `sha512WithRSAEncryption`) — only ASN.1 constant names and RFC citations change. RFC 4055 ("Additional Algorithms and Identifiers for RSA Cryptography") is the canonical IETF reference for these SHA/RSA PKCS#1 v1.5 combinations. RFC 7427 covers RSA with PSS padding (a different mode); its prior appearance in Annex B was an error.

## Edition History Context

This corrigendum belongs to the 9th edition lineage of X.509, which has been under continuous standardization since 1988:

| Edition | Date | Key content |
|---------|------|-------------|
| 1st | 1988-11-25 | Original "Authentication Framework" in X.500 series |
| 3rd | 1997-08-09 | Formal certification path algorithm; PKI elaboration |
| 4th | 2000-03-31 | Attribute certificate framework added |
| 8th | 2016-10-14 | Preceding edition |
| **9th** | **2019-10-14** | Current base edition |
| **9.1 (Cor.1)** | **2021-10-14** | **This document — DRs 431 and 432** |

The complete corrigendum history through the 9th edition records 30+ approved technical corrections, with Study Group 17 handling X.509 since the 5th edition (2005). Study Group 7 handled earlier editions.
