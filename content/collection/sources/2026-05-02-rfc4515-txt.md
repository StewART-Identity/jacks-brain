RFC 4515 defines the human-readable string format for representing [[collection/concepts/ldap-search-filters|LDAP search filters]]. Published June 2006, edited by [[collection/entities/mark-smith|Mark Smith]] of Pearl Crescent, LLC with co-author [[collection/entities/tim-howes|Tim Howes]] of Opsware, Inc., it is part of the RFC 4510 [[collection/concepts/ldap|LDAP]] technical specification series and obsoletes [[collection/sources/2026-05-02-rfc2254-txt|RFC 2254]] (December 1997).

## Context

LDAP transmits search filters over the network as ASN.1 BER-encoded binary per RFC 4511. RFC 4515 defines a parallel human-readable string form used in LDAP URLs (RFC 4516) and other applications. The string form is never transmitted on the wire — it is a developer-facing representation only.

This RFC is part of the June 2006 RFC 4510 series that revised and replaced the original 1997 LDAPv3 specifications (RFCs 2251–2256). It obsoletes RFC 2254, which served the same purpose but lacked a formal UTF-8 encoding statement and used less precise ABNF.

## Filter Grammar

The ABNF uses prefix notation over a UTF-8-encoded Unicode string:

```
filter       = LPAREN filtercomp RPAREN
filtercomp   = and / or / not / item
and          = AMPERSAND filterlist
or           = VERTBAR filterlist
not          = EXCLAMATION filter
filterlist   = 1*filter
item         = simple / present / substring / extensible
simple       = attr filtertype assertionvalue
filtertype   = equal / approx / greaterorequal / lessorequal
extensible   = ( attr [dnattrs] [matchingrule] COLON EQUALS assertionvalue )
             / ( [dnattrs] matchingrule COLON EQUALS assertionvalue )
present      = attr EQUALS ASTERISK
substring    = attr EQUALS [initial] any [final]
```

Key differences from RFC 2254's ABNF:
- `value` renamed to `assertionvalue` (aligns with RFC 4517 and RFC 4511)
- `greater`/`less` renamed to `greaterorequal`/`lessorequal` (removes ambiguity)
- Formal `valueencoding` rule introduced, making UTF-8 constraints explicitly derivable from the grammar

## Value Encoding

RFC 4515 introduces a formal `valueencoding` rule dividing assertion values into `normal` (unreserved UTF-8 octets) and `escaped` (`\xx` hex-escaped octets):

```
valueencoding = 0*(normal / escaped)
normal        = UTF1SUBSET / UTFMB
escaped       = ESC HEX HEX
UTF1SUBSET    = %x01-27 / %x2B-5B / %x5D-7F
                ; excludes NUL, LPAREN, RPAREN, ASTERISK, ESC
```

Five characters must be escaped in assertion values: `*` (0x2a), `(` (0x28), `)` (0x29), `\` (0x5c), NUL (0x00). Any other octet may optionally be escaped.

RFC 4515 adds explicit interoperability requirements absent from RFC 2254: implementations MUST escape all octets > 0x7F that are not part of a valid UTF-8 sequence when generating filters; implementations SHOULD accept non-valid UTF-8 input for backward compatibility with RFC 2254-era implementations that did not treat the filter string as UTF-8.

## Key Filter Types

| Filter type | String syntax | Description |
|---|---|---|
| Equality | `(cn=Babs Jensen)` | Exact attribute match |
| Presence | `(cn=*)` | Attribute exists |
| Substring | `(o=univ*of*mich*)` | Wildcard prefix/infix/suffix |
| Approximate | `(cn~=Howes)` | Phonetic or fuzzy match |
| Greater-or-equal | `(age>=18)` | Lexicographic ordering |
| Less-or-equal | `(age<=65)` | Lexicographic ordering |
| Extensible | `(cn:caseExactMatch:=Fred Flintstone)` | Named matching rule |

## Extensible Matching

The extensible filter allows specifying a matching rule by OID or descriptive name, and optionally signals that DN component attributes should participate via the `:dn` flag:

- `(cn:caseExactMatch:=Fred Flintstone)` — named matching rule (descriptive form)
- `(cn:=Betty Rubble)` — MatchingRuleAssertion with no explicit matching rule
- `(sn:dn:2.4.6.8.10:=Barney Rubble)` — OID matching rule with DN attributes included
- `(o:dn:=Ace Industry)` — equality match including DN components
- `(:1.2.3:=Wilma Flintstone)` — apply rule to any attribute supporting it (attr omitted)
- `(:DN:2.4.6.8.10:=Dino)` — `<dnattrs>` is case-insensitive; apply to any supporting attribute with DN included

Note that RFC 4515 updated the first extensible example from RFC 2254's OID form to the descriptive name `caseExactMatch` to demonstrate the named-rule syntax.

## Changes from RFC 2254

RFC 4515 Appendix A catalogs both technical and editorial changes:

**Technical:**
- Clarified that the string representation is a UTF-8-encoded Unicode string (RFC 2254 was silent on this)
- Revised all ABNF to use common productions from RFC 4512
- Introduced `valueencoding`, `normal`, and `escaped` rules, reducing dependence on descriptive text
- Renamed `value` → `assertionvalue`; `greater`/`less` → `greaterorequal`/`lessorequal`
- Added parentheses around extensible subproductions for clarity
- Added four new filter examples

**Editorial:**
- Added separate Abstract and Introduction sections
- Removed IESG note discouraging update-capable LDAPv3 deployments (authentication concerns addressed by RFC 4513)
- Added security references to RFC 4511 and RFC 4513
- Updated author affiliations (Howes moved from Netscape to Opsware; Smith added as editor)

## Security Considerations

The string representation itself has no known security implications; security concerns attach to LDAP search filter semantics. RFC 4515 defers to RFC 4511 (protocol) and RFC 4513 (authentication and security mechanisms).
