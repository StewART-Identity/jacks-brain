RFC 4525 (Informational, June 2006) defines the [[collection/concepts/ldap-modify-increment|LDAP Modify-Increment Extension]] — an addition to the [[collection/concepts/ldap|LDAP]] Modify operation that allows a client to increment integer attribute values atomically, without the read-modify-retry loop previously required. Authored by [[collection/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[collection/entities/openldap-foundation|OpenLDAP Foundation]], the RFC extends [[collection/sources/2026-05-02-rfc4511-txt|RFC 4511]]'s `ModifyRequest` with a new operation enumeration value.

## The Problem It Solves

Before RFC 4525, incrementing an attribute value in LDAP required:

1. Read the current value of the attribute
2. Compute the incremented value
3. Submit a Modify request using `replace` (or `add`/`delete`) for the new value
4. If another client modified the value between steps 1 and 3, the client must detect the conflict (typically via [[collection/concepts/ldap-assertion-control|Assertion Control]] or optimistic retry) and repeat from step 1

This read-modify-retry loop is both complex to implement correctly and fragile under concurrent write load. It is a common pattern in provisioning — for example, assigning the next available `uidNumber` from a max-counter entry.

## The increment(3) Operation

RFC 4525 extends [[collection/sources/2026-05-02-rfc4511-txt|RFC 4511]]'s `ModifyRequest` by adding a fourth enumeration value to the operation type:

| Value | Name | Meaning |
|---|---|---|
| 0 | add | Add values to the attribute |
| 1 | delete | Delete values from the attribute |
| 2 | replace | Replace all values of the attribute |
| **3** | **increment** | **Increment all existing values by the listed amount** |

The `increment` operation:
- Increments **all** existing values of the modification attribute by the specified amount
- The modification MUST provide exactly **one** value (the increment delta); providing multiple values is a `protocolError`
- The attribute MUST have incrementable values (e.g., INTEGER syntax); non-incrementable types MUST return `constraintViolation` or another appropriate error
- Servers not implementing the extension treat the unknown operation value as a protocol error

## Feature OID: 1.3.6.1.1.14

Servers implementing Modify-Increment SHOULD publish OID `1.3.6.1.1.14` as a value of the `supportedFeatures` attribute in the root DSE (per [[collection/sources/2026-05-02-rfc4512-txt|RFC 4512]]). Clients SHOULD check for this OID before using the feature to avoid protocol errors on non-supporting servers.

The OID `1.3.6.1.1.14` is in the IANA-assigned LDAP arc (`1.3.6.1.1`) — consistent with [[collection/sources/2026-05-02-rfc4527-txt|RFC 4527]]'s Pre-Read and Post-Read control OIDs (`1.3.6.1.1.13.1` and `1.3.6.1.1.13.2`) and [[collection/sources/2026-05-02-rfc4528-txt|RFC 4528]]'s Assertion Control OID (`1.3.6.1.1.12`).

## LDIF Representation

RFC 4525 extends the [[collection/concepts/ldap|LDAP]] Data Interchange Format (LDIF, RFC 2849) ABNF production `<mod-spec>` to support the `increment:` change type:

```
mod-spec =/ "increment:" FILL AttributeDescription SEP attrval-spec "-" SEP
```

Example — increment `uidNumber` by 1:

```ldif
# Increment uidNumber
dn: cn=max-assigned uidNumber,dc=example,dc=com
changetype: modify
increment: uidNumber
uidNumber: 1
-
```

## Combination with RFC 4527 and RFC 4528

The RFC explicitly identifies that Modify-Increment is most powerful when combined with:

- **[[collection/concepts/ldap-read-entry-controls|Pre-Read or Post-Read controls]]** ([[collection/sources/2026-05-02-rfc4527-txt|RFC 4527]]): reading the entry's counter value before or after the increment in a single atomic round trip, without an additional Search operation
- **[[collection/concepts/ldap-assertion-control|Assertion Control]]** ([[collection/sources/2026-05-02-rfc4528-txt|RFC 4528]]): conditioning the increment on a precondition, yielding **test-and-increment** semantics — execute only if the current value satisfies a filter

Together, `increment` + Post-Read + Assertion enables: "if `uidNumber` is below this threshold, increment it and return the resulting value" — a single atomic operation covering what previously required a multi-step optimistic retry loop.

## Security Considerations

RFC 4525 notes that the extension reduces application complexity and should aid in producing correct and secure implementations — fewer operations means fewer atomicity windows. General [[collection/concepts/ldap|LDAP]] security considerations from [[collection/sources/2026-05-02-rfc4510-txt|RFC 4510]] and modify-specific considerations from [[collection/sources/2026-05-02-rfc4511-txt|RFC 4511]] both apply.

## Provenance

The RFC was registered in IANA's LDAP registry under both the **LDAP Object Identifier** category (OID `1.3.6.1.1.14`, Modify-Increment feature) and the **LDAP ModifyRequest Operation Type** category (operation type value 3, name `increment`). Published Informational (not Standards Track) — the same status as the closely related [[collection/sources/2026-05-02-rfc4526-txt|RFC 4526]] (Absolute True and False Filters).

For cross-cutting analysis of how this RFC extends the LDAP protocol's operation vocabulary alongside the controls mechanism and other extension patterns, see [[collection/synthesis/ldap-protocol-extensibility|LDAPv3 Extensibility: Controls, Features, and Companion RFCs]].
