RFC 2696 defines an [[collection/concepts/ldap|LDAPv3]] control extension enabling clients to retrieve search results in pages — useful when a client has limited resources or operates over a low-bandwidth connection. Published September 1999 as an Informational RFC (not a standard), it is co-authored by Chris Weider, Andy Herron, and Anoop Anantha of [[collection/entities/microsoft|Microsoft Corp.]] and [[collection/entities/tim-howes|Tim Howes]] of [[collection/entities/netscape-communications|Netscape Communications Corp.]] The RFC explicitly notes it does not provide sophisticated result-set management — only simple paging.

## The Control

The paged results control (OID `1.2.840.113556.1.4.319`, in Microsoft's OID arc) is attached to `SearchRequest` and `SearchResultDone` messages via the LDAPMessage `controls` field defined in RFC 2251 Section 4.1.12. Its BER-encoded `searchControlValue` carries two fields:

| Field | Direction | Meaning |
|---|---|---|
| `size` | Client → Server | Desired page size |
| `size` | Server → Client | Server's estimate of total result set size (may be 0 if unknown) |
| `cookie` | Client → Server | Opaque resumption token from previous response; empty string on first request |
| `cookie` | Server → Client | Resumption token; empty string when result set is exhausted |

The cookie is explicitly opaque — clients must not interpret its internal structure.

## Client-Server Interaction

1. **Initial request**: Client sends `SearchRequest + pagedResultsControl(pageSize, "")`.
2. **Server response**: Returns up to `pageSize` entries plus `searchResultDone + pagedResultsControl(totalEstimate, "opaqueToken")`.
3. **Subsequent requests**: Client reissues the *identical* search (same filter, base DN, scope, attributes) with the cookie from the last response. MessageID and optionally page size may differ.
4. **Termination**: Server returns an empty cookie when no more entries remain.
5. **Abandonment**: Client sends `pagedResultsControl(0, "lastCookie")` to terminate early.

If a server cannot resume a paged search (e.g., the result set aged out), it returns `unwillingToPerform`. Servers may impose limits on the number of concurrent paged searches and may age out stale ones.

## Consistency Caveat

Result consistency is not guaranteed across pages. If the directory changes between requests, entries may appear more than once or be skipped entirely.

## Relationship to X.500

The control maps to the `PagedResultsRequest` defined in X.511 for X.500(93) directories: `size` → `pageSize`, `cookie` → `queryReference`. The `sortKeys` and `reverse` fields from X.500 are excluded.

## Security Considerations

Servers must manage resources carefully — a client misusing the control could lock out other operations. Additionally, because the server's `size` estimate reveals the total match count before entries are returned, servers performing access-control checks must decide whether to disclose that count.

- **Depends on**: RFC 2251 (LDAPv3 core — defines the controls mechanism used here); [[collection/sources/2026-05-02-rfc2254-txt|RFC 2254]] (string filter syntax used to express the searches being paged)
- **Compare with**: [[collection/concepts/ldap-search-filters|LDAP search filters]] govern *what* entries match; [[collection/concepts/ldap-paged-results|LDAP paged results]] governs *how many* are returned per round trip
