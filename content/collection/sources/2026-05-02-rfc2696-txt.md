RFC 2696 defines an [[collection/concepts/ldap|LDAPv3]] control extension enabling clients to retrieve search results in pages. Published September 1999 as an Informational RFC (not a standard), it is co-authored by Chris Weider, Andy Herron, and Anoop Anantha of [[collection/entities/microsoft|Microsoft Corp.]] and [[collection/entities/tim-howes|Tim Howes]] of [[collection/entities/netscape-communications|Netscape Communications Corp.]] The RFC scopes itself deliberately narrow: simple paging only, with the explicit statement that it "is not designed to provide more sophisticated result set management." Sort-aware paging requires [[collection/sources/2026-05-02-rfc2891-txt|RFC 2891]], which defines a sorting control intended to compose with this one.

## The Control

The paged results control (OID `1.2.840.113556.1.4.319`, in Microsoft's OID arc) is attached to `SearchRequest` and `SearchResultDone` messages via the `controls` field of the `LDAPMessage` envelope defined in RFC 2251 §4.1.12. The wire structure:

```asn1
pagedResultsControl ::= SEQUENCE {
  controlType  1.2.840.113556.1.4.319,
  criticality  BOOLEAN DEFAULT FALSE,
  controlValue searchControlValue
}

realSearchControlValue ::= SEQUENCE {
  size    INTEGER (0..maxInt),
  cookie  OCTET STRING
}
```

The `size` field is directionally overloaded: on the request, it carries the client's desired page size; on the response, it carries the server's *estimate* of the total result set size. Servers that cannot estimate set `size` to `0`. The `cookie` is fully opaque — the spec explicitly prohibits clients from inferring anything about its internal structure or reusing cookies from any response other than the most recent.

## Client-Server Interaction

The exchange follows a stateful cookie-resumption pattern:

1. **Initial request**: Client sends `SearchRequest + pagedResultsControl(pageSize, "")`.
2. **First response**: Server returns up to `pageSize` entries plus `SearchResultDone + pagedResultsControl(totalEstimate, "opaqueToken")`.
3. **Subsequent requests**: Client reissues the *identical* search (same filter, base DN, scope, attributes) with the cookie from the most recent response. MessageID and optionally page size may differ.
4. **Final response**: Server returns an empty cookie when no more entries remain.

The provided example in the RFC: a client requests pages of 3 from a 5-entry result set. The server returns 3 entries with a non-empty cookie, then 2 entries with an empty cookie.

## Abandonment and Session Lifecycle

A client terminates a paged sequence early by sending `SearchRequest + pagedResultsControl(size=0, lastCookie)`. Using the LDAP `Abandon` operation is explicitly discouraged — it may invalidate the cookie.

Once the full sequence completes (server returns empty cookie), the cookie is no longer valid and MUST NOT be reused. A server may impose a limit on concurrent paged searches from a single client and may choose to either reject new requests with `unwillingToPerform` or age out stale sessions. If a session is aged out and the client attempts to resume it, the server SHOULD return `unwillingToPerform`. If a server cannot resume a paged search for any reason, it returns an appropriate error in `SearchResultDone` and both parties treat the session as closed.

## Consistency Caveat

Result consistency is not guaranteed across pages. If the directory changes between round trips, entries may appear more than once or be missed entirely. The RFC acknowledges this explicitly — the no-duplication guarantee holds only if the result set is stable across the entire paging sequence.

## Relationship to X.500

For LDAPv3 servers fronting X.500 (93) directories, the control maps directly onto X.500's `PagedResultsRequest` defined in X.511: `size` → `pageSize`, `cookie` → `queryReference`. X.500's `PagedResultsRequest` also carries `sortKeys` and `reverse` fields — both are explicitly excluded from RFC 2696's mapping, reflecting the RFC's narrow scope. Sort-aware paging via X.500's ordering semantics falls under [[collection/sources/2026-05-02-rfc2891-txt|RFC 2891]].

## Security Considerations

Two concerns are identified:

1. **Resource exhaustion**: A client misusing the control could hold many open paged sessions, locking out other users. The server is expected to enforce limits and age out sessions as a pressure release. The RFC also permits servers to enforce an overriding `sizeLimit` to prevent mass retrieval from publicly accessible directories.

2. **Result count disclosure**: The server's `size` estimate reveals the total match count *before* entries are returned to the client. A client can determine how many entries match a given filter without ever receiving any of the entries themselves. Servers implementing per-entry access control must decide whether disclosing the aggregate count is acceptable even when individual entries are access-controlled.
