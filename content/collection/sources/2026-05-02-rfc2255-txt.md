RFC 2255 defines the original [[collection/concepts/ldap-url|LDAP URL]] format — the `ldap://` URI scheme for encoding [[collection/concepts/ldap|LDAP]] search operations as human-readable strings. Published December 1997 by [[collection/entities/tim-howes|Tim Howes]] and [[collection/entities/mark-smith|Mark Smith]] of [[collection/entities/netscape-communications|Netscape Communications Corp.]], it is part of the cluster of December 1997 LDAPv3 RFCs and supersedes RFC 1959. It was obsoleted by [[collection/sources/2026-05-02-rfc4516-txt|RFC 4516]] (June 2006).

## Purpose and Scope

An LDAP URL encodes a complete LDAP search operation — host, port, base DN, attributes to return, search scope, and search filter — in a single, bookmarkable string. The URL format is particularly important for:

- Embedding LDAP searches in applications, configuration files, and web pages
- Expressing **referrals** — when an LDAP server cannot satisfy an operation, it may return LDAP URLs pointing to other servers the client should contact
- Providing a self-contained, stateless description of a directory query

## URL Grammar

The LDAP URL grammar in BNF:

```
ldapurl    = scheme "://" [hostport] ["/"
             [dn ["?" [attributes] ["?" [scope]
             ["?" [filter] ["?" extensions]]]]]]
scheme     = "ldap"
attributes = attrdesc *("," attrdesc)
scope      = "base" / "one" / "sub"
dn         = distinguishedName          ; RFC 2253 string form
hostport   = hostport from RFC 1738
attrdesc   = AttributeDescription       ; RFC 2251 §4.1.5
filter     = filter from RFC 2254
extensions = extension *("," extension)
extension  = ["!"] extype ["=" exvalue]
extype     = token / xtoken
exvalue    = LDAPString                 ; RFC 2251 §4.1.2
token      = oid from RFC 2252
xtoken     = ("X-" / "x-") token
```

The `?` character delimits each component in sequence. All components after the host/port are optional; trailing `?` delimiters may be omitted when all subsequent components are also absent.

## URL Components

| Component | Description | Default when absent |
|-----------|-------------|---------------------|
| `host` / `port` | LDAP server address | Client uses prior knowledge; default port TCP 389 |
| `dn` | Base DN for the search ([[collection/concepts/distinguished-name|Distinguished Name]] in RFC 2253 string form) | (client chooses) |
| `attributes` | Comma-separated list of attributes to return | All user attributes |
| `scope` | Search scope: `base`, `one`, or `sub` | `base` |
| `filter` | Search filter ([[collection/concepts/ldap-search-filters|RFC 2254]] string form) | `(objectClass=*)` |
| `extensions` | Extension type=value pairs | None |

## The Bindname Extension

RFC 2255 defines one URL extension: **`bindname`**. It specifies the [[collection/concepts/distinguished-name|Distinguished Name]] the client should use when authenticating to the LDAP server during URL resolution.

- The extension value is a DN string (same form as the `dn` component); a NULL string indicates unauthenticated access
- If critical (prefixed `!`): the client MUST authenticate using the given DN
- If non-critical: the client MAY bind using the given DN

Examples from the RFC:

```
ldap:///??sub??bindname=cn=Manager%2co=Foo        # non-critical
ldap:///??sub??!bindname=cn=Manager%2co=Foo       # critical (must authenticate)
```

The comma within the DN value must itself be percent-encoded (`%2c`) to avoid ambiguity with the extension-list delimiter.

**The bindname extension was removed in [[collection/sources/2026-05-02-rfc4516-txt|RFC 4516]]** because no implementations were known. Its removal is noted in RFC 4516's change log as a deliberate simplification — the mechanism existed in the spec but saw no real-world adoption.

## Extension Mechanism

The `<extensions>` component provides forward extensibility via type=value pairs. Extension types follow the same criticality contract as [[collection/concepts/ldap-controls|LDAP controls]]:

- **Critical extension** (prefixed `!`): if unsupported, the client MUST NOT process the URL
- **Non-critical extension** (no prefix): if unsupported, the client MUST ignore it and continue

Extension types using the `X-` or `x-` prefix are reserved for bilateral agreements. All other types must be defined in RFCs. RFC 2255 defines only the `bindname` extension; the mechanism was designed to accommodate future extensions.

## Percent-Encoding

Characters that are URL-illegal or conflict with the URL's component-delimiter syntax must be percent-encoded (`%xx`). Notable cases:

- The `?` character inside a DN, filter, or other component must be encoded as `%3f`
- The `,` character inside an extension value must be encoded as `%2c`
- LDAP filter escape sequences (`\xx`) and URL percent-encoding stack: the `\` in a filter value must be encoded as `%5c` in the URL

Example — a filter `(int=\00\00\00\04)` in a URL becomes `(int=%5c00%5c00%5c00%5c04)`.

## URL Processing

The RFC specifies client resolution behavior:

1. Open a TCP connection to the specified LDAP server, or reuse an existing connection
2. Optionally authenticate — required only if a critical `bindname` extension with a non-NULL DN is present
3. Execute the LDAP search operation described by the URL components
4. Optionally close the connection or reuse it

The connection MAY use TLS for confidentiality and integrity; security service use is at the client's discretion unless mandated by the URL.

## Illustrative Examples

```
# Base search on any server (client chooses)
ldap:///o=University%20of%20Michigan,c=US

# Same search on a specific server
ldap://ldap.itd.umich.edu/o=University%20of%20Michigan,c=US

# Return only postalAddress attribute
ldap://ldap.itd.umich.edu/o=University%20of%20Michigan,c=US?postalAddress

# Subtree search on non-default port
ldap://host.com:6666/o=University%20of%20Michigan,c=US??sub?(cn=Babs%20Jensen)

# One-level search retrieving objectClass attribute
ldap://ldap.itd.umich.edu/c=GB?objectClass?one

# DN containing '?' (must be percent-encoded)
ldap://ldap.question.com/o=Question%3f,c=US?mail

# Filter with binary value (backslash-hex LDAP escaping stacked with URL encoding)
ldap://ldap.netscape.com/o=Babsco,c=US??(int=%5c00%5c00%5c00%5c04)
```

## Security Considerations

RFC 2255 carries the same IESG advisory as the other December 1997 LDAPv3 RFCs: update-capable deployments are discouraged until mandatory authentication mechanisms are standardized. For URL-specific concerns:

- Clients SHOULD have a configurable policy about which servers to connect to and which security mechanisms to use — URLs are often resolved automatically, without user intervention
- Without explicit policy, clients should use anonymous connections
- Reusable passwords sent during URL processing are particularly risky and should not be used unless explicitly permitted by policy
- Following an LDAP URL may unexpectedly retrieve large amounts of data or initiate long-lived searches
