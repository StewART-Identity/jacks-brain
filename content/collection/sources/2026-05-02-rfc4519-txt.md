RFC 4519 (Standards Track, June 2006) is the definitive specification of attribute types and object classes intended for use by [[collection/concepts/ldap|LDAP]] directory clients for user applications — commonly called "White Pages" services. It is the successor to [[collection/sources/2026-05-02-rfc2256-txt|RFC 2256]] (December 1997) and forms the user-schema component of the nine-document [[collection/sources/2026-05-02-rfc4510-txt|RFC 4510]] LDAPv3 Technical Specification series. The editor is [[collection/entities/andrew-sciberras|Andrew Sciberras]] of eB2Bcom (Box Hill North, Victoria, Australia); [[collection/entities/steven-legg|Steven Legg]] and [[collection/entities/kurt-zeilenga|Kurt Zeilenga]] are acknowledged for significant contribution to the revision.

## Scope

RFC 4519 specifies:
- **43 attribute types** for representing people, organizations, groups, locations, devices, and domain names
- **14 object classes** defining the structural shape of common directory entries

The document explicitly excludes attributes used for directory server administration. PKI-related schema elements — certificate attributes and classes from [[collection/sources/2026-05-02-rfc2256-txt|RFC 2256]] — are not included; they are now specified in RFC 4523.

## Relationship to Other Standards

RFC 4519 sits at layers 3–4 of the [[collection/synthesis/ldap-schema-architecture|LDAP four-layer schema architecture]] — it populates the framework defined in [[collection/sources/2026-05-02-rfc4512-txt|RFC 4512]] with standard attribute type and object class definitions, using syntaxes and matching rules from [[collection/sources/2026-05-02-rfc4517-txt|RFC 4517]]. The conceptual role of these definitions is described in [[collection/concepts/x500-user-schema|X.500 User Schema]].

It explicitly updates three other RFCs:
- **RFC 2247** — Supersedes the `dc` attribute type and `dcObject` object class definitions
- **RFC 2798** — Provides the definitive `uid` attribute type description, replacing the informative description there
- **RFC 2377** — Provides the definitive `uidObject` object class description

## Attribute Types

All 43 attribute types originate from X.520 (via [[collection/entities/itu-t|ITU-T]] X.500) or from earlier IETF RFCs for domain and user identity attributes. Two abstract supertypes form the structural base:

- **`name`** (2.5.4.41) — abstract supertype for string naming attributes; subclassed by `cn`, `sn`, `o`, `ou`, `c`, `l`, `st`, `title`, `givenName`, `initials`, `generationQualifier`
- **`distinguishedName`** (2.5.4.49) — abstract base for DN-valued attributes; subclassed by `member`, `owner`, `seeAlso`, `roleOccupant`

Key attribute types for user and organization representation:

| `cn` (commonName) | 2.5.4.3 | Name of a person or object; SUP name |
| `sn` (surname) | 2.5.4.4 | Family name; SUP name |
| `givenName` | 2.5.4.42 | Given name; SUP name |
| `o` (organizationName) | 2.5.4.10 | Organization name; SUP name |
| `ou` (organizationalUnitName) | 2.5.4.11 | OU name; SUP name |
| `c` (countryName) | 2.5.4.6 | Two-letter ISO 3166 code; SINGLE-VALUE |
| `l` (localityName) | 2.5.4.7 | City or geographic region; SUP name |
| `st` (stateOrProvinceName) | 2.5.4.8 | State/province; SUP name |
| `dc` (domainComponent) | 0.9.2342.19200300.100.1.25 | Single DNS label; case-insensitive; IDN via ToASCII |
| `uid` (userid) | 0.9.2342.19200300.100.1.1 | Computer login name; definitive spec superseding RFC 2798 |

Two attribute types are explicitly flagged as deprecated and their use is "greatly discouraged": `searchGuide` (superseded by `enhancedSearchGuide`) and `teletexTerminalIdentifier` (Recommendation F.200 withdrawn).

### The `dc` Attribute and Internationalized Domain Names

The `dc` (domainComponent) attribute holds a single DNS label — not a full domain name. Valid: `"example"`, `"com"`. Invalid: `"example.com"` (multiple components). Matching is case-insensitive (`caseIgnoreIA5Match`), consistent with DNS behavior. For Internationalized Domain Names (IDN), RFC 4519 requires client applications to apply the ToASCII method (RFC 3490) to produce the domain component label, with separate handling for "stored" vs. "query" uses per RFC 3490 §4.

### The `uid` Attribute

The `uid` (userid, OID 0.9.2342.19200300.100.1.1) attribute holds computer system login names. RFC 4519 is the *definitive* specification of this attribute, superseding the informative descriptions in RFC 1274 (COSINE schema, where it was called `userid`) and RFC 2798 (inetOrgPerson object class).

## Object Classes

The 14 object classes define the structural types for the most common directory entries:

```
top (ABSTRACT — from RFC 4512)
├── country (STRUCTURAL) — MUST c
├── locality (STRUCTURAL) — MAY street, seeAlso, st, l, description
├── organization (STRUCTURAL) — MUST o; broad MAY set incl. postal/contact attrs
├── organizationalUnit (STRUCTURAL) — MUST ou; same broad MAY set
├── organizationalRole (STRUCTURAL) — MUST cn; represents a job, function, or position
├── groupOfNames (STRUCTURAL) — MUST member, cn
├── groupOfUniqueNames (STRUCTURAL) — MUST uniqueMember, cn
├── applicationProcess (STRUCTURAL) — MUST cn; represents an executing application
├── device (STRUCTURAL) — MUST cn; appliance, computer, or network element
├── person (STRUCTURAL) — MUST sn, cn
│   ├── organizationalPerson (STRUCTURAL) — adds address/contact attributes
│   └── residentialPerson (STRUCTURAL) — MUST l; adds residential address attrs
├── dcObject (AUXILIARY) — MUST dc; adds domain component to an existing entry
└── uidObject (AUXILIARY) — MUST uid; adds user identification to an existing entry
```

Design notes:
- `dcObject` is **auxiliary** — it augments an existing structural entry with a `dc` value, enabling domain-named entries without conflicting with the entry's structural class
- `uidObject` is similarly **auxiliary**
- `groupOfUniqueNames` uses `uniqueMember` (DN + optional bit string UID) rather than plain `member`, preventing ambiguity when [[collection/concepts/distinguished-name|distinguished names]] are reused and reassigned

## Key Differences from RFC 2256

| Change | Detail |
|--------|--------|
| PKI elements removed | `userCertificate`, `cACertificate`, CRL attrs, `strongAuthenticationUser`, `certificationAuthority` → RFC 4523 |
| Legacy server objects removed | `applicationEntity`, `dSA`, `dmd`, `knowledgeInformation`, `presentationAddress`, `dmdName`, etc. → Historic |
| `dc` attribute incorporated | From RFC 2247; adds IDN ToASCII requirement |
| `uid` attribute incorporated | Definitive description superseding RFC 2798 / RFC 1274 |
| `uidObject` class incorporated | Definitive description superseding RFC 2377 |
| `dcObject` class incorporated | Auxiliary class from RFC 2247 |
| `userPassword` guidance added | Unicode/SASLprep/UTF-8 requirements for textual passwords |
| Syntaxes and matching rules removed | Moved entirely to [[collection/sources/2026-05-02-rfc4517-txt|RFC 4517]] |

The PKI removal is the most architecturally significant change. [[collection/sources/2026-05-02-rfc2256-txt|RFC 2256]] included X.509 certificate attributes as unified user schema; RFC 4519 explicitly scopes itself to *user applications* and delegates PKI to RFC 4523, reflecting a recognition that PKI schema has a different maintenance and stability profile than person/organization schema.

## Security Considerations

1. **Privacy**: Most jurisdictions have privacy laws governing publication of directory attributes describing real people.
2. **Cleartext passwords**: `userPassword` stores values as unencrypted Octet String; cleartext transfer strongly discouraged without transport confidentiality — see [[collection/concepts/ldap-tls|LDAP TLS]].
3. **Multiple `userPassword` values**: Resetting a password when multiple values exist requires care; an administrator who does not know the current password may be unable to perform a targeted delete. Applications should use `modify/replaceValues` or `modify/deleteAttribute+addAttribute`.
4. **Authentication**: When `userPassword` is used for authentication per [[collection/sources/2026-05-02-rfc4513-txt|RFC 4513]], the user need only prove knowledge of *one* value — relevant for multi-value deployments.
