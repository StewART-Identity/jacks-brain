RFC 4530 (June 2006), authored by [[collection/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[collection/entities/openldap-foundation|OpenLDAP Foundation]], defines the `entryUUID` operational attribute and its associated schema elements for LDAP directories. Published as Standards Track alongside the [[collection/sources/2026-05-02-rfc4510-txt|RFC 4510 series]], it addresses a fundamental limitation of [[collection/concepts/distinguished-name|Distinguished Names]]: DNs are not stable identifiers.

[Download original](/api/originals/2026-05-02-rfc4530.txt.docx)

## Problem Addressed

In X.500 and LDAP directories, an entry is identified by its [[collection/concepts/distinguished-name|Distinguished Name]] â€” a hierarchical path through the Directory Information Tree. However, a DN is not a stable identifier: if an entry is renamed or deleted, its former DN becomes available and may subsequently identify a completely different entry. Clients that cache a DN to track a specific object over time can silently switch to referring to the wrong object.

The `entryUUID` attribute solves this by providing an immutable, server-assigned Universally Unique Identifier (UUID) that uniquely identifies each entry independent of its naming or position in the DIT.

## Schema Elements Defined

RFC 4530 defines three interrelated [[collection/concepts/ldap-syntaxes|schema]] elements, all registered in the IANA LDAP parameters OID arc (`1.3.6.1.1.16`):

### UUID Syntax (OID: 1.3.6.1.1.16.1)

A 16-octet (128-bit) value conforming to RFC 4122 UUID format. In LDAP, UUID values are encoded as the ASCII string representation defined in RFC 4122 â€” e.g., `597ae2f6-16a6-1027-98f4-d28b5365dc14`. The ASN.1 definition:

```
UUID ::= OCTET STRING (SIZE(16))
  -- constrained to a UUID [RFC4122]
```

The UUID syntax is distinct from the general [[collection/concepts/ldap-syntaxes|Octet String syntax]] â€” it constrains both the length (exactly 16 octets) and the encoding format (the UUID string representation in the LDAP protocol layer).

### `uuidMatch` Matching Rule (OID: 1.3.6.1.1.16.2)

An equality [[collection/concepts/ldap-matching-rules|matching rule]] for UUID values. Its semantics are equivalent to `octetStringMatch`, but assertion values use the UUID string representation (`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`) rather than the raw octet string encoding. This allows clients to search for an entry by UUID using the familiar dash-delimited string form.

### `uuidOrderingMatch` Matching Rule (OID: 1.3.6.1.1.16.3)

An ordering [[collection/concepts/ldap-matching-rules|matching rule]] for UUID values. Semantics equivalent to `octetStringOrderingMatch`, with assertion values in UUID string form. The RFC notes that not all UUID variants define an ordering, and servers are not obligated to assign UUIDs in any particular sequence â€” this rule exists for completeness, not as a guarantee of meaningful ordering.

### `entryUUID` Attribute Type (OID: 1.3.6.1.1.16.4)

```
( 1.3.6.1.1.16.4 NAME 'entryUUID'
  DESC 'UUID of the entry'
  EQUALITY uuidMatch
  ORDERING uuidOrderingMatch
  SYNTAX 1.3.6.1.1.16.1
  SINGLE-VALUE
  NO-USER-MODIFICATION
  USAGE directoryOperation )
```

Key characteristics:

- **Operational** (`USAGE directoryOperation`) â€” not returned by the `*` attribute selector; clients must explicitly request it with `+` (all operational attributes) or by name `entryUUID`
- **NO-USER-MODIFICATION** â€” the server assigns the UUID; clients cannot set or modify it
- **SINGLE-VALUE** â€” exactly one UUID per entry, always
- **Immutable** â€” once assigned upon entry creation, the UUID never changes, even if the entry is renamed (Modify DN) or its attributes are modified

Servers SHALL generate and assign a new UUID to each entry upon addition and SHALL ensure uniqueness in space and time per RFC 4122 Â§4.

## Security Considerations

UUIDs do not encode any descriptive information about the entry they identify â€” unlike a DN, which often contains the person's name, user ID, or organizational position. RFC 4530 notes this makes `entryUUID` suitable as a reference in contexts where the DN's descriptive content cannot be disclosed to all parties.

## OID Registration

The four OIDs (`1.3.6.1.1.16.1` through `1.3.6.1.1.16.4`) were registered by IANA as LDAP values per RFC 4520. The arc `1.3.6.1.1` is the IANA-assigned LDAP parameters arc â€” distinct from the [[collection/entities/openldap-foundation|OpenLDAP Foundation]]'s private enterprise arc (`1.3.6.1.4.1.4203`) that Zeilenga used for other extensions (RFC 3062, RFC 4529, RFC 4533). Using the IANA arc rather than the OpenLDAP arc reflects RFC 4530's status as a core Standards Track addition to the LDAP schema, not an experimental extension.
