**ITU-T Recommendation X.530** (November 2008), published in parallel as **ISO/IEC 9594-10:2008**, defines how OSI Systems Management services and protocols — principally [[reflect/concepts/osi-systems-management|CMIP/CMIS]] — are used to manage the components of an X.500 [[reflect/concepts/x500-distributed-directory|distributed directory]]. It is the sixth edition of this standard, technically revising the fifth (2005) edition.

The standard covers management of the [[reflect/concepts/directory-information-tree|DIT]] domain, individual [[reflect/concepts/x500-distributed-directory|DSAs]] and [[reflect/concepts/x500-distributed-directory|DUAs]], and describes managed object definitions used to instrument those components. Management of the Directory Management Domain (DMD) as a whole was declared "for further study."

## Scope

X.530 defines management in four segments:

| Segment | Description |
|---------|-------------|
| **DIT domain management** | Managing Directory information itself — schema, entry consistency, aliases, lists |
| **DSA management** | Configuration, fault, performance, security, and accounting for a single DSA |
| **DUA management** | Configuration for a single DUA within a DMD |
| **DMD management** | Integrated management of the full domain — declared out of scope for further study |

## Management Requirements

Section 6 structures management requirements through roles derived from the **Telecommunications Management Network (TMN)** hierarchy:

- **Directory user** — Consumers who trigger management information collection by making requests; their service agreement creates monitoring obligations.
- **Directory customer** — Organizations procuring directory services; entitled to billing, status queries, and service agreement enforcement.
- **Directory service manager** — Responsible for implementing service agreements; creates directory configuration, handles billing, inhibits unauthorized binds, validates operations.
- **Directory system administrator** — Handles reconfiguration, troubleshooting, monitoring of DSA operational limits.
- **Directory business manager** — Sets policy for demand monitoring, capacity expansion, marketing-oriented usage tracking.

Five management functional areas apply to the DSA:

| Area | Key requirements |
|------|-----------------|
| **Configuration** | Enable/disable DSAs, manage operational bindings, set size/time limits, manage schema subentries, redistribute DIB |
| **Fault** | Detect and report service errors, support backup/restore, correlate faults, log chaining inconsistencies |
| **Performance** | Count operation types (read, search, modify, etc.), detect threshold crossings, log shadow update statistics |
| **Security** | Audit trail of accesses, log security violations (wrong credentials, insufficient rights, invalid signatures), configurable audit granularity |
| **Accounting** | Generate usage records per customer, support charging/billing, track cost of service provision |

The security management section specifies that each audit record must include: event time, origin of request/response, operation, target of operation, and outcome. Seven specific error codes are enumerated as mandatory audit events: `inappropriateAuthentication`, `invalidCredentials`, `insufficientAccessRights`, `invalidSignature`, `protectionRequired`, `blockedCredentials`, `noInformation`.

## Directory Management Model

The **Directory Management Model** (Section 7) identifies managed objects, participating entities, and data flows. Management entities include DUAs, DSAs, Directory managers, Directory agents, Directory users, and Directory customers.

Management roles (subdivisions of the Directory manager entity):

| Role | Responsibility |
|------|---------------|
| Helpdesk | Receive and route user/customer help requests |
| Subscription manager | Add/remove users, change credentials |
| Configuration manager | Add/remove/change Directory components |
| Fault manager | Act on event reports indicating service faults |
| Security manager | Protect against unauthorized access |
| Accounting manager | Determine settlement from accounting data |
| Schema manager | Manage DIT structure rules, name forms, attribute types, matching rules |
| Planning manager | Statistical analysis and capacity planning |

### Layered Management Model

X.530 adopts a five-layer hierarchy aligned with the TMN model:

| Layer | Name | Content |
|-------|------|---------|
| 1 | Network Element Layer | Managed object instances — DSAs, DUAs, DSEs, etc. |
| 2 | Network Element Management Layer | Per-class object managers (DSA manager, DUA manager, entry manager) |
| 3 | Network Management Layer | Functional area managers: configuration, fault, security, performance |
| 4 | Service Management Layer | Accounting, helpdesk, service contract management, security management toward customers |
| 5 | Business Management Layer | Enterprise-wide coordination; billing and settlement between providers |

### Management Information Model

Management information divides between the **DIB** and a separate **Management Information Base (MIB)**:

- *Static* operational attributes (schema rules, prescriptive ACI) — placed in the DIB; manageable via DAP
- *Dynamic* counters and gauges (traffic counters, association counts) — placed in the MIB; manageable via [[reflect/concepts/osi-systems-management|CMIP]]

This division reflects the principle that the DIB is not designed to hold rapidly changing data.

## Directory Service Model

The **Directory Service** is a managed object representing the service presented to a Directory customer. It has two conditional packages:

**Directory Information Service** — Controls what operations and DIT subtrees are accessible:
- Allowed/disallowed operation combinations (by operation type, attribute type, attribute value, matching rule)
- Allowed/disallowed distinguished names (restricting entry points into the DIT — prevents full country-level scans)
- Service identifier and description

**Directory Control Service** — Limits on service activity:
- Maximum result count (`maxEntriesReturned`)
- Maximum time for results (`maxTimeForResults`)

## Managed Object Hierarchy

The management containment hierarchy (Figure 5 in the specification) positions objects as follows:

```
DMD
├── DSA
│   ├── DSE (per DIT entry in the DSA's naming context)
│   ├── Known DSA (per peer DSA this DSA knows about)
│   ├── Known DUA (per DUA that has connected)
│   │   └── UL Connection Endpoint (per active association)
│   ├── HOB (Hierarchical Operational Binding)
│   ├── NHOB (Non-Specific Hierarchical Operational Binding)
│   └── Shadowing Agreement
├── Directory Customer
│   └── Directory Service
│       └── Directory User
└── DUA
    └── Known DSA
        └── UL Connection Endpoint
```

### DSA Managed Object

The DSA managed object (Annex A, `dSA MANAGED OBJECT CLASS`) is the central management point. The `dSAPackage` exposes:

- **Access Point** — presentation address, protocol information, and AE Title
- **Operational/Administrative State** — start/stop state via [[reflect/concepts/osi-systems-management|CMIP]] state change notifications
- **Error counters** — security errors, name errors, service errors, alias problems, loops detected, invalid references, etc.
- **Operation counters** — per operation type (read, compare, search, list, add, remove, modify, modifyDN) both direct and chained
- **Association configuration** — max/timeout for DAP, DSP, DOP, DISP associations; scope of referral and chaining
- **Authentication policies** — peer entity, request, and result authentication types

### Shadowing Agreement Managed Object

The `shadowingAgreementMO` managed object tracks [[reflect/concepts/x500-directory-replication|replication agreements]]:
- Shadow subject (unit of replication), update mode, supplier/consumer role
- Last and next update time, update schedule
- Notifications for DOP errors, DOP completion, shadow update success, and shadow errors
- An `updateShadow` action to force an out-of-band update

### Known DSA / Known DUA Managed Objects

These track peer relationships: credentials for mutual authentication, association statistics (current active, accumulated, failed), request/reply counters, and time of last attempt/success. They are derived from `communicationsEntity` (X.723 | ISO/IEC 10165-5).

## Protocol Layering for Management

Management services are satisfied through a combination of:

1. **Directory services (DAP)** — managing user attributes and static operational attributes
2. **DOP** — updating knowledge references and operational bindings
3. **DISP** — managing shadowing replication propagation
4. **CMIP/CMIS** — managing DSA operational state, event reporting, and performance counters

A **management workstation** integrates both a CMIS manager and a Directory User Agent to present a unified interface to the DSA administrator. The interface between a DSA and its collocated CMIS agent is explicitly *not standardized* by X.530 — it is local.

## ASN.1 and GDMO Definitions

Annex A provides full **Guidelines for Definition of Managed Objects (GDMO)** specifications for all managed object classes, name bindings, packages, notifications, parameters, and behaviours. The corresponding [[reflect/concepts/asn1|ASN.1]] module (`DirectoryManagement`, registered under `joint-iso-itu-t ds(5) module(1) directoryManagement(27) 6`) defines all attribute syntaxes and object identifier arcs used by the managed object definitions.

Key OID arc structure:
- `id-mac` — actions
- `id-mat` — attributes
- `id-moc` — managed object classes
- `id-mnb` — name bindings
- `id-mp` — packages
- `id-mpa` — notification parameters
