RFC 2252 (Standards Track, December 1997) defines the [[reflect/concepts/ldap|LDAPv3]] schema framework — the formal grammar for describing attribute types, object classes, [[reflect/concepts/ldap-syntaxes|syntaxes]], and [[reflect/concepts/ldap-matching-rules|matching rules]]. Authored by [[reflect/entities/mark-wahl|Mark Wahl]] (Critical Angle Inc.), [[reflect/entities/andy-coulbeck|Andy Coulbeck]] ([[reflect/entities/isode-ltd|Isode Inc.]]), [[reflect/entities/tim-howes|Tim Howes]] ([[reflect/entities/netscape-communications|Netscape Communications Corp.]]), and [[reflect/entities/steve-kille|Steve Kille]] (Isode Limited), it was a companion to [[reflect/sources/2026-05-02-rfc2251-txt|RFC 2251]] in the original LDAPv3 specification cluster. In 2006, its content was divided: information model material (attribute type and object class description grammars) migrated to RFC 4512, and the syntax/matching rule definitions went to [[reflect/sources/2026-05-02-rfc4517-txt|RFC 4517]]. The full decomposition is documented in [[reflect/synthesis/ldap-technical-specification-architecture|LDAPv3 Specification Architecture: The RFC 4510 Modular Reorganization]].

## IESG Authentication Caveat

Like [[reflect/sources/2026-05-02-rfc2251-txt|RFC 2251]], RFC 2252 carries the same IESG Note warning that update operations are unsupported by any mandatory authentication mechanism. Implementations "UNLIKELY TO INTEROPERATE, or MAY INTEROPERATE ONLY IF AUTHENTICATION IS REDUCED TO AN UNACCEPTABLY WEAK LEVEL." This gap was closed nine years later by [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]], whose historical arc is the subject of [[reflect/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture: Authentication Gap to RFC 4513]].

## Schema Framework

RFC 2252 defines formal BNF grammars for four types of schema definitions, each identified by a numeric OID and an optional short name. Schema elements are published by servers in the `subschema` entry, accessible via the `subschemaSubentry` operational attribute on any entry.

### Attribute Type Description

```
AttributeTypeDescription = "(" whsp
    numericoid whsp             ; AttributeType identifier
    [ "NAME" qdescrs ]          ; name used in AttributeType
    [ "DESC" qdstring ]
    [ "OBSOLETE" whsp ]
    [ "SUP" woid ]              ; derived from this AttributeType
    [ "EQUALITY" woid ]         ; Matching Rule name
    [ "ORDERING" woid ]         ; Matching Rule name
    [ "SUBSTR" woid ]           ; Matching Rule name
    [ "SYNTAX" whsp noidlen whsp ]
    [ "SINGLE-VALUE" whsp ]     ; default multi-valued
    [ "COLLECTIVE" whsp ]
    [ "NO-USER-MODIFICATION" whsp ]
    [ "USAGE" whsp AttributeUsage ]
    whsp ")"
```

`AttributeUsage` values: `userApplications` (default), `directoryOperation`, `distributedOperation`, `dSAOperation`.

### Object Class Description

```
ObjectClassDescription = "(" whsp
    numericoid whsp             ; ObjectClass identifier
    [ "NAME" qdescrs ]
    [ "DESC" qdstring ]
    [ "OBSOLETE" whsp ]
    [ "SUP" oids ]              ; Superior ObjectClasses
    [ ( "ABSTRACT" / "STRUCTURAL" / "AUXILIARY" ) whsp ]
    [ "MUST" oids ]             ; required AttributeTypes
    [ "MAY" oids ]              ; permitted AttributeTypes
    whsp ")"
```

Default class type is `STRUCTURAL`. [[reflect/concepts/ldap-object-classes|Object classes]] and their three types are covered in detail in the concept page.

### Matching Rule and Syntax Descriptions

Matching rules are declared with a numeric OID, optional name, optional DESC, and the assertion syntax OID. Syntaxes are declared with an OID and an optional DESC string. Both are stored in the `subschema` entry for server-side schema publication.

## Attribute Syntaxes

RFC 2252 defines approximately 50 syntaxes identified by OID. Each is classified as human-readable (H-R) or binary. Binary syntaxes — Certificate, Audio, Fax, JPEG, Binary — must be transferred using the binary encoding option (`attributeName;binary`).

Key syntaxes defined (the OID arc `1.3.6.1.4.1.1466.115.121.1` is abbreviated as `...` below):

| Syntax | OID | H-R |
|--------|-----|-----|
| Directory String | ….15 | Y |
| DN | ….12 | Y |
| Generalized Time | ….24 | Y |
| Integer | ….27 | Y |
| Boolean | ….7 | Y |
| OID | ….38 | Y |
| Printable String | ….44 | Y |
| Country String | ….11 | Y |
| IA5 String | ….26 | Y |
| Telephone Number | ….50 | Y |
| Postal Address | ….41 | Y |
| Bit String | ….6 | Y |
| Numeric String | ….36 | Y |
| Octet String | ….40 | Y |
| UTC Time | ….53 | Y |
| Certificate | ….8 | N |
| Certificate List | ….9 | N |
| JPEG | ….28 | N |
| Binary | ….5 | N |
| Supported Algorithm | ….49 | N |
| Fax | ….23 | N |
| Attribute Type Description | ….3 | Y |
| Object Class Description | ….37 | Y |
| Matching Rule Description | ….30 | Y |
| LDAP Syntax Description | ….54 | Y |

An upper bound may be appended to a syntax OID in an attribute type definition as `{N}` (e.g., `….15{64}`). This bound is advisory — servers may allow longer values — and is not part of the syntax's OID.

The Directory String syntax uses UTF-8 encoding of ISO 10646/Unicode. A single character may occupy multiple bytes; a `{64}`-bounded attribute may be more than 64 bytes on the wire.

## Operational Attributes

RFC 2252 defines two groups of operational attributes, returned only when explicitly requested by clients.

### Standard Operational Attributes (from X.501)

| Attribute | OID | Purpose |
|-----------|-----|---------|
| `createTimestamp` | 2.5.18.1 | Generalized Time of entry creation |
| `modifyTimestamp` | 2.5.18.2 | Generalized Time of last modification |
| `creatorsName` | 2.5.18.3 | DN of the creator |
| `modifiersName` | 2.5.18.4 | DN of the last modifier |
| `subschemaSubentry` | 2.5.18.10 | DN of the subschema entry for this entry |
| `attributeTypes` | 2.5.21.5 | Attribute type definitions (subschema) |
| `objectClasses` | 2.5.21.6 | Object class definitions (subschema) |
| `matchingRules` | 2.5.21.4 | Matching rule definitions (subschema) |
| `matchingRuleUse` | 2.5.21.8 | Matching rule applicability (subschema) |

`createTimestamp`, `modifyTimestamp`, `creatorsName`, and `modifiersName` use `NO-USER-MODIFICATION` — they are maintained exclusively by the server, not writable by clients.

### LDAP Operational Attributes (Root DSE only)

These attributes appear only in the Root DSE — the entry with the zero-length DN:

| Attribute | OID | Purpose |
|-----------|-----|---------|
| `namingContexts` | 1.3.6.1.4.1.1466.101.120.5 | Naming contexts mastered or shadowed by this server |
| `altServer` | 1.3.6.1.4.1.1466.101.120.6 | Alternate server URLs if this server is unavailable |
| `supportedExtension` | 1.3.6.1.4.1.1466.101.120.7 | OIDs of supported extended operations |
| `supportedControl` | 1.3.6.1.4.1.1466.101.120.13 | OIDs of supported [[reflect/concepts/ldap-controls|controls]] |
| `supportedSASLMechanisms` | 1.3.6.1.4.1.1466.101.120.14 | Names of supported [[reflect/concepts/sasl|SASL]] mechanisms |
| `supportedLDAPVersion` | 1.3.6.1.4.1.1466.101.120.15 | LDAP protocol versions implemented |
| `ldapSyntaxes` | 1.3.6.1.4.1.1466.101.120.16 | Syntaxes implemented (subschema attribute) |

`namingContexts` describes the portions of the directory information tree the server holds. If a server contains the entire directory, this attribute has a single value: the empty string (the null DN of the root).

## Object Classes

RFC 2252 defines two object classes:

**`extensibleObject`** (`1.3.6.1.4.1.1466.101.120.111`, `AUXILIARY`) — An auxiliary class whose implicit `MAY` list is the complete set of all attributes. An entry carrying `extensibleObject` may hold any attribute while the `MUST` requirements of its other object classes still apply. Not all servers implement this class; those that do not will reject `add` or `modify` operations that include it.

**`subschema`** (`2.5.20.1`, `AUXILIARY`) — Used in the subschema entry to aggregate all schema-publishing attributes: `dITStructureRules`, `nameForms`, `ditContentRules`, `objectClasses`, `attributeTypes`, `matchingRules`, `matchingRuleUse`. The `ldapSyntaxes` operational attribute may also appear. See [[reflect/concepts/ldap-object-classes|LDAP Object Classes]] for context on auxiliary class semantics.

## Matching Rules

RFC 2252 specifies matching rules across four categories.

**Equality** (Search/Compare operations):

| Rule | OID |
|------|-----|
| `objectIdentifierMatch` | 2.5.13.0 |
| `distinguishedNameMatch` | 2.5.13.1 |
| `caseIgnoreMatch` | 2.5.13.2 |
| `numericStringMatch` | 2.5.13.8 |
| `caseIgnoreListMatch` | 2.5.13.11 |
| `integerMatch` | 2.5.13.14 |
| `bitStringMatch` | 2.5.13.16 |
| `telephoneNumberMatch` | 2.5.13.20 |
| `presentationAddressMatch` | 2.5.13.22 |
| `uniqueMemberMatch` | 2.5.13.23 |
| `protocolInformationMatch` | 2.5.13.24 |
| `generalizedTimeMatch` | 2.5.13.27 |
| `caseExactIA5Match` | 1.3.6.1.4.1.1466.109.114.1 |
| `caseIgnoreIA5Match` | 1.3.6.1.4.1.1466.109.114.2 |

For `caseIgnoreMatch`, `caseIgnoreListMatch`, `telephoneNumberMatch`, `caseExactIA5Match`, and `caseIgnoreIA5Match`: multiple adjacent whitespace characters are treated as a single space, and leading/trailing whitespace is ignored.

**Ordering** (inequality filters `>=` / `<=`):
- `generalizedTimeOrderingMatch` (2.5.13.28)
- `caseIgnoreOrderingMatch` (2.5.13.3) — sort order is implementation-dependent

**Substrings** (substring filters):
- `caseIgnoreSubstringsMatch` (2.5.13.4)
- `telephoneNumberSubstringsMatch` (2.5.13.21)
- `numericStringSubstringsMatch` (2.5.13.10)

The Substring Assertion syntax (`….58`) is used only as the assertion value syntax in extensible match; it is not used as an attribute value syntax.

**Subschema** (required for modifiable subschema entries):
- `integerFirstComponentMatch` (2.5.13.29) — compares assertion INTEGER against the first component of a structured value
- `objectIdentifierFirstComponentMatch` (2.5.13.30) — compares assertion OID against the first component

These "first-component" rules have assertion syntaxes (INTEGER, OID) that differ from the value syntaxes of the attributes they govern — a notable exception to the usual rule that assertion and value syntaxes match.

## Security Considerations

RFC 2252 notes that LDAP string representations of attribute values are not always reversible to the same BER/DER form. A distinguished name round-tripped through LDAP string encoding may not reproduce the original DER bytes. Applications requiring the DER form — such as X.509 certificate verification — SHOULD use Binary syntax rather than the string representation. This non-canonicality limitation is inherited by all successor documents and is discussed in [[reflect/concepts/ldap-syntaxes|LDAP Attribute Syntaxes]].
