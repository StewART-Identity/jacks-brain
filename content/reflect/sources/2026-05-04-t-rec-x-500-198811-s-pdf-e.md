**ITU-T Recommendation X.500** (November 1988, originally CCITT Melbourne 1988 session) is the foundational overview for the entire X.500 directory series — published jointly with ISO as ISO 9594-1. It introduces the Directory concept, defines the core information model ([[reflect/concepts/directory-information-tree|DIB/DIT]]), describes the abstract directory service, and overviews the [[reflect/concepts/x500-distributed-directory|distributed directory model]]. Every other recommendation in the series (X.501 through X.525) builds on this framework.

## Background and Purpose

The Directory was designed as a foundational OSI infrastructure service addressing three needs:

- **User-friendly naming** — mapping human-readable names to network addresses so that application entities, people, and devices can be referred to without knowledge of underlying addresses
- **Name-to-address mapping** — dynamic binding between names and locations, allowing OSI networks to be "self-configuring": additions, removals, and location changes don't disrupt operation
- **Multi-application support** — a single unified namespace serving many applications simultaneously; one object's information maintained once and shared across authentication, messaging, and lookup applications

The document explicitly frames the Directory as distinct from a general-purpose database: query frequency far exceeds update frequency, update rates are governed by organizational change (not network dynamics), and transient inconsistencies from partially-propagated updates are acceptable.

## The Directory Information Base

The **Directory Information Base (DIB)** is the complete set of information held in the Directory, organized as the [[reflect/concepts/directory-information-tree|Directory Information Tree (DIT)]] — a hierarchy in which higher entries represent large-scope objects (countries, organizations) and lower entries represent fine-grained objects (people, application entities).

Each entry consists of **attributes**, each with a type and one or more values. Entry membership in an **object class** determines which attribute types are required or permitted. **Schema rules** ensure the DIB remains well-formed as entries change over time. Every entry has a **[[reflect/concepts/distinguished-name|distinguished name]] (DN)** — a unique hierarchical identifier built from the entry's relative distinguished name (RDN) appended to its superior's DN.

## The Abstract Directory Service

X.500 defines the directory service in terms of what a DUA can request. The full formal service definition is in X.511; X.500 provides the overview.

### Interrogation Operations

| Operation | Description |
|-----------|-------------|
| **Read** | Return attribute values from a named entry |
| **Compare** | Test whether a named entry has a specific attribute value, without returning it |
| **List** | Return the immediate subordinate entries of a named entry |
| **Search** | Return entries within a DIT subtree satisfying a filter |
| **Abandon** | Cancel an outstanding request |

Compare is architecturally notable: it confirms or denies whether a value matches (e.g., password verification) without exposing the stored value to the requestor — a meaningful access control design that LDAP inherited directly.

### Modification Operations

| Operation | Description |
|-----------|-------------|
| **Add Entry** | Add a new leaf entry (object or alias) to the DIT |
| **Remove Entry** | Remove a leaf entry from the DIT |
| **Modify Entry** | Apply a sequence of attribute changes atomically — all succeed or none |
| **Modify RDN** | Change the relative distinguished name of a leaf entry |

The 1988 edition restricts Add/Remove to leaf entries only; the document acknowledges full subtree operations would require future enhancement.

### Service Qualification

Requests are qualified with **service controls** limiting resource consumption: time limits, size limits, search scope, priority, and interaction mode preferences (chaining vs. referral). Requests may carry **security parameters** including digital signatures. **Filters** attached to Search operations restrict results to entries satisfying attribute conditions.

These controls map directly to the `sizelimit`, `timelimit`, and `scope` parameters in [[reflect/concepts/ldap|LDAP]] search operations — LDAP preserved the service qualification concept while simplifying the underlying mechanism.

## Authentication

X.500 defines two authentication tiers:

**Simple authentication** — The Directory holds a `User Password` attribute in entries for authenticating users. A service verifies identity by requesting a Compare against the stored password. Optional one-way function application protects against replay. This is the direct ancestor of LDAP simple bind.

**Strong authentication** — Based on public-key cryptography, with the Directory acting as a repository for users' public encryption keys. The detailed procedures are specified in companion **Recommendation X.509**, co-developed and co-published as part of the same 1988 X.500 release set. This origin is significant: [[reflect/concepts/x509-pki|X.509 PKI]] began not as an internet security standard but as the authentication mechanism for a directory system.

## Distributed Directory Model

X.500 envisions a widely distributed directory spanning organizations and telecommunication carriers. The detailed model is covered in [[reflect/concepts/x500-distributed-directory|X.500 Distributed Directory Model]]; in brief:

- **Directory System Agents (DSAs)** are the server-side processes holding portions of the DIB; they cooperate via the Directory System Protocol (DSP) specified in X.519
- **Directory User Agents (DUAs)** represent users, interacting with DSAs via the Directory Access Protocol (DAP)
- **Directory Management Domains (DMDs)** group DSAs under single organizational management — Administration DMDs (ADDMDs) operated by telecommunications carriers, Private DMDs (PRDMDs) by enterprises
- DSAs cooperate through **chaining** (forwarding requests transparently), **multicasting** (broadcasting to multiple DSAs), and **referrals** (directing clients to alternative access points)

## Patterns of Use

Annex A catalogues use patterns that remain relevant in modern LDAP deployments:

- **Look-up** — supply a DN and attribute type, receive a value; the classic "white pages" function and the most frequent directory query type
- **User-friendly naming** — names composed of attributes inherent to the object (country + organization + common name), shared across all applications referencing the object
- **Browsing** — wander the DIB via List and Search when a complete name is unavailable ("know it when you see it")
- **"Yellow Pages"** — find objects by attribute category rather than known identity; implemented via filtering on attributes like `businessCategory`, or via dedicated alias-entry subtrees organized for category lookup
- **Groups** — objects with multi-valued `Member` attributes encoding sets with changing membership; checked via Compare, enumerated via Read
- **Authentication** — directory as credential store for both simple (password) and strong (X.509) authentication

## Relationship to Later Editions and LDAP

The 1988 recommendation is the 1st edition of X.500. Subsequent editions (1993, 1997, 2001) substantially elaborated access control (X.501), the abstract service (X.511), distributed operations (X.518), protocol specifications (X.519), and PKI (X.509). Corrections to the 3rd and 4th editions are documented in the [[reflect/sources/2026-05-04-t-rec-x-imp500-200109-i-msw-e|Directory Implementors' Guide Version 15]].

[[reflect/concepts/ldap|LDAP]] inherits the X.500 data model — DIT, entries, attributes, object classes, distinguished names — substantially from this specification and its successors. The [[reflect/concepts/x500-user-schema|X.500 user schema]] that LDAP adopted (via RFC 2256 → RFC 4519) traces to X.520 and X.521, co-published in the same 1988 release. The key departures: LDAP replaces the OSI DAP with a TCP/IP protocol, and LDAP clients handle referrals directly rather than relying on server-side chaining or multicasting.
