RFC 4516 defines the [[reflect/concepts/ldap-url|LDAP URL]] format — the `ldap://` URI scheme for encoding [[reflect/concepts/ldap|LDAP]] search operations and referral targets in a human-readable, bookmarkable string. Published June 2006, edited by [[reflect/entities/mark-smith|Mark Smith]] of Pearl Crescent, LLC with co-author [[reflect/entities/tim-howes|Tim Howes]] of Opsware, Inc. It is part of the [[reflect/sources/2026-05-02-rfc4510-txt|RFC 4510]] series and obsoletes RFC 2255 (December 1997).

## Purpose and Scope

An LDAP URL encodes either:

- A **search operation** — host, port, base DN, attributes to return, scope, and filter — that a client resolves by opening an LDAP session and executing the described search
- A **referral target** — a service to which a client should redirect an LDAP operation already in progress

Not all parameters of an LDAP search (as defined in [[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]]) can be expressed in the URL format. In particular, controls and most extended request fields have no URL representation.

## URL Grammar

The LDAP URL grammar follows ABNF (RFC 4234):

```
ldapurl = scheme COLON SLASH SLASH [host [COLON port]]
          [SLASH dn
            [QUESTION [attributes]
              [QUESTION [scope]
                [QUESTION [filter]
                  [QUESTION extensions]]]]]

scheme     = "ldap"
dn         = distinguishedName     ; RFC 4514 string form
attributes = attrdesc *(COMMA attrdesc)
attrdesc   = selector *(COMMA selector)
selector   = attributeSelector     ; RFC 4511 §4.5.1
scope      = "base" / "one" / "sub"
extensions = extension *(COMMA extension)
extension  = [EXCLAMATION] extype [EQUALS exvalue]
extype     = oid                   ; RFC 4512
exvalue    = LDAPString            ; RFC 4511 §4.1.2
```

The `?` (QUESTION) character delimits each URL component in sequence. Trailing `?` separators may be omitted if all following components are also absent. Note that `<host>` may contain literal IPv6 addresses per RFC 3986 §3.2.2.

## URL Components

| Component | Description | Default when absent |
|-----------|-------------|---------------------|
| `host` | LDAP server hostname or IP | Client uses prior knowledge |
| `port` | TCP port | 389 |
| `dn` | Base DN for the search (RFC 4514 string form) | Zero-length DN `""` |
| `attributes` | Comma-separated list of attributes to return | All user attributes (`*`) |
| `scope` | Search scope: `base`, `one`, or `sub` | `base` |
| `filter` | Search filter (RFC 4515 string form) | `(objectClass=*)` |
| `extensions` | Extension type=value pairs | None |

## Percent-Encoding

A generated LDAP URL MUST consist only of characters from RFC 3986's `<reserved>`, `<unreserved>`, or `<pct-encoded>` productions. An octet MUST be percent-encoded in two situations:

1. It falls outside both the reserved and unreserved sets defined by RFC 3986 §§2.2–2.3
2. It is the `?` character appearing inside a `<dn>`, `<filter>`, or other component (rather than as a component delimiter)
3. It is a `,` character inside an `<exvalue>`

RFC 4516 also notes that after the update to RFC 3986 (from the earlier RFC 2396), two additional characters must be percent-encoded everywhere in LDAP URL data: `[` and `]`.

Implementations SHOULD accept valid UTF-8 strings as input even if they contain characters outside the strict URL production rules.

## Extension Mechanism

The `<extensions>` component provides forward extensibility. Extensions are comma-separated `type=value` pairs (the `=value` part is optional for flag-style extensions):

- **Critical extension**: prefixed with `!` — if the client does not implement the extension, it MUST NOT process the URL
- **Non-critical extension**: no prefix — if the client does not implement the extension, it MUST ignore it and continue processing

Extension types are OIDs in either dotted-decimal (`1.2.3.4`) or descriptor form (`myExtension`). Descriptor form SHOULD be restricted to registered names per RFC 4520. No extensions are defined in RFC 4516 itself; the mechanism exists for future specifications.

## Illustrative Examples

RFC 4516 provides a rich set of examples demonstrating the grammar:

```
# Base search of a DN on any server (client chooses)
ldap:///o=University%20of%20Michigan,c=US

# Same search on a specific server
ldap://ldap1.example.net/o=University%20of%20Michigan,c=US

# Return only postalAddress
ldap://ldap1.example.net/o=University%20of%20Michigan,c=US?postalAddress

# Subtree search on non-default port, any attributes
ldap://ldap1.example.net:6666/o=University%20of%20Michigan,c=US??sub?(cn=Babs%20Jensen)

# Percent-encoding '?' inside a DN value
ldap://ldap2.example.com/o=Question%3f,c=US?mail

# Binary value in filter (backslash-hex LDAP escaping, then percent-encoding the backslashes)
ldap://ldap3.example.com/o=Babsco,c=US???(%34-octet=%5c00%5c00%5c00%5c04)

# Three equivalent URLs (all point to root DSE)
ldap://ldap.example.net
ldap://ldap.example.net/
ldap://ldap.example.net/?
```

The examples demonstrate a common source of confusion: LDAP filter escaping (`\xx`) and URL percent-encoding (`%xx`) stack — the `\` character in an LDAP filter value must itself be percent-encoded as `%5c` in the URL.

## Security Considerations

RFC 4516 identifies three security concerns:

1. **Policy-controlled session establishment**: Clients SHOULD have a user-configurable policy controlling which servers they will connect to and under what security mechanisms, since LDAP URLs are often resolved automatically without user interaction.

2. **Authentication information and privacy**: Without an explicit policy permitting it, clients should use anonymous LDAP sessions. Reusable passwords (simple bind) sent to a server reached via URL are particularly risky and SHOULD NOT be used unless explicitly permitted by policy. Confirmation by the user is appropriate for any URL that causes authentication information to be sent.

3. **Unexpected search results**: Following an LDAP URL may retrieve large amounts of data or initiate long-lived searches. Security implications are the same as resolving an LDAP search query directly.

RFC 4516 defers detailed authentication guidance to [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]].

## Changes from RFC 2255

Key technical changes (see Appendix A):

- Revised ABNF to use common productions from RFC 4512
- Replaced RFC 2396 references with RFC 3986 (enabling literal IPv6 in `<host>`)
- `<attrdesc>` now references `<attributeSelector>` from RFC 4511, enabling `*` in the attributes component
- `<extype>` is now an OID from RFC 4512 (previously ad hoc)
- Removed the **bindname extension** (no known implementations)
- Added requirement that critical extensions prevent URL processing if unimplemented (aligned with RFC 4511's controls semantics)
- Added "Defaults for Fields" as a separate section
