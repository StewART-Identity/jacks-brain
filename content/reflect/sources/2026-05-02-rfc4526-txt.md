RFC 4526 extends [[reflect/concepts/ldap|LDAPv3]] to support absolute True and False assertions in Search filters by permitting empty `and` and `or` filter sets — a capability present in the X.500 Directory Access Protocol (DAP) that was inadvertently excluded from LDAPv3.

## Background: The LDAPv2 Gap

X.500 DAP [X.511] supports absolute True and False assertions. The rule is simple:

- An `and` filter with **zero elements** always evaluates to **True**
- An `or` filter with **zero elements** always evaluates to **False**

LDAPv2 (RFC 1777) placed no restriction on the number of filter elements, but the LDAPv2 string representation (RFC 1960) could not represent empty `and` or `or` sets. As a consequence, when LDAPv3 was drafted, absolute True and False filters were excluded — a practical problem for DAP-to-LDAP gateway implementations.

The gap matters for DSA-specific Entries (DSEs) — directory server administrative entries that do not necessarily carry `objectClass` attributes. A presence filter like `(objectClass=*)` may legitimately evaluate to False on these entries, leaving gateways that need to pass absolute True or False assertions with no LDAPv3 equivalent to fall back on.

## The Extension

RFC 4526 restores this capability by amending two existing specifications:

- **[[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]]** (the protocol): `and` and `or` filter choices are now explicitly permitted with zero filter elements.
- **[[reflect/sources/2026-05-02-rfc4515-txt|RFC 4515]]** (string representation): empty filter lists are now valid in the string syntax. The resulting string forms are:
  - `(&)` → absolute **True** filter
  - `(|)` → absolute **False** filter

Servers implementing this extension SHOULD advertise OID `1.3.6.1.4.1.4203.1.5.3` in the `supportedFeatures` attribute of the root DSE (defined in [[reflect/sources/2026-05-02-rfc4512-txt|RFC 4512]]). Clients SHOULD NOT use the feature unless they have confirmed server support via this advertisement.

## Relationship to the LDAP Extension Landscape

RFC 4526 follows the same **feature-OID + grammar extension** pattern as [[reflect/sources/2026-05-02-rfc4529-txt|RFC 4529]] (`@classname` attribute selection). Both extensions:

- Add new syntactic forms to existing LDAP grammar productions without introducing new controls or operations
- Gate client use behind a `supportedFeatures` OID for capability negotiation
- Use OIDs in the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]]'s `.5` feature sub-arc (`1.3.6.1.4.1.4203.1.5`)

RFC 4529 registered OID `.5.2`; RFC 4526 registers `.5.3` — adjacent in the same feature-OID sub-arc, and both authored by [[reflect/entities/kurt-zeilenga|Zeilenga]].

The extension introduces no new wire-format state, no new controls, and no new extended operations — it is purely a semantic extension to filter evaluation rules and a syntactic extension to the string representation of [[reflect/concepts/ldap-search-filters|LDAP search filters]].

## Security Considerations

RFC 4526 explicitly states the extension introduces no new security considerations. General LDAPv3 security considerations from [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]] apply.
