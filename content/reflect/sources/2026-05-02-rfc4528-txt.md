RFC 4528 (Standards Track, June 2006) defines the [[reflect/concepts/ldap-assertion-control|LDAP Assertion Control]], authored by [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]]. The control enables conditional execution of [[reflect/concepts/ldap|LDAP]] operations: a client attaches a [[reflect/concepts/ldap-search-filters|Filter]] to any request, and the server only processes the request if that filter evaluates to TRUE against the target entry. If the filter evaluates to FALSE or Undefined, the server returns `assertionFailed` (result code 122) and takes no action.

## The Assertion Control

The control is identified by `controlType` OID `1.3.6.1.1.12` — an IANA-assigned OID in the LDAP-specific `1.3.6.1.1` arc (not in the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]]'s private enterprise arc). Its `controlValue` is a BER-encoded `Filter`, the same ASN.1 type used in [[reflect/concepts/ldap-search-filters|Search operations]] (RFC 4511 §4.5.1). The criticality may be TRUE or FALSE — the RFC does not mandate a specific value.

There is no corresponding response control.

### Conditional Logic

| Filter Evaluation | Result |
|---|---|
| TRUE | Operation proceeds normally |
| FALSE | `assertionFailed` (122) returned; no action taken |
| Undefined | `assertionFailed` (122) returned; no action taken |

The strict treatment of Undefined as a failure (rather than mapping it to FALSE and returning normally) is significant for security: if the filter cannot be evaluated — because an attribute is absent, a matching rule is unavailable, or the evaluation is indeterminate — the operation is blocked rather than silently proceeding.

### Applicable Operations

The control is appropriate for both interrogation and update operations:

- **Add** — target is the `entry` field of AddRequest
- **Compare** — target is the `entry` field; assertion and compare evaluate atomically
- **Delete** — target is the `DelRequest` type
- **Modify** — target is the `object` field; assertion and modify execute atomically
- **ModifyDN (rename)** — target is the `entry` field; assertion and rename execute atomically
- **Search** — target is the `baseObject`; assertion is evaluated after "finding" the base but before "searching"; no entries or continuation references are returned if the assertion fails

The control is **not appropriate** for Abandon, Bind, Unbind, and StartTLS.

## Atomic Semantics

For Compare and all update operations (Add, Delete, Modify, ModifyDN), the evaluation of the assertion and the normal processing of the operation SHALL be done as one atomic action. This is the mechanism that enables "test and set" and "test and clear" patterns — for example, modifying an attribute only if it currently holds a specific value, without a separate read-then-write round trip that could race with another client.

For Search, atomicity is not required in the same sense: the assertion gates whether the search proceeds at all, but the search itself is not a mutating operation.

## Result Code: assertionFailed (122)

The RFC registers a new LDAP result code, `assertionFailed` (122), via IANA. This result code is returned whenever the filter does not evaluate to TRUE, regardless of whether the filter evaluated to FALSE or Undefined. The code is distinct from any pre-existing result codes and provides a clear signal to clients that the operation was blocked by the assertion condition rather than by an access-control, schema, or protocol error.

## Security Considerations

The RFC notes three security concerns:

1. **Filter confidentiality**: filters may contain sensitive information (attribute values, distinguished names) and should be protected by the transport layer (e.g., [[reflect/concepts/ldap-tls|TLS]]).
2. **Directory content inference**: the control can be used to probe directory content by testing assertions and observing whether operations succeed or fail — it SHOULD be subject to appropriate access controls.
3. **Evaluation complexity**: complex filters may be expensive to evaluate; the mechanism SHOULD be subject to administrative rate-limiting or complexity controls.

## Provenance

The assertion control concept is attributed to Morteza Ansari. The RFC was authored solely by [[reflect/entities/kurt-zeilenga|Zeilenga]] and published as part of the broader 2006 LDAP standards effort alongside the [[reflect/sources/2026-05-02-rfc4510-txt|RFC 4510]] series, though it is not itself part of the nine-document technical specification. It extends the [[reflect/concepts/ldap-controls|controls mechanism]] defined in [[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]] §4.1.11.

Discovery: servers implementing the control SHOULD publish OID `1.3.6.1.1.12` in the `supportedControl` attribute of their root DSE (per [[reflect/sources/2026-05-02-rfc4512-txt|RFC 4512]]). A server MAY advertise the control only when the client is authorized to use it.
