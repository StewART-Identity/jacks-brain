RFC 4533 (Experimental, June 2006) defines the **LDAP Content Synchronization Operation** — informally called the *Sync Operation* or *SyncRepl* — authored by [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]] ([[reflect/entities/openldap-foundation|OpenLDAP Foundation]]) and [[reflect/entities/jong-hyuk-choi|Jong Hyuk Choi]] ([[reflect/entities/ibm-corporation|IBM Corporation]]). The operation allows an [[reflect/concepts/ldap|LDAP]] client to maintain a synchronized copy of a fragment of the [[reflect/concepts/directory-information-tree|Directory Information Tree]] (DIT), supporting both polling for changes and listening for changes.

## Background and Status

RFC 4533 carries the unusual status of **Experimental** — it does not represent IETF consensus. The IESG note explicitly states that the LDUP (LDAP Duplication/Replication/Update Protocols) working group reached consensus on a *different* approach: RFC 3928 (LDAP Client Update Protocol, LCUP), which is on the standards track. RFC 4533 was published alongside the RFC 4510 series but stands apart — it was not part of the coordinated LDAPv3 specification reorganization that produced [[reflect/sources/2026-05-02-rfc4510-txt|RFC 4510]]–4519.

The [[reflect/entities/openldap-foundation|OpenLDAP Foundation]] holds the OID arc `1.3.6.1.4.1.4203.1.9.1` used by all of RFC 4533's protocol elements — the same IANA-assigned private enterprise arc used for the Password Modify and attribute selection extensions.

## Design Goals

RFC 4533 was designed to avoid the shortcomings of earlier synchronization proposals:

- **No pre-arranged synchronization agreements** required
- **No server-side history required** — operates on current state alone (though servers may optionally maintain history for efficiency)
- **No per-client state required** on the server (though optional for efficiency)
- **Bounded traffic** — exchange size bounded by content size, not unbounded history

The operation achieves *eventual convergence*: after each synchronization stage completes, the client copy either matches the server's content or the client is explicitly told a full reload is required.

## Modes of Operation

The Sync Operation extends the LDAP Search operation via [[reflect/concepts/ldap-controls|LDAP Controls]] and runs in one of two modes, specified in the Sync Request Control:

### refreshOnly (Polling)

The client issues a search with the Sync Request Control set to `refreshOnly`. Each cycle:

1. Server returns changed entries with state `add` (carrying full attributes)
2. Server identifies unchanged entries via state `present` (empty entries) or the more efficient `syncIdSet` Sync Info Messages
3. Server may use a **delete phase** to enumerate entries no longer in content instead of enumerating present entries
4. A `SearchResultDone` with Sync Done Control terminates the cycle, providing a new `syncCookie`

On the next poll, the client sends the `syncCookie` to resume incremental synchronization.

### refreshAndPersist (Listening)

The client issues one request and keeps the connection open:

1. **Refresh stage**: same as `refreshOnly` but ends with a Sync Info Message with `refreshDone = TRUE`
2. **Persist stage**: server sends real-time change notifications — `add`, `modify`, or `delete` state entries — as the DIT changes

The operation persists until canceled via the LDAP Cancel Operation (RFC 3909). A Sync Done Control is attached to the `SearchResultDone` when the operation terminates.

## Protocol Elements

| Element | OID | Type |
|---|---|---|
| Sync Request Control | `1.3.6.1.4.1.4203.1.9.1.1` | LDAP Control (criticality TRUE or FALSE) |
| Sync State Control | `1.3.6.1.4.1.4203.1.9.1.2` | LDAP Control (criticality FALSE) |
| Sync Done Control | `1.3.6.1.4.1.4203.1.9.1.3` | LDAP Control (criticality FALSE) |
| Sync Info Message | `1.3.6.1.4.1.4203.1.9.1.4` | Intermediate Response Message |
| `e-syncRefreshRequired` | `4096` | LDAP Result Code |

All four OIDs fall under the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]]'s private enterprise arc (`1.3.6.1.4.1.4203`).

### syncCookie

The `syncCookie` is an opaque OCTET STRING carrying session state. Clients pass it back to the server on subsequent requests to continue incremental synchronization. The server embeds state information — typically a commit sequence number, change sequence number, or timestamp — but this is an implementation detail opaque to clients.

This design is stateless from the server's perspective: the server need not maintain a record of what it sent each client. The client's cookie carries sufficient state for the server to resume incremental synchronization.

### syncUUID

Entries are tracked by `entryUUID` — a stable 128-bit UUID that, unlike the [[reflect/concepts/distinguished-name|Distinguished Name]], does not change when an entry is renamed or moved. Clients MUST use `entryUUID` values as primary keys for correlating synchronization messages.

### Sync State Values

| State | Meaning |
|---|---|
| `present (0)` | Entry exists and is unchanged (empty entry, no attributes) |
| `add (1)` | Entry is new or has changed (full attributes returned) |
| `modify (2)` | Entry was modified within content (persist stage only) |
| `delete (3)` | Entry has been removed from content (empty entry) |

## Present Phase and Delete Phase

Content update polls may involve one or both of two phases for communicating the fate of unchanged-or-gone entries:

- **Present phase**: server sends `present` state markers for all *unchanged* entries. Client removes any prior-copy entry not identified as present.
- **Delete phase**: server sends `delete` state markers for entries that *left* content. More efficient than the present phase when fewer entries were deleted than remained unchanged — it avoids enumerating everything that stayed.

The `refreshDeletes` flag in the Sync Done Control indicates which phase ended the operation. The `syncIdSet` Sync Info Message coalesces multiple UUID notifications into a single message — a key optimization for reducing protocol chattiness.

## Synchronization Sessions

A sequence of Sync Operations where each request carries the previous response's `syncCookie` constitutes a **Synchronization Session**. The client MUST use the same content-controlling search parameters throughout the session. If the server does not recognize the cookie, it returns either initial content or `e-syncRefreshRequired`.

## Meta Information Considerations

The RFC notes that [[reflect/concepts/distinguished-name|entry DNs]] are meta information — renaming or moving an entry does not itself trigger a synchronization message unless the move causes the entry to enter or leave the synchronized content scope. Collective attributes (values shared across entry collections) SHOULD NOT be synchronized due to the multiplier effect: one modification would require transmitting updates for every member entry.

## Security Considerations

- Servers should guard against malicious or tampered cookie content; digital signatures in cookies are suggested.
- The `refreshAndPersist` persist stage may disclose more information than polling — a client can infer the sequence of update operations from the notification stream.
- Small directory updates can generate disproportionate synchronization traffic; servers should guard against abuse of update privileges as an indirect denial-of-service vector.
- The `e-syncRefreshRequired` result code provides a circuit-breaker when incremental synchronization becomes impractical or expensive.

## Appendix: CSN-Based Implementation

RFC 4533 includes a non-normative appendix on Change Sequence Number (CSN) based server implementations — targeted at servers that do not maintain change logs and must operate purely on current directory state. The server maintains a *context CSN* (the highest committed entry CSN not exceeding any outstanding uncommitted CSN). This context CSN is embedded in `syncCookie` values. Subsequent polls compare the cookie's embedded CSN against entry CSNs to determine what changed.
