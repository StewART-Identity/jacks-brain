RFC 4529 (Informational, June 2006) extends [[reflect/concepts/ldap|LDAP]]'s search operation to allow `@classname` shorthand in attribute lists, causing the server to return all attributes permitted by the named [[reflect/concepts/ldap-object-classes|object class]]. Authored by [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]], it is part of the RFC 4510 series that revised and replaced the original LDAPv3 RFCs (2251–2256).

## The Problem

[[reflect/concepts/ldap|LDAPv3]]'s search `attributeSelector` already offers two special descriptors:

| Descriptor | Meaning | Defined in |
|---|---|---|
| `*` | All user attributes | RFC 4511 |
| `+` | All operational attributes | RFC 3673 |

But there is no convenient way to request all attributes that entries of a particular object class carry. A client wanting all attributes for a `country` entry had to enumerate `c`, `searchGuide`, `description`, and `objectClass` individually.

## The Extension

RFC 4529 adds a third class of descriptor, the `objectclassdescription`, extending the `attributeSelector` ABNF from RFC 4511:

```
attributeSelector =/ objectclassdescription
objectclassdescription = ATSIGN oid options
ATSIGN = %x40  ; "@" U+0040
```

`@classname` in an attribute list expands to all attributes the named [[reflect/concepts/ldap-object-classes|object class]] permits — those required (`MUST`) or allowed (`MAY`), including attributes inherited via `SUP` (superclass) chains as defined in RFC 4512.

**Example**: `@country` expands to `c`, `searchGuide`, `description`, `objectClass` — all attributes of the `country` class (defined in RFC 4519).

**LDAP URL example**: `ldap:///dc=example,dc=com?@organization?base` requests all organization-class attributes without enumerating them — the scenario RFC 4529 identifies as its primary use case (a user entering an LDAP URL directly into a web browser).

The extension also applies wherever the `AttributeSelection` data type appears, including pre/post read controls (RFC 4527) and `<attributeSelector>` productions in LDAP URLs (RFC 4516).

## Error Handling

If the OID or short name is unrecognized, or does not refer to an object class, the `@descriptor` is treated as an unrecognized attribute description and silently ignored — no error is returned, no attributes are delivered for that descriptor. An `@descriptor` with an unrecognized option is similarly treated as unrecognized. The `options` field in the grammar exists for extensibility but RFC 4529 defines no options.

## Discovery and Compatibility

Servers supporting this feature SHOULD publish OID `1.3.6.1.4.1.4203.1.5.2` in the root DSE's `supportedFeatures` attribute (RFC 4512). Clients SHOULD NOT use the feature unless they have confirmed support via this OID. The OID sits in the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]]'s IANA-assigned private enterprise allocation.

## Security Considerations

RFC 4529 explicitly notes that `@classname` is purely a shorthand — it makes no attributes accessible that could not have been requested individually. No additional security considerations beyond those for the base LDAP search operation (RFC 4511) apply.
