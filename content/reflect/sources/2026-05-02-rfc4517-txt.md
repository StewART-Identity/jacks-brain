RFC 4517 is the normative specification of [[reflect/concepts/ldap-syntaxes|LDAP attribute syntaxes]] (data types) and [[reflect/concepts/ldap-matching-rules|matching rules]] (comparison semantics) for the Lightweight Directory Access Protocol. Edited by [[reflect/entities/steven-legg|Steven Legg]] of eB2Bcom and published in June 2006, it is part of the RFC 4510 technical specification series that replaced the original 1997 LDAPv3 RFCs (2251–2256).

The document obsoletes the relevant portions of RFC 2252 and RFC 2256, and updates RFC 3698. It is an integral part of the [[reflect/concepts/ldap|LDAP]] technical specification alongside RFC 4510 (road map), RFC 4511 (protocol), RFC 4512 (directory information models), RFC 4514 (DN string representation), [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]] (authentication), RFC 4518 (string preparation), RFC 4519 (user schema), and RFC 4520 (IANA considerations).

## Core Distinction: Syntaxes vs. Matching Rules

RFC 4517 draws a sharp boundary between two concepts that are often conflated:

- **Syntax** — defines the *structure and format* of attribute values; constrains what values are legal. The syntax defines how values are encoded in the LDAP protocol (LDAP-specific encoding), which differs from the BER encoding used by X.500.
- **Matching rule** — defines *comparison semantics*; specifies how attribute values are compared against assertion values in Search and Compare operations. A syntax does not imply its matching rules — the rules are separately defined and separately registered.

This separation allows, for example, the `caseIgnoreMatch` and `caseExactMatch` rules to both apply to the `Directory String` syntax: same legal values, different comparison semantics.

## Syntaxes Defined (34)

Each syntax has a unique OID (listed in RFC 4517 Appendix A) and an LDAP-specific encoding expressed as ABNF:

| Syntax | OID | Description |
|--------|-----|-------------|
| Directory String | 1.3.6.1.4.1.1466.115.121.1.15 | UTF-8 string of ≥1 UCS characters |
| DN | 1.3.6.1.4.1.1466.115.121.1.12 | Distinguished name (RFC 4514 encoding) |
| Generalized Time | 1.3.6.1.4.1.1466.115.121.1.24 | ISO 8601-based datetime |
| Integer | 1.3.6.1.4.1.1466.115.121.1.27 | Whole number, unlimited magnitude |
| Boolean | 1.3.6.1.4.1.1466.115.121.1.7 | `TRUE` / `FALSE` |
| OID | 1.3.6.1.4.1.1466.115.121.1.38 | Object identifier |
| Octet String | 1.3.6.1.4.1.1466.115.121.1.40 | Arbitrary byte sequence |
| JPEG | 1.3.6.1.4.1.1466.115.121.1.28 | JFIF-encoded image |
| Bit String | 1.3.6.1.4.1.1466.115.121.1.6 | Binary digit sequence |
| Country String | 1.3.6.1.4.1.1466.115.121.1.11 | Two-character ISO 3166 code |
| Printable String | 1.3.6.1.4.1.1466.115.121.1.44 | Latin alphanumeric + selected punctuation |
| IA5 String | 1.3.6.1.4.1.1466.115.121.1.26 | 7-bit ASCII |
| Numeric String | 1.3.6.1.4.1.1466.115.121.1.36 | Digits and spaces |
| Telephone Number | 1.3.6.1.4.1.1466.115.121.1.50 | E.123 international format |
| Postal Address | 1.3.6.1.4.1.1466.115.121.1.41 | Multi-line UCS address |
| UTC Time | 1.3.6.1.4.1.1466.115.121.1.53 | Deprecated; prefer Generalized Time |
| … | … | Plus schema-description and legacy types |

Each syntax maps explicitly to an ASN.1 type from X.501, X.520, or X.680, enabling interoperability between LDAP and X.500 directories.

LDAP-specific encodings are **not canonical**: a round-trip through BER and back may not reproduce the original octets. LDAP encodings must not be used where reversibility to DER is required (e.g., digital signature verification). Security-sensitive fields must be compared at the abstract value level, not the encoding level.

## Matching Rules Defined (32)

RFC 4517 defines 32 matching rules across three types used in [[reflect/concepts/ldap-search-filters|search filters]] and Compare operations:

**Equality rules** (evaluate TRUE when values match): `caseIgnoreMatch`, `caseExactMatch`, `distinguishedNameMatch`, `generalizedTimeMatch`, `objectIdentifierMatch`, `integerMatch`, `octetStringMatch`, `booleanMatch`, `telephoneNumberMatch`, `caseIgnoreIA5Match`, `uniqueMemberMatch`, `bitStringMatch`, `numericStringMatch`, `objectIdentifierFirstComponentMatch`, `directoryStringFirstComponentMatch`, `integerFirstComponentMatch`

**Ordering rules** (evaluate TRUE when attribute < assertion): `caseIgnoreOrderingMatch`, `caseExactOrderingMatch`, `generalizedTimeOrderingMatch`, `integerOrderingMatch`, `octetStringOrderingMatch`, `numericStringOrderingMatch`

**Substrings rules** (evaluate TRUE when pattern matches within value): `caseIgnoreSubstringsMatch`, `caseExactSubstringsMatch`, `caseIgnoreIA5SubstringsMatch`, `caseIgnoreListSubstringsMatch`, `telephoneNumberSubstringsMatch`, `numericStringSubstringsMatch`

String matching rules apply **string preparation algorithms** (RFC 4518) before comparison: Transcode, Map (case-folding varies by rule), Normalize, Prohibit, Check bidi, and Insignificant Character Handling. The specific Map and Insignificant Character Handling steps differ per rule.

## Relationship to X.500

RFC 4517 explicitly maps every LDAP syntax to its corresponding ASN.1 type from the X.500 series. This makes LDAP's type system a deliberate simplification of X.500's: LDAP uses human-readable LDAP-specific encodings on the wire while preserving the abstract type structure of X.500. Gateways between the two must transcode Directory String values per RFC 4518's Transcode step.
