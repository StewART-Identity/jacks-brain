RFC 4512 — "Lightweight Directory Access Protocol (LDAP): Directory Information Models" — is the normative specification of the X.500 Directory Information Models as adapted for LDAP. Authored by [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]] and published in June 2006, it is part of the [[reflect/sources/2026-05-02-rfc4510-txt|RFC 4510]] series that replaced the earlier 1997 LDAPv3 technical specification. It obsoletes sections of RFC 2251, RFC 2252, RFC 2256, and all of RFC 3674.

## Scope

The RFC describes how information is represented and organized in the Directory. It covers two primary models and their supporting mechanisms:

1. **Model of Directory User Information** — how user-visible data is organized
2. **Directory Administrative and Operational Information** — how servers manage and operate the directory
3. **Directory Schema** — the formal definitions governing structure and content
4. **DSA (Server) Informational Model** — what information a server exposes about itself

## Directory Information Tree

The [[reflect/concepts/directory-information-tree|Directory Information Tree (DIT)]] is a hierarchical tree whose vertices are directory entries. All information in the Directory Information Base (DIB) is held in entries organized within this tree.

Entries have three types:
- **Object entries** — represent real-world objects (people, organizations, devices)
- **Alias entries** — provide alternative names for objects via the `aliasedObjectName` attribute
- **Subentries** — hold administrative/operational data associated with a subtree

Parent–child relationships are called superior–subordinate. An entry's full name is its Distinguished Name (DN), formed by concatenating its Relative Distinguished Name (RDN) with the DN of its parent. RDNs must be unique among all siblings (immediate subordinates of the same parent).

## Object Classes

The RFC provides the definitive specification of [[reflect/concepts/ldap-object-classes|LDAP object classes]], organized into three kinds:

| Kind | Description |
|------|-------------|
| **ABSTRACT** | Base for other classes; entries cannot belong to an abstract class unless also in a structural or auxiliary subclass. The `top` abstract class is the root of all structural class hierarchies. |
| **STRUCTURAL** | Defines what the entry *is*; every entry has exactly one structural object class chain. The structural object class of an entry cannot be changed. |
| **AUXILIARY** | Augments entries with additional attributes; entries can belong to multiple auxiliary classes, subject to DIT content rules. |

Every entry must have an `objectClass` attribute. Adding an object class implicitly adds all its superclasses. All structural classes ultimately derive from `top` (OID `2.5.6.0`).

## Attribute Descriptions

An attribute description combines an attribute type OID (or short name) with zero or more options. Options include **tagging options** (e.g., `lang-en`, `lang-de`), which form hierarchies: `cn;lang-de;lang-en` is a sub-description of `cn;lang-de` and `cn;lang-en`. Attribute description hierarchies allow retrieval at varying levels of granularity.

Attribute types declare:
- The value syntax (via OID)
- Equality, ordering, and substring matching rules
- Whether multi-valued or `SINGLE-VALUE`
- Whether `COLLECTIVE`, `NO-USER-MODIFICATION`
- Usage: `userApplications` (user attributes) or operational (`directoryOperation`, `distributedOperation`, `dSAOperation`)

## Directory Schema

The RFC defines the full schema definition framework:

| Schema Element | Purpose |
|---------------|---------|
| Object Class definitions | Mandatory/optional attribute sets per class |
| Attribute Type definitions | Syntax and matching rule bindings per attribute |
| Matching Rule definitions | Comparison semantics |
| Matching Rule Use | Lists attribute types applicable to each matching rule |
| LDAP Syntax definitions | Wire encoding types |
| DIT Content Rules | Which auxiliary classes and extra attributes entries of a structural class may carry |
| DIT Structure Rules | Where entries may appear in the DIT by structural object class |
| Name Forms | Which attribute types are required/allowed in an entry's RDN |

Schema definitions are stored in **subschema subentries** — special entries that hold all schema definitions for a portion of the DIT. A client discovers the subschema entry governing any given entry by reading that entry's `subschemaSubentry` operational attribute, then querying the subschema entry with `filter=(objectClass=subschema)`.

The `extensibleObject` auxiliary class (`1.3.6.1.4.1.1466.101.120.111`) allows an entry to carry any user attribute type regardless of its structural class, while still requiring all mandatory attributes of its declared object classes.

## Operational Attributes

The RFC specifies core operational attributes maintained by servers:

| Attribute | OID | Description |
|-----------|-----|-------------|
| `objectClass` | 2.5.4.0 | All object classes the entry belongs to |
| `creatorsName` | 2.5.18.3 | DN of the user who created the entry |
| `createTimestamp` | 2.5.18.1 | Timestamp of entry creation |
| `modifiersName` | 2.5.18.4 | DN of the last modifier |
| `modifyTimestamp` | 2.5.18.2 | Timestamp of last modification |
| `structuralObjectClass` | 2.5.21.9 | The entry's structural object class |
| `governingStructureRule` | 2.5.21.10 | The DIT structure rule governing the entry |
| `subschemaSubentry` | 2.5.18.10 | DN of the subschema subentry controlling this entry |

Operational attributes are not returned in search results unless explicitly requested by name.

## DSA (Server) Informational Model

Servers expose metadata about themselves in the **root DSE** — a special entry at the empty-string DN (`""`) that is not part of any naming context. The root DSE carries DSA-specific operational attributes:

| Attribute | Purpose |
|-----------|---------|
| `namingContexts` | Context prefixes of naming contexts the server masters or shadows |
| `altServer` | URIs of alternative servers |
| `supportedControl` | OIDs of supported request controls |
| `supportedExtension` | OIDs of supported extended operations |
| `supportedFeatures` | OIDs of supported elective features |
| `supportedLDAPVersion` | Supported LDAP version numbers |
| `supportedSASLMechanisms` | Supported SASL mechanisms |

The root DSE is only accessible via a baseObject search at the empty-string base with `filter=(objectClass=*)`.

## Relationship to the RFC 4510 Series

RFC 4512 is tightly coupled with [[reflect/sources/2026-05-02-rfc4517-txt|RFC 4517]] (syntaxes and matching rules): attribute type definitions in RFC 4512 reference syntax OIDs and matching rule names from RFC 4517. Together they define the full schema layer; the [[reflect/synthesis/ldap-schema-architecture|four-layer schema architecture]] (syntaxes → matching rules → attribute types → object classes) spans both documents. [[reflect/concepts/distinguished-name|Distinguished names]] are the key naming mechanism defined here and used throughout the protocol ([[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]]).
