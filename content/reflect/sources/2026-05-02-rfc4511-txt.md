RFC 4511 (Standards Track, June 2006) defines the protocol elements, semantics, and encodings of the Lightweight Directory Access Protocol ([[reflect/concepts/ldap|LDAP]]). Edited by Jim Sermersheim of [[reflect/entities/novell|Novell, Inc.]], it is the authoritative wire protocol specification for LDAPv3, and part of the RFC 4510 LDAP Technical Specification series. It obsoletes RFC 2251 (the original LDAPv3 protocol spec), RFC 2830 §§2 and 4 (StartTLS protocol aspects), and RFC 3771 (IntermediateResponse) entirely.

Significant technical contributors credited in the RFC: [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]], [[reflect/entities/steven-legg|Steven Legg]], and Hallvard Furuseth.

## Protocol Model

LDAP follows a client-server model: clients send requests describing operations; servers perform directory operations and return responses. Operations are generally independent and asynchronous — multiple requests and responses may be in flight simultaneously. Protocol operations map to a subset of the X.500 (1993) Directory Abstract Service (X.511), though there is no one-to-one correspondence with X.500 DAP operations.

All protocol exchanges are wrapped in the `LDAPMessage` envelope:

```
LDAPMessage ::= SEQUENCE {
    messageID   MessageID,
    protocolOp  CHOICE { bindRequest, bindResponse, searchRequest, ... },
    controls    [0] Controls OPTIONAL
}
MessageID ::= INTEGER (0 .. maxInt)   -- maxInt = 2147483647
```

The `messageID` must be non-zero and unique among outstanding requests in the session. Zero is reserved for unsolicited server-to-client notifications. The `controls` field carries extension data — see [[reflect/concepts/ldap-controls|LDAP Controls]].

## Core Protocol Operations

RFC 4511 defines the following operations:

| Operation | Direction | Purpose |
|---|---|---|
| **Bind** | client → server | Authenticate; exchange credentials (simple or [[reflect/concepts/sasl|SASL]]) |
| **Unbind** | client → server | Terminate the LDAP session gracefully ("quit", not the antithesis of Bind) |
| **Search** | client → server | Query entries matching a filter within a scope |
| **Modify** | client → server | Add/delete/replace attribute values atomically |
| **Add** | client → server | Create a new directory entry |
| **Delete** | client → server | Remove a leaf entry |
| **ModifyDN** | client → server | Rename or move an entry (change RDN and/or parent) |
| **Compare** | client → server | Test whether an entry has a specific attribute value |
| **Abandon** | client → server | Request the server stop processing a prior operation |
| **Extended** | bidirectional | New operations identified by OID (e.g., StartTLS) |
| **IntermediateResponse** | server → client | Streaming responses for Extended operations or controls |
| **Unsolicited Notification** | server → client | Server-initiated advisory messages (e.g., Notice of Disconnection) |

### Bind Operation

The Bind operation authenticates a client to the server using one of two `AuthenticationChoice` variants:

- **Simple**: Three sub-cases — anonymous (empty name + empty password), unauthenticated (non-empty name + empty password, trace only, NOT authorized), or name/password (cleartext credentials, requires TLS protection)
- **SASL**: Pluggable framework; carries mechanism name and optional credentials; multi-stage exchanges use multiple Bind request/response pairs

Text passwords SHOULD be prepared with SASLprep (RFC 4013) before transmission. While a Bind is in progress, clients MUST NOT send other requests. See [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]] for authentication semantics, the authorization state model, and StartTLS procedures.

### Search Operation

Search is the most complex operation. Key parameters:

- `baseObject` — the DN at which to root the search
- `scope` — `baseObject` (entry only), `singleLevel` (immediate children), or `wholeSubtree`
- `derefAliases` — whether to dereference alias entries (`neverDerefAliases`, `derefInSearching`, `derefFindingBaseObj`, `derefAlways`)
- `sizeLimit` / `timeLimit` — client-imposed resource limits (servers may further constrain)
- `typesOnly` — if TRUE, return attribute descriptions only, not values
- `filter` — a [[reflect/concepts/ldap-search-filters|boolean filter expression]] (equality, substring, presence, approximate, greater/less-than, extensible match)
- `attributes` — list of attributes to return; special values: `*` (all user attributes), `1.1` (no attributes)

Filters evaluate to TRUE/FALSE/Undefined per X.511 three-valued logic. Servers MUST NOT return errors for unrecognized attribute descriptions in filters — such items evaluate to Undefined. Results are returned as zero or more `SearchResultEntry` and `SearchResultReference` messages followed by `SearchResultDone`.

### Modify Operation

Changes are applied atomically: all changes succeed or none are applied. Each change specifies an `operation` (add / delete / replace) and a `PartialAttribute`. The entry's RDN attribute values cannot be removed via Modify — ModifyDN handles that. Intermediate states during a multi-change list may temporarily violate schema; only the final state must conform.

### Extended Operation and IntermediateResponse

Extended operations add new operations to the protocol identified by OID in the `requestName` field. The **StartTLS** extended operation (OID `1.3.6.1.4.1.1466.20037`) is defined in RFC 4511 §4.14; it installs a [[reflect/concepts/ldap-tls|TLS layer]] over the existing session. Servers advertise supported operations in the `supportedExtension` root DSE attribute.

`IntermediateResponse` messages (APPLICATION 25) enable single-request/multiple-response patterns for Extended operations and request controls. They are only sent when explicitly solicited; those used with request controls MUST include a `responseName` to identify their source.

### Unsolicited Notifications

The server may send unsolicited messages (messageID = 0, using ExtendedResponse format) to notify clients of extraordinary conditions. The most important is the **Notice of Disconnection** (OID `1.3.6.1.4.1.1466.20036`), sent before the server terminates the session. When accompanied by `strongerAuthRequired`, it signals an unexpected failure or compromise of the security association.

## Result Codes

Every operation response carries an `LDAPResult` with:
- `resultCode` — enumerated status (see below)
- `matchedDN` — last matched DN (for `noSuchObject`, `invalidDNSyntax`, alias errors)
- `diagnosticMessage` — human-readable detail; MUST NOT be relied on programmatically
- `referral` — URIs for other servers; present only when `resultCode = referral`

**Non-error result codes**: `success` (0), `compareFalse` (5), `compareTrue` (6), `referral` (10), `saslBindInProgress` (14).

**Common error codes** (selected):

| Code | Value | Meaning |
|---|---|---|
| `operationsError` | 1 | Operation sequencing problem (e.g., StartTLS with outstanding ops) |
| `protocolError` | 2 | Malformed PDU, unsupported version, unrecognized Extended operation |
| `timeLimitExceeded` | 3 | Client-specified time limit exceeded |
| `sizeLimitExceeded` | 4 | Client-specified size limit exceeded |
| `authMethodNotSupported` | 7 | Authentication method or SASL mechanism not supported |
| `strongerAuthRequired` | 8 | Server requires stronger authentication |
| `unavailableCriticalExtension` | 12 | Unrecognized critical control |
| `noSuchAttribute` | 16 | Named attribute not present on entry |
| `constraintViolation` | 19 | Attribute value violates schema constraints |
| `noSuchObject` | 32 | Target entry not found in the DIT |
| `invalidCredentials` | 49 | Wrong name or password |
| `insufficientAccessRights` | 50 | Client lacks access to perform the operation |
| `unwillingToPerform` | 53 | Server decline without more specific reason |
| `namingViolation` | 64 | Entry name violates naming restrictions |
| `objectClassViolation` | 65 | Entry violates object class constraints |
| `entryAlreadyExists` | 68 | Target entry for add/move already exists |
| `other` | 80 | Internal server error |

Servers may substitute result codes to prevent unauthorized disclosure (e.g., returning `noSuchObject` instead of `insufficientAccessRights`).

## Referrals and Continuation References

When a server cannot service a request but knows of other servers that may, it returns a `referral` result code with one or more URIs (typically LDAP URLs per RFC 4516). For Search operations, mid-stream `SearchResultReference` messages indicate subtrees held by other servers.

Clients following referrals MUST detect and prevent loops — they MUST NOT contact the same server with the same request, and implementations MUST support at least ten nested referrals.

## Protocol Encoding

LDAP protocol elements are encoded using a restricted subset of ASN.1 Basic Encoding Rules (BER):
- Only definite-length encoding
- OCTET STRING in primitive form only
- Boolean TRUE is always `0xFF`
- Default values are absent from the encoding

Encoded `LDAPMessage` PDUs are mapped directly onto TCP (default port 389). Port 636 (LDAPS — direct TLS wrapping) is not defined in this RFC; StartTLS is the RFC-defined mechanism for establishing TLS over an existing session.

## Session Layer Model

An LDAP session stacks four layers:

```
LDAP message layer    ← LDAP PDUs
SASL layer            ← optional; authentication + optional integrity/confidentiality
TLS layer             ← optional; transport confidentiality + integrity
transport connection  ← TCP
```

See [[reflect/concepts/ldap-tls|TLS and StartTLS in LDAP]] and [[reflect/concepts/sasl|SASL]] for details on the security layers. This layering model is the basis for the security architecture analyzed in [[reflect/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture: Authentication Gap to RFC 4513]].

## Controls

[[reflect/concepts/ldap-controls|LDAP Controls]] (RFC 4511 §4.1.11) allow additional semantics to be attached to any operation via the `controls` field of `LDAPMessage`. Each control carries a `controlType` OID, a `criticality` flag, and an optional BER-encoded `controlValue`. Critical controls force the server to reject the operation if it cannot process them; non-critical controls are silently ignored if unrecognized.

## Security Considerations

Cleartext passwords (simple name/password Bind) are strongly discouraged without transport confidentiality. SASL authentication does not protect the Bind request's `version` and `name` fields, nor the response's `resultCode`, `diagnosticMessage`, or `referral` fields — these SHOULD NOT be relied on unless TLS or other transport-layer protection is in place.

Security factors may change during a session (credential expiration, access control changes, TLS layer replacement). Implementations should be robust to such changes and should fail operations that would return sensitive data after security was downgraded.

Implementations MUST handle invalid and arbitrary-length protocol encodings without crashing — the LDAPv3 PROTOS test suite provides a comprehensive set of encoding edge cases.

## Key Changes from RFC 2251

RFC 4511 introduced numerous clarifications and tightening:
- Renamed `errorMessage` to `diagnosticMessage` (allowed on non-error results)
- Non-zero requirement for request `messageIDs`; zero reserved for unsolicited notifications
- `present` filter now evaluates to Undefined for unknown attributes (was FALSE in RFC 2251, creating counter-intuitive NOT-filter behavior)
- Added explicit ASN.1 extensibility markers (`...`) throughout
- Removed the Binary Option (interoperability problems)
- Strengthened Notice of Disconnection handling (SHOULD return notice; MUST terminate session)
- Specified how Extended operation values defined in ASN.1 are to be encoded
- Clarified control combinations and ordering
