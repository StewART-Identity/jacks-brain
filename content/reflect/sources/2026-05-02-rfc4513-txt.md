RFC 4513 (Standards Track, June 2006) specifies the authentication methods and security mechanisms for [[reflect/concepts/ldap|LDAP]]. Edited by [[reflect/entities/roger-harrison|Roger Harrison]] of [[reflect/entities/novell|Novell, Inc.]], it is part of the RFC 4510 LDAP Technical Specification series that replaced the original December 1997 RFCs (2251–2256). It obsoletes RFC 2251 (Sections 4.2.1 and 4.2.2), RFC 2829 (entirely), and RFC 2830 (in part).

This document directly addresses the authentication gap flagged in the IESG note embedded in [[reflect/sources/2026-05-02-rfc2254-txt|RFC 2254]] (1997), which had discouraged update-capable LDAPv3 deployments until authentication was standardized. RFC 4513 closes that loop — [[reflect/sources/2026-05-02-rfc4515-txt|RFC 4515]] (2006) removed the IESG note from the search filter spec, citing RFC 4513's resolution.

## Security Threat Model

RFC 4513 catalogs eight threat categories for LDAP directory services:

1. Unauthorized access via data retrieval
2. Unauthorized access via monitoring others' access (passive)
3. Unauthorized access to reusable authentication credentials (passive)
4. Unauthorized directory modification
5. Unauthorized configuration modification
6. Denial of service
7. Spoofing — false data in transit, false server identity, or false client identity
8. Hijacking — seizing control of an established session

Threats 1, 4, 5, 6, 7, 8 are active attacks; 2 and 3 are passive. LDAP's security mechanisms address all eight categories through a combination of TLS, SASL, and access control.

## Implementation Requirements

The RFC establishes a mandatory baseline:

- All LDAP servers MUST support anonymous authentication
- Any server supporting non-anonymous auth MUST also support name/password Bind AND MUST support TLS via StartTLS to protect it
- Servers SHOULD default to disallowing name/password auth when data security is not in place
- Servers SHOULD support [[reflect/concepts/sasl|SASL]] EXTERNAL
- Implementations supporting TLS MUST support `TLS_RSA_WITH_3DES_EDE_CBC_SHA`

## StartTLS Operation

StartTLS (OID `1.3.6.1.4.1.1466.20037`) is an LDAP extended operation defined in RFC 4511 §4.14 that installs a [[reflect/concepts/ldap-tls|TLS layer]] over an existing LDAP session. It provides confidentiality and integrity; TLS authentication services are available to LDAP only in combination with [[reflect/concepts/sasl|SASL EXTERNAL]].

**Request sequencing**: A client MUST NOT send StartTLS when TLS is already established, a multi-stage SASL negotiation is in progress, or there are outstanding unresponded operation requests. Violations return `operationsError`.

**Recommended order**: StartTLS SHOULD be performed before any Bind operation so that credentials are protected by TLS in transit.

**Server identity check**: To prevent man-in-the-middle attacks, clients MUST verify the server's identity by comparing the reference identity (typically the hostname used to connect) against `subjectAltName` values in the server's certificate:

| Reference identity type | Certificate field | Matching rule |
|---|---|---|
| DNS name | `dNSName` subjectAltName | Case-insensitive; `*.example.com` wildcards match only leftmost label; IDNs converted to ACE before comparison |
| IP address | `iPAddress` subjectAltName | Network byte order, octet-for-octet |
| Common Name (deprecated) | CN in subject DN | DNS name rules apply, but NO wildcard matching |

If identity verification fails, user-facing clients SHOULD warn the user; automated clients SHOULD close the connection and log the error.

**Capability refresh**: After TLS is installed, clients SHOULD discard server capabilities obtained before TLS (a MITM could have altered them). `supportedSASLMechanisms` in particular may change — EXTERNAL and PLAIN are typically advertised only after TLS is established.

**Effect on authorization state**: Establishing, changing, or closing a TLS layer may move the session's authorization state. Unlike earlier RFC 2830, RFC 4513 does not require forcing the state to anonymous on TLS closure — local policy governs the transition.

## Authorization State

Every LDAP session has an associated *authorization state* comprised of authentication state, how it was established, and what security services are in place. Sessions begin anonymous. The Bind operation is the primary mechanism for moving to an authenticated state.

Upon receipt of any Bind request, the server immediately moves the session to anonymous. If the Bind succeeds, the session moves to the requested authentication state. If the Bind fails, the session remains anonymous.

The authorization identity may differ from the authentication identity — [[reflect/concepts/sasl|SASL]] allows a proxy to authenticate with its own credentials while operating under a different authorization identity. If the asserted authorization identity is not permitted, the server MUST reject the Bind with `invalidCredentials`.

## Simple Authentication Method

The simple Bind method provides three mechanisms:

### Anonymous Authentication

A Bind request with zero-length name AND zero-length password. Explicitly establishes an anonymous authorization state.

### Unauthenticated Authentication

A Bind request with non-zero-length name but zero-length password. The DN is used for trace/logging only — it is NOT authenticated and MUST NOT be used for authorization. This mechanism has significant security risks:

- Clients making authorization decisions based on Bind success may be misled — the server may return `success` but has established only an anonymous state
- Users intending name/password auth may accidentally submit an empty password, silently triggering this mechanism

RFC 4513 recommends: clients SHOULD disallow empty password input to the name/password interface; servers SHOULD by default reject unauthenticated Binds with `unwillingToPerform`.

### Name/Password Authentication

A Bind request with non-zero-length DN and non-zero-length password. Result codes: `invalidDNSyntax` (malformed DN), `invalidCredentials` (invalid credentials), `success`. The password is transmitted in cleartext and is **not suitable without confidentiality protection** (TLS or equivalent).

## SASL Authentication Method

[[reflect/concepts/sasl|SASL]] (Simple Authentication and Security Layer, RFC 4422) authentication uses the Bind operation's `sasl` authentication choice. The server responds with `saslBindInProgress` for each challenge round; the final response carries a non-`saslBindInProgress` result code.

SASL layers take effect after the final successful Bind response's first octet and stack on top of any TLS layer, regardless of negotiation order. An installed SASL layer survives a subsequent failed or non-SASL Bind.

**Downgrade attack detection**: Because a MITM can modify `supportedSASLMechanisms` before integrity protection is installed, clients should retrieve this attribute both before and after installing data integrity service. If the integrity-protected list contains stronger mechanisms than the unprotected list, assume tampering — close and reconnect.

**SASL authorization identities** take one of two forms:
- `dn:<DN>` — matched via `distinguishedNameMatch`
- `u:<userid>` — normalized via SASLprep (RFC 4013) then compared octet-wise

## SASL EXTERNAL Mechanism

EXTERNAL allows a client to request that the server authenticate and authorize it using credentials already established at a lower layer (typically a TLS client certificate):

- **Implicit assertion**: Client sends EXTERNAL Bind without credentials field — server derives authorization identity from TLS credentials per local policy
- **Explicit assertion**: Client sends EXTERNAL Bind with credentials field containing the asserted `authzId` — server verifies whether the TLS credentials permit the asserted identity

If no lower-layer credentials exist, the EXTERNAL Bind MUST fail with `inappropriateAuthentication`.

## Security Considerations

- **Cleartext passwords**: Strongly discouraged over open networks. Servers MUST support policy requiring TLS or another confidentiality mechanism before accepting cleartext password operations — or returning `confidentialityRequired`
- **Hashed passwords**: Mechanisms such as DIGEST-MD5 send a hash rather than the cleartext password but remain vulnerable to offline dictionary attacks; protect hash values with TLS during transmission
- **StartTLS quality**: A MITM can downgrade TLS negotiation to weak or no confidentiality; both parties must independently verify the negotiated security level before proceeding
- **No standard access control**: LDAP defines no standard access control facility; access control is implementation-specific and vendor-dependent
