The **OSI Implementers' Guide Version 1.1** (December 2006) is the companion to the [[reflect/sources/2026-05-04-t-rec-x-imp500-200109-i-msw-e|Directory Implementors' Guide]] for the OSI application and security protocol layer. Where the Directory Guide covers the X.500-series (directory data models, X.509 PKI), this guide covers the **X.200-series** (OSI session and presentation), **X.600-series** (excluding ASN.1 X.680/X.690), and **X.800-series** (OSI security architecture) recommendations — and their ISO/IEC equivalent standards.

The Guide is approved by [[reflect/entities/itu-t|ITU-T]] Study Group 17 and distributed free of charge. Like the Directory Guide, it is neither a Recommendation nor an International Standard, but records agreed corrections to formally approved defects. It is intended as an additional authoritative reference for implementers to read alongside the Recommendations themselves.

## Scope

The Guide records defect resolutions in three categories:
- Editorial errors
- Technical errors (omissions, inconsistencies)
- Ambiguities impeding implementation

New features or proposed changes to the specifications are explicitly out of scope; those follow the normal ITU-T contribution process through Study Group 17 and ISO/IEC JTC 1.

## Key Defects and Resolutions

### X.225 — OSI Session Protocol

**Defect 1** (corrected by TC1, 03/2000): Two fixes:
1. Cross-reference correction in §7.4.1 b) 5) — CONNECT SPDU reference corrected from `[see 7.1.1, b), 4)]` to `[see 7.1.1, b), 5)]`
2. Numeric typo in §8.3.4.13 — value `654539` corrected to `65539`

### X.227 — ACSE Connection-oriented Protocol

Three defects documented against [[reflect/concepts/acse|ACSE]]:

**Defect 1**: References to withdrawn Recommendation X.208 — replaced throughout with X.680 (the current [[reflect/concepts/asn1|ASN.1]] recommendations). X.208 was the original ASN.1 standard; X.680 is its successor.

**Defect 2**: OID conflict — the authentication mechanism OID in Annex B (`authentication-mechanism(3)`) collides with the ASE OID defined in the body (`as-id(3)`). The resolution notes that since no implementation has raised an interoperability problem, the OID is deliberately left unchanged. It now ambiguously references both the authentication mechanism using password and the Association Control ASE.

**Defect 3**: ASN.1 module `ACSE-1` imports from `InformationFramework` with an obsolete joint-iso-ccitt OID — corrected to `joint-iso-itu-t ds(5) module(1) informationFramework(1) 3`.

### X.227bis — ACSE Connection-mode (ASO) Protocol

**Defect 1**: Same OID conflict as X.227 Defect 2 — same resolution (left unchanged).

**Defect 2**: `AP-Title` and `ASO-qualifier` types lacked a fourth form using the `RELATIVE-OID` ASN.1 type. The guide adds extensibility:
```asn1
AP-title-form4 ::= [0] IMPLICIT RELATIVE-OID
ASO-qualifier-form4 ::= [0] IMPLICIT RELATIVE-OID
    -- This is a RELATIVE-OID with exactly one arc.
```
This is an extension addition — implementations unaware of it behave appropriately by ignoring the new alternative.

### X.228 — Reliable Transfer Protocol

**Defect 1** (corrected by TC1, 03/2000): Clarification of encoding comparison behavior — components compared using BER with identifier and length octets stripped; local matter how responding RTPM performs the comparison.

**Defects 2–5**: ASN.1 module `Reliable-Transfer-APDU` corrections:
- Replace deprecated `APPLICATION-SERVICE-ELEMENT` import with `CONTRACT` from Remote-Operations-Information-Objects
- Redefine `rTSE` from `APPLICATION-SERVICE-ELEMENT` to `CONTRACT`
- Add definition of the `OPEN` class: `OPEN ::= CLASS{&Type} WITH SYNTAX { TYPE &.Type }`
- Replace all occurrences of `ANY` with `OPEN.&Type`

### X.237 — ACSE Connectionless Protocol

**Defect 1** (corrected by TC1, 10/1996): Two syntactic fixes in §9.1 ASN.1 module — missing semicolon at end of IMPORTS statement, and missing tag `[12]` between `calling-authentication-value` and `EXPLICIT`.

### X.274 — Transport Layer Security Protocol

**Defect 1**: Object identifier for `ASSR-ID` used arc 3 below `joint-iso-ccitt`, which is reserved. Corrected to use the ISO arc:
```asn1
-- Before (wrong):
ASSR-ID {joint-iso-ccitt(2) identifier organization(3) oiw(14) ...}
-- After (correct):
ASSR-ID {iso(1) identifier-organization(3) oiw(14) ...}
```

### X.638 — Minimal OSI Facilities

**Defect 1**: An object identifier in §B.5.1 uses `joint-iso-itu-t(2) standard(0) X.637(11188-1)` when it should use `iso(0) standard(0) 11188-1`. The same error appears in §B.5.2, Table D.2, Table E.3, and §F.1–F.3.

### X.830 — Generic Upper Layers Security (GULS)

Two ASN.1 module corrections to the [[reflect/concepts/guls|GULS]] security framework:

**Defect 1**: References to withdrawn Recommendation X.208 — replaced with X.680 (same correction as X.227 Defect 1).

**Defect 2**: Annex A module `GulsSecurityNotation` — the `PROTECTED` parameterized type had an incorrect `data-value` encoding specification in the `embedded` alternative:
```asn1
-- Before (wrong):
data-value (WITH COMPONENTS {notation (BaseType)})

-- After (correct):
data-value (CONTAINING BaseType)
```
The `WITH COMPONENTS {notation (BaseType)}` form was an outdated notational construct; `CONTAINING BaseType` is the correct current ASN.1 notation for embedding a value of `BaseType` within an `EMBEDDED PDV`.

**Defect 3**: Annex A module `GulsSecurityTransformations` — the `gulsSignedTransformation` definition had the same `data-value` error in its `intermediateValue` component:
```asn1
-- Before (wrong):
data-value WITH COMPONENTS ({notation (IntermediateType{{SupportedKIClasses}})})

-- After (correct):
data-value (CONTAINING IntermediateType{{SupportedKIClasses}})
```

### X.880 — Remote Operations: Concepts

Four significant corrections to the [[reflect/concepts/rose-remote-operations|Remote Operations]] model, corrected by TC1 (07/1995):

**a) ReturnResult (§9.4.1 and Annex A)**: The `ReturnResult` type was incorrectly structured. The corrected definition properly nests the `result` SEQUENCE with both `opcode` and `result` fields inside a single OPTIONAL sequence, with `CONSTRAINED BY` error handling for `returnResult-unrecognisedInvocation`, `returnResult-resultResponseUnexpected`, and `returnResult-mistypedResult`.

**b) `recode` macro (§10.11 and Annex A)**: The `recode` operation-parameterization macro was incomplete — corrected definition includes `ALWAYS RESPONDS`, `INVOKE PRIORITY`, and `RESULT-PRIORITY` fields.

**c) `switch` macro (§10.12 and Annex A)**: The `switch` operation-package macro corrected to include `OPERATIONS`, `CONSUMER INVOKES`, `SUPPLIER INVOKES`, and `ID` fields.

**d) `combine` macro (§10.13.1 and Annex A)**: The `combine` operation-package combination macro corrected to include proper set-union operations for `OPERATIONS`, `CONSUMER INVOKES`, and `SUPPLIER INVOKES`.

### X.882 — Remote Operations Service Element (ROSE)

Three defects against the ROSE protocol realization:

**Defect 1** (TC1, 07/1995): The `AllValues` type in §10.5.3 and Annex B — corrected ASN.1 properly references `combine` to aggregate all operations from the application context's consumer/supplier roles.

**Defect 2**: A further compilation fix to `AllValues` — the `combine` call required an explicit OPERATION-PACKAGE information object with an OID placeholder to compile correctly.

**Defect 3**: Object identifier for `association-by-rtse` used arc `{joint-iso-itu-t association-realizations(10) ...}` — corrected to `{joint-iso-itu-t remote-operations(4) association-realizations(10) association-by-rtse(2)}`.

## Relationship to the Directory Implementors' Guide

This Guide is the OSI protocol layer complement to the [[reflect/sources/2026-05-04-t-rec-x-imp500-200109-i-msw-e|Directory Implementors' Guide]]. The X.500 directory series relies on several of these X.200/X.800 protocols:

- **X.227 (ACSE)** is used for establishing associations in X.500 DAP/DSP — the Directory Access Protocol and Directory System Protocol
- **X.880/X.882 (ROSE)** provides the request/response operation framework for all X.500 directory operations
- **X.830 (GULS)** was the X.500 series' intended OSI security framework — deprecated across the entire X.500 series in DIG v15 Defect 228, but its ASN.1 module errors are documented here

The [[reflect/entities/itu-t|ITU-T]] Study Group 17 approves both guides, reflecting their common governance. The fixes documented here represent the normative substrate that X.500 DAP implementations must correctly implement to interoperate at the OSI protocol binding level.
