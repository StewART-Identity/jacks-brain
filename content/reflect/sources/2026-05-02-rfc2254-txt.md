RFC 2254 defines the human-readable string format for representing [[reflect/concepts/ldap-search-filters|LDAP search filters]]. Published December 1997 by [[reflect/entities/tim-howes|Tim Howes]] of [[reflect/entities/netscape-communications|Netscape Communications Corp.]], it supersedes RFC 1960 and extends filter syntax to cover LDAPv3 features — particularly extensible match filters.

## Context

[[reflect/concepts/ldap|LDAP]] (Lightweight Directory Access Protocol) defines a binary network encoding of search filters using ASN.1 Basic Encoding Rules (BER). RFC 2254 provides a parallel *string* representation — a human-readable form suitable for configuration files, APIs, and command-line tools. The two representations are semantically equivalent; the string form is never transmitted on the wire.

## Filter Grammar

The string representation uses a prefix notation defined in ABNF:

```
filter     = "(" filtercomp ")"
filtercomp = and / or / not / item
and        = "&" filterlist
or         = "|" filterlist
not        = "!" filter
filterlist = 1*filter
item       = simple / present / substring / extensible
simple     = attr filtertype value
filtertype = equal / approx / greater / less
equal      = "="
approx     = "~="
greater    = ">="
less       = "<="
extensible = attr [":" "dn"] [":" matchingrule] ":=" value
           / [":" "dn"] ":" matchingrule ":=" value
present    = attr "=*"
substring  = attr "=" [initial] any [final]
```

Five characters require backslash-hex escaping in values: `*` (0x2a), `(` (0x28), `)` (0x29), `\` (0x5c), and NUL (0x00).

## Key Filter Types

| Filter type | String syntax | Description |
|---|---|---|
| Equality | `(cn=Babs Jensen)` | Exact attribute match |
| Presence | `(cn=*)` | Attribute exists |
| Substring | `(o=univ*of*mich*)` | Wildcard prefix/infix/suffix |
| Approximate | `(cn~=Howes)` | Phonetic or fuzzy match |
| Greater/Less | `(age>=18)` | Lexicographic ordering |
| Extensible | `(cn:1.2.3.4.5:=Fred)` | OID-specified matching rule |

## Extensible Matching

The LDAPv3 extensible match filter (absent from LDAPv2/RFC 1960) allows a named matching rule OID to govern comparison, and optionally signals that DN component attributes should participate in the match via the `:dn` flag. Examples from the RFC:

- `(cn:1.2.3.4.5:=Fred Flintstone)` — use matching rule 1.2.3.4.5
- `(sn:dn:2.4.6.8.10:=Barney Rubble)` — use matching rule with DN attributes included
- `(o:dn:=Ace Industry)` — equality match including DN components
- `(:dn:2.4.6.8.10:=Dino)` — apply rule to any supporting attribute, including from DN

## Escaping Mechanism

Values containing reserved characters are escaped as `\xx` (two-digit hex). This supports:
- Literal `*`, `(`, `)`, `\` in values: `(o=Parens R Us \28for all your parenthetical needs\29)`
- Arbitrary binary values: `(bin=\00\00\00\04)`
- Non-ASCII UTF-8 sequences: `(sn=Lu\c4\8di\c4\87)`

## Security Note

The IESG approved RFC 2254 as a Proposed Standard despite the absence of mandatory LDAPv3 authentication standards. The note explicitly discourages deploying update-capable LDAPv3 clients/servers until authentication is standardized, but approves read-only deployments and interoperability testing.

## Relationship to Other RFCs

- **Supersedes:** RFC 1960 (LDAPv2 string filter syntax)
- **Obsoleted by:** [[reflect/sources/2026-05-02-rfc4515-txt|RFC 4515]] (June 2006 — part of the RFC 4510 LDAP revision series)
- **Depends on:** RFC 2251 (LDAPv3 core protocol), RFC 2252 (attribute syntax definitions)
- **Encoding context:** ITU-T X.690 (ASN.1 BER) governs wire encoding; this RFC covers only the human-readable string form
