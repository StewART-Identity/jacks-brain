RFC 2256 (Standards Track, December 1997) is a normative reference authored by [[reflect/entities/mark-wahl|Mark Wahl]] (Critical Angle Inc.) that compiles the user-facing attribute types and object classes from the ISO/[[reflect/entities/itu-t|ITU-T]] X.500(96) directory standard series — specifically X.501, X.509, X.520, and X.521 — into a form suitable for use with [[reflect/concepts/ldap|LDAPv3]]. It does not invent new attributes; it is a carefully scoped summary of what LDAP servers SHOULD recognize to interoperate with the most widely deployed directory schema. The [[reflect/concepts/ldap-syntaxes|syntaxes]] and [[reflect/concepts/ldap-matching-rules|matching rules]] on which these definitions depend are specified separately in [[reflect/sources/2026-05-02-rfc2252-txt|RFC 2252]]. The 2006 successor is RFC 4519, which obsoletes RFC 2256 as part of the [[reflect/synthesis/ldap-technical-specification-architecture|RFC 4510 modular reorganization]].

## IESG Authentication Caveat

RFC 2256 carries the same IESG Note as its companion documents [[reflect/sources/2026-05-02-rfc2251-txt|RFC 2251]] and [[reflect/sources/2026-05-02-rfc2252-txt|RFC 2252]]: update operations are unsupported by any mandatory authentication mechanism; implementations are "UNLIKELY TO INTEROPERATE, or MAY INTEROPERATE ONLY IF AUTHENTICATION IS REDUCED TO AN UNACCEPTABLY WEAK LEVEL." This gap was closed in 2006 by [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]]. The full arc of this problem is documented in [[reflect/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture: Authentication Gap to RFC 4513]].

## Attribute Types

RFC 2256 specifies 55 attribute types that LDAP server implementations SHOULD recognize. These are drawn from multiple X.500 source documents:

| RFC 2256 sections | X.500 source |
|-------------------|-------------|
| 5.1–5.2 | X.501(93) |
| 5.3–5.36 | X.520(88) |
| 5.37–5.41 | X.509(93) |
| 5.42–5.52 | X.520(93) |
| 5.53–5.54 | X.509(96) |
| 5.55 | X.520(96) |

### PKI Certificate Attributes

Several attributes support X.509 PKI integration. All must be stored and requested in binary form using the `;binary` transfer option:

| Attribute | OID | Purpose |
|-----------|-----|---------|
| `userCertificate` | 2.5.4.36 | User's X.509 certificate |
| `cACertificate` | 2.5.4.37 | CA's certificate |
| `authorityRevocationList` | 2.5.4.38 | Authority Revocation List for cross-certification |
| `certificateRevocationList` | 2.5.4.39 | CRL published by a CA |
| `crossCertificatePair` | 2.5.4.40 | Cross-certificates between CAs |
| `supportedAlgorithms` | 2.5.4.52 | Algorithms supported — new in X.500(96) |
| `deltaRevocationList` | 2.5.4.53 | Delta CRL — new in X.500(96) |

Three of these — `supportedAlgorithms`, `deltaRevocationList`, and `dmdName` (2.5.4.54) — are new additions in the X.500(96) revision, appearing for the first time in this RFC.

### Security Note: userPassword

The `userPassword` attribute stores passwords as unencrypted Octet String values. RFC 2256 explicitly warns that cleartext password transfer is strongly discouraged where the transport cannot guarantee confidentiality — a concern addressed by [[reflect/concepts/ldap-tls|LDAP TLS]] and [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]] authentication practices.

## Syntaxes

RFC 2256 defines seven syntaxes for attribute types introduced in this document:

| Syntax | OID |
|--------|-----|
| Delivery Method | 1.3.6.1.4.1.1466.115.121.1.14 |
| Enhanced Guide | 1.3.6.1.4.1.1466.115.121.1.21 |
| Guide | 1.3.6.1.4.1.1466.115.121.1.25 |
| Octet String | 1.3.6.1.4.1.1466.115.121.1.40 |
| Teletex Terminal Identifier | 1.3.6.1.4.1.1466.115.121.1.51 |
| Telex Number | 1.3.6.1.4.1.1466.115.121.1.52 |
| Supported Algorithm | 1.3.6.1.4.1.1466.115.121.1.49 |

The Supported Algorithm syntax has no printable representation; clients MUST use `supportedAlgorithms;binary` for transfer. The Guide syntax is deprecated for new attributes — superseded by Enhanced Guide.

The syntax definitions in RFC 2256 are credited as based on the ISODE "QUIPU" implementation of X.500, the same codebase behind [[reflect/entities/isode-ltd|Isode Ltd.'s]] directory software heritage.

## Object Classes

RFC 2256 defines 22 [[reflect/concepts/ldap-object-classes|object classes]]. Servers MUST recognize `top` and `subschema`; they SHOULD recognize the rest. The core structural hierarchy:

```
top (ABSTRACT)
├── alias (STRUCTURAL) — entries that are aliases
├── country (STRUCTURAL) — MUST c
├── locality (STRUCTURAL) — geographic region
├── organization (STRUCTURAL) — MUST o
├── organizationalUnit (STRUCTURAL) — MUST ou
├── organizationalRole (STRUCTURAL) — MUST cn
├── groupOfNames (STRUCTURAL) — MUST member, cn
├── groupOfUniqueNames (STRUCTURAL) — MUST uniqueMember, cn
├── applicationProcess (STRUCTURAL) — MUST cn
├── applicationEntity (STRUCTURAL) — MUST presentationAddress, cn
│   └── dSA (STRUCTURAL) — directory system agents
├── device (STRUCTURAL) — MUST cn
├── cRLDistributionPoint (STRUCTURAL) — MUST cn
├── dmd (STRUCTURAL) — MUST dmdName [new in X.500(96)]
└── person (STRUCTURAL) — MUST sn, cn
    ├── organizationalPerson (STRUCTURAL) — adds address/contact attrs
    └── residentialPerson (STRUCTURAL) — MUST l
```

Auxiliary classes:
- **`strongAuthenticationUser`** — MUST `userCertificate`
- **`certificationAuthority`** — MUST `authorityRevocationList`, `certificateRevocationList`, `cACertificate`; MAY `crossCertificatePair`
- **`certificationAuthority-V2`** (SUP certificationAuthority) — MAY `deltaRevocationList`
- **`userSecurityInformation`** — MAY `supportedAlgorithms`

## Matching Rule

RFC 2256 adds one matching rule beyond those in [[reflect/sources/2026-05-02-rfc2252-txt|RFC 2252]]:

**`octetStringMatch`** (2.5.13.17) — equality comparison for Octet String syntax values. Servers implementing extensibleMatch filters SHOULD allow this rule in extensible match operations.
