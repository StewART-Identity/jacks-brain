RFC 2891 (August 2000) is a Standards Track specification defining two [[reflect/concepts/ldap-controls|LDAPv3 control]] extensions that together allow a client to request server-sorted search results. Authored by [[reflect/entities/tim-howes|Tim Howes]] (Loudcloud, Inc.), [[reflect/entities/mark-wahl|Mark Wahl]] (Sun Microsystems, Inc.), and Anoop Anantha ([[reflect/entities/microsoft|Microsoft Corp.]]), the RFC addresses the common case where an LDAP client has limited processing capability or cannot sort results locally.

## The Two Controls

RFC 2891 defines a paired request/response control:

### Request Control: SortKeyList (OID `1.2.840.113556.1.4.473`)

Attached to a `SearchRequest` in the `LDAPMessage` controls field (defined in §4.1.12 of [[reflect/sources/2026-05-02-rfc2251-txt|RFC 2251]]). Its `controlValue` is BER-encoded as:

```
SortKeyList ::= SEQUENCE OF SEQUENCE {
    attributeType    AttributeDescription,
    orderingRule     [0] MatchingRuleId OPTIONAL,
    reverseOrder     [1] BOOLEAN DEFAULT FALSE
}
```

Keys are listed in precedence order (highest to lowest). Each key names an attribute type, an optional [[reflect/concepts/ldap-matching-rules|ordering matching rule]] override, and a reversal flag. If `orderingRule` is omitted, the attribute's default ordering rule applies. Each `attributeType` must appear at most once — duplicates trigger `unwillingToPerform`.

### Response Control: SortResult (OID `1.2.840.113556.1.4.474`)

Returned by the server in the `searchResultDone` message. Its `controlValue` encodes:

```
SortResult ::= SEQUENCE {
    sortResult    ENUMERATED {
        success (0),
        operationsError (1),
        timeLimitExceeded (3),
        strongAuthRequired (8),
        adminLimitExceeded (11),
        noSuchAttribute (16),
        inappropriateMatching (18),
        insufficientAccessRights (50),
        busy (51),
        unwillingToPerform (53),
        other (80)
    },
    attributeType [0] AttributeDescription OPTIONAL
}
```

The client can confirm sorted results only when `sortResult = success`. If the server omits the response control entirely, the client SHOULD assume sorting was ignored.

## Client-Server Interaction Scenarios

RFC 2891 enumerates six interaction scenarios based on whether the server supports the control and the `criticality` flag value:

1. **Unsupported + criticality TRUE**: Server returns `unavailableCriticalExtension`; no results sent.
2. **Unsupported + criticality FALSE**: Server ignores the control; results returned unsorted.
3. **Supported but cannot sort + criticality TRUE**: Server returns `unavailableCriticalExtension` + SortResult; no results sent.
4. **Supported but cannot sort + criticality FALSE**: Server returns results unsorted + SortResult with error code.
5. **Supported and can sort**: Server returns sorted results + SortResult with `success`.
6. **Search failed / no results**: Server SHOULD omit the SortResult control.

## NULL Handling and Multi-Valued Attributes

Entries missing a sort key attribute are treated as having value NULL, which sorts **after** all present values (or **before** all present values when `reverseOrder = TRUE`). For multi-valued attributes with no other controls affecting order, the server SHOULD use the **least value** per the attribute's ORDERING rule as the entry's representative sort value.

## Interaction with Paged Results

When used together with [[reflect/concepts/ldap-paged-results|paged results]] ([[reflect/sources/2026-05-02-rfc2696-txt|RFC 2696]]), the server must sort the **entire** result set before returning any page — not sort each page independently. The `SortKeyList` must be identical across all `SearchRequest` messages in the paged sequence. The server SHOULD return a SortResult control on every `searchResultDone` in the sequence, allowing clients to confirm sorted order without waiting for all pages.

## Chained Environments

When a server chains the client's request to multiple LDAP servers and the client set `criticality = TRUE`, the receiving server MUST merge-sort the combined results before returning them or MUST return `unwillingToPerform`.

## Security Considerations

RFC 2891 notes that server-side sorting can be used to **circumvent administrative record-count limits**: a client can request sorted results and update its search filter progressively to retrieve the full directory in small chunks, bypassing `sizeLimit`. The defense is fine-grained access control on entries, not solely size limits.

The OIDs `1.2.840.113556.1.4.473` and `1.2.840.113556.1.4.474` fall within [[reflect/entities/microsoft|Microsoft's]] registered OID arc (`1.2.840.113556`), following the same pattern as the paged results OID (`1.2.840.113556.1.4.319` from [[reflect/sources/2026-05-02-rfc2696-txt|RFC 2696]]) — suggesting the sorting control may also have originated as an Active Directory proprietary extension before IETF submission.
