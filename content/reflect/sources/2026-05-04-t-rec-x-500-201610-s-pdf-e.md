**ITU-T Recommendation X.500 (October 2016)** is the eighth edition of the foundational Directory overview, co-published as ISO/IEC 9594-1:2017. It supersedes the [[reflect/sources/2026-05-04-t-rec-x-500-198811-s-pdf-e|1988 first edition]] and six subsequent editions, preserving the core directory model while extending it with full [[reflect/concepts/ldap|LDAP]] integration into the distributed agent model, three access control schemes, richer replication semantics, service administration capabilities, and Directory-IdM Support (Amendment 2).

The eighth edition specifies both version 1 (original) and version 2 (signed errors) of the Directory protocols. Most services and protocols are designed to function under version 1; signed errors and certain enhanced services require all involved directory entities to have negotiated version 2.

## Edition History

| Edition | Date | Key development |
|---------|------|-----------------|
| 1st | 1988-11-25 | Foundational specification: DIB, DIT, DSA/DUA, DAP/DSP, simple and strong authentication |
| 2nd | 1993-11-16 | Revision; concurrent with LDAPv1 (RFC 1487) — X.500 and LDAP begin diverging |
| 3rd | 1997-08-09 | Access control elaboration; X.509 PKI enhancements; same year as LDAPv3 |
| 4th | 2001-02-02 | Related entries; multiple-view DIT; multiple independent DITs |
| 5th | 2005-08-29 | Refinement; concurrent with RFC 4510 LDAP modular reorganization |
| 6th | 2008-11-13 | Additional enhancements |
| 7th | 2012-10-14 | Preceding this edition |
| **8th** | **2016-10-14** | **Full LDAP integration in distributed model; three access control schemes; compound entries; multiple master replication; Directory-IdM support** |

## Directory Information Base: New Structural Mechanisms

The [[reflect/concepts/directory-information-tree|DIT]] and DIB model is extended in the eighth edition with two mechanisms:

**Compound entries** represent a single object as an aggregate of *member entries*, each holding information about a particular aspect of that object. The directory treats compound operations against the whole compound entry coherently: a Search may return compound entry information packaged together, and a Modify Distinguished Name may move family members within the same compound entry. A Modify Entry, however, operates on only a single family member and cannot manipulate the compound entry as a whole.

**Hierarchical groups** establish an alternative hierarchical relationship among entries, independent of the DIT tree structure. A Search matching an entry that belongs to a hierarchical group can return both the matched entry and other group members. This allows hierarchical relationships to be changed without renaming entries, which would alter their [[reflect/concepts/distinguished-name|distinguished names]].

## LDAP Integration in the Distributed Model

The eighth edition formally integrates [[reflect/concepts/ldap|LDAP]] agents into the [[reflect/concepts/x500-distributed-directory|distributed directory model]], which previously described the architecture in terms of DSAs and DUAs communicating via DAP/DSP alone.

### New Agent Types

| Agent | Definition |
|-------|-----------|
| **LDAP server** | Application process that responds to LDAP requests; serves LDAP clients and/or LDAP requesters |
| **LDAP requester** | A DSA capable of issuing LDAP requests and understanding LDAP responses |
| **LDAP responder** | A DSA capable of understanding and responding to LDAP requests |

A DSA can be simultaneously an LDAP responder (accepting LDAP requests from clients) and an LDAP requester (forwarding requests via LDAP to other DSAs or LDAP servers). This supports topologies where X.500 and LDAP infrastructure interoperate without requiring full protocol translation at every boundary.

### Referrals and Chaining with LDAP

The edition specifies how referrals and chaining work when LDAP clients and servers are in the path:

- When a DSA returns a referral to an LDAP client, the referral must be in LDAP referral form; the client then binds directly to the referred DSA, which must be an LDAP responder
- A DSA acting as LDAP requester may translate a non-LDAP referral from another DSA into an LDAP request directed to a target DSA
- Uni-chaining and multi-chaining patterns are each specified in three configurations: DUA/DSA-only, LDAP client with DSA acting as LDAP responder, and LDAP client with DSA acting as both LDAP responder and LDAP requester
- An LDAP client bound to an LDAP server cannot have requests served by a different server (unlike a DUA, which may interact with multiple DSAs)

## Access Control

[[reflect/entities/itu-t|ITU-T]] X.501 (normative reference) defines three access control schemes, all overviewed in the eighth edition:

| Scheme | Basis | Notes |
|--------|-------|-------|
| **Basic access control** | User, right needed, component accessed, security policy | Original scheme; fine-grained entry/attribute-level control |
| **Simplified access control** | Subset of basic | Reduced capability for simpler deployments |
| **Rule-based access control** | Clearances and security labels | Mandatory access control model; usable alone or layered with basic/simplified |

The rule-based scheme involves a clearance associated with the requesting user, security labels on the information being accessed, and policy rules mapping clearance–label relationships to access decisions. It can be combined with either simplified or basic access control, or used independently.

See [[reflect/concepts/x500-directory-access-control]] for the full access control model.

## Service Administration

The eighth edition formalizes service administration — capabilities allowing administrative authorities to constrain and adapt the directory service:

- Restrict which attribute types may appear in search filters
- Limit what information may be returned from searches
- Prevent misuse (e.g., mass collection of personal data) by requiring minimum string length for searches, prohibiting certain attribute combinations, or returning sanitized results
- Differentiate service constraints by user group

Service administration complements access control: access control determines whether information *may* be returned; service administration shapes how searches *may be formulated* and what results are permissible.

## Replication

Section 12 provides the most systematic treatment of directory replication in any X.500 overview edition. Three forms of replicated entry information are defined:

**Cache copies** — obtained and maintained in ways not specified by the Directory Specifications; may become and remain permanently inconsistent with master information.

**Shadowed copies** — obtained via the Directory Information Shadowing Protocol (DISP, X.525) under a formal shadowing agreement between a shadow supplier and a shadow consumer; brought into consistency on a contracted schedule.

**Multiple master implementations** — more than one writeable instance of each entry; exactly one is designated the primary master. The mechanism for obtaining writeable copies and achieving post-modification consistency is outside this standard's scope.

The consistency model acknowledges no specific time constraints: shadowed information achieves eventual consistency with the shadow supplier; cached information may remain inconsistent indefinitely. A directory user is notified when replicated information satisfies a request and may request master-only access when needed.

See [[reflect/concepts/x500-directory-replication]] for the full replication model.

## Directory Protocols

Four protocols are defined, normatively specified in X.519:

| Protocol | Acronym | Parties | Purpose |
|----------|---------|---------|---------|
| Directory Access Protocol | DAP | DUA ↔ DSA | Client–server request/response |
| Directory System Protocol | DSP | DSA ↔ DSA | Inter-server cooperation for distributed requests |
| Directory Information Shadowing Protocol | DISP | DSA ↔ DSA | Replication/shadowing under shadowing agreements |
| Directory Operational Binding Management Protocol | DOP | DSA ↔ DSA | Administrative operational bindings |

## Relationship to the 1988 Edition and LDAP

The eighth edition preserves the structural core established in the [[reflect/sources/2026-05-04-t-rec-x-500-198811-s-pdf-e|1988 first edition]]: DIT, distinguished names, attributes, object classes, the DSA/DUA agent model, and the abstract service operations (Read, Compare, List, Search, Abandon, Add, Remove, Modify, Modify DN). The eighth edition extends that foundation by:

1. Formally integrating LDAP clients, servers, requesters, and responders into the distributed model
2. Introducing compound entries and hierarchical groups
3. Defining three access control schemes (vs. one in earlier editions)
4. Formalizing service administration
5. Elaborating the replication model with shadowing agreements, multiple master, and explicit consistency semantics
6. Adding Directory-IdM Support (Amendment 2)

See [[reflect/synthesis/x500-ldap-convergence]] for analysis of how X.500 and LDAP diverged after 1993 and how the 2016 eighth edition formally reintegrated LDAP into the X.500 framework.
