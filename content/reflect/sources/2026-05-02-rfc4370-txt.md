RFC 4370 (Standards Track, February 2006) defines the **Proxied Authorization Control** for [[reflect/concepts/ldap|LDAP]], authored by [[reflect/entities/rob-weltman|Rob Weltman]] of Yahoo!, Inc. The control allows a client to request that a single directory operation execute under a specified authorization identity rather than the connection's current authorization identity.

[Download original](/api/originals/2026-05-02-rfc4370.txt.docx)

## Purpose

Where [[reflect/concepts/sasl|SASL]] provides session-level authorization identity substitution (an `authzId` set during Bind applies to the whole session), the Proxy Authorization Control provides per-operation scope. A client — typically a proxy server or middleware — can perform operations on behalf of different end users without maintaining separate connections or re-authenticating for each request.

The control may be included in any Search, Compare, Modify, Add, Delete, Modify DN, or extended operation. The prohibited case: any extension that itself changes authentication, authorization, or data confidentiality — Start TLS is the cited example, per [[reflect/concepts/ldap-tls|LDAP TLS]].

## Wire Format

```
controlType   = "2.16.840.1.113730.3.4.18"
criticality   = TRUE (MUST be present, MUST be TRUE)
controlValue  = authzId | empty (for anonymous association)
```

The `authzId` format (from RFC 2829) takes two forms:
- `dn:<distinguished-name>` — a [[reflect/concepts/distinguished-name|directory entry DN]]
- `u:<userid>` — a non-DN user identifier
- Empty value — anonymous association

## Mandatory Criticality

Unlike most [[reflect/concepts/ldap-controls|LDAP controls]] where criticality defaults to FALSE, the Proxy Authorization Control MUST have criticality present and set to TRUE. Servers MUST reject any request carrying this control without a criticality flag, or with criticality FALSE, returning `protocolError`.

This strict requirement protects clients from a silent security failure: a non-critical control that is dropped by a non-supporting server would cause the operation to proceed under the wrong identity without the client being aware. With mandatory criticality, the server must explicitly surface its inability to process the control.

## Discovery

Support is indicated by the presence of OID `2.16.840.1.113730.3.4.18` in the `supportedControl` attribute of the server's root DSE, as defined in [[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]].

## Access Control and Result Code 123

Whether the client may assume the requested identity is governed by server policy — the RFC defines the wire format and failure mode but not the authorization policy itself. If the client has proxy rights and the identity is recognized, the operation proceeds as if submitted directly by the proxy identity. If not, the server returns result code `123` (`proxyAuthzFailure`), an IANA-assigned code specific to this control.

Proxy rights may be granted at varying granularity within a tree. A client with proxy rights at the root of a search tree may lack rights for specific subtrees; the server evaluates proxy rights per entry, silently omitting entries where rights are absent rather than failing the entire operation. This means a proxied search may return fewer results than the same search executed directly by the target identity.

## Security Considerations

- Authorization identity strings may carry confidential information; deployments should protect the channel with [[reflect/concepts/ldap-tls|TLS]].
- Anonymous connections SHOULD NOT be permitted to assume other identities via this control.
- The control operates within the general LDAP security model defined in [[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]] and [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]].

## OID Provenance

The OID `2.16.840.1.113730.3.4.18` sits in [[reflect/entities/netscape-communications|Netscape Communications Corp.'s]] OID arc (`2.16.840.1.113730`). Although the RFC was authored at Yahoo!, Inc., [[reflect/entities/rob-weltman|Rob Weltman]] was formerly at Netscape, and the OID registration predates the RFC's publication — suggesting the control was developed and the OID assigned during his Netscape tenure. This pattern — proprietary vendor OID carried into IETF standardization — is also visible in the Microsoft-arc OIDs used by [[reflect/sources/2026-05-02-rfc2696-txt|RFC 2696]] (paged results) and [[reflect/sources/2026-05-02-rfc2891-txt|RFC 2891]] (server-side sorting). Result code `123` was assigned by IANA; the OID is registered as an LDAP Protocol Mechanism per RFC 3383.
