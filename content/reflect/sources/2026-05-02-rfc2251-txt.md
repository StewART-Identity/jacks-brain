RFC 2251 (Standards Track, December 1997) is the original specification for Lightweight Directory Access Protocol version 3 ([[reflect/concepts/ldap|LDAPv3]]). Authored by [[reflect/entities/mark-wahl|Mark Wahl]] (Critical Angle Inc.), [[reflect/entities/tim-howes|Tim Howes]] ([[reflect/entities/netscape-communications|Netscape Communications Corp.]]), and [[reflect/entities/steve-kille|Steve Kille]] ([[reflect/entities/isode-ltd|Isode Limited]]), it defines the core wire protocol, data model, all protocol operations, the controls extensibility mechanism, result codes, and BER encoding. It was obsoleted in June 2006 when its content was reorganized: protocol elements and operations went into [[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]], and directory information model material (DIT structure, schema abstractions, subschema) went into RFC 4512. The full decomposition is analyzed in [[reflect/synthesis/ldap-technical-specification-architecture|LDAPv3 Specification Architecture: The RFC 4510 Modular Reorganization]].

## IESG Authentication Caveat

RFC 2251 carries a prominent IESG Note: at the time of publication, no mandatory authentication mechanism for update operations had been standardized. The IESG approved the protocol for read-only deployment and interoperability testing while explicitly discouraging production use of update functionality until a Proposed Standard for mandatory authentication was approved. The note warns directly: implementations "UNLIKELY TO INTEROPERATE, or MAY INTEROPERATE ONLY IF AUTHENTICATION IS REDUCED TO AN UNACCEPTABLY WEAK LEVEL."

This gap was resolved nine years later by [[reflect/sources/2026-05-02-rfc4513-txt|RFC 4513]] (2006), which established name/password Bind over TLS as the mandatory-to-implement password mechanism and named the unauthenticated authentication mechanism explicitly. The historical arc of this gap — from RFC 2251's deferral through RFC 4513's resolution — is the subject of [[reflect/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture: Authentication Gap to RFC 4513]].

## Protocol Model

LDAP follows a client-server model: clients transmit protocol requests to servers, which perform directory operations and return responses. Operations are asynchronous — multiple requests and responses may be in flight simultaneously, correlated by `messageID`. RFC 2251 emphasizes minimizing client complexity to facilitate widespread deployment of directory-capable applications.

The protocol runs directly over TCP (default port 389), bypassing the session/presentation overhead of X.500's Directory Access Protocol (DAP). LDAPv3 adds three significant capabilities over LDAPv2 (RFC 1777): SASL authentication, server-to-client referrals, and the controls mechanism for extending existing operations.

## Data Model

RFC 2251 describes the X.500 data model as used by LDAP:

- **Directory Information Tree (DIT)**: A hierarchical tree of entries. Each entry has a [[reflect/concepts/distinguished-name|Distinguished Name (DN)]] — unique across the tree — formed by concatenating Relative Distinguished Names (RDNs) from root to the entry. An example from the RFC: `CN=Steve Kille, O=Isode Limited, C=GB`.
- **Attributes**: Each entry contains a set of typed attributes with one or more values. Attribute types are identified by OIDs and optional short textual names (e.g., `mail`, `cn`). Values must conform to syntax constraints; matching rules govern comparison.
- **objectClass**: Every entry must carry an `objectClass` attribute defining its schema constraints. The `extensibleObject` class permits all user attributes.
- **Schema / Subschema**: Schema definitions are published in subschema entries accessible via the `subschemaSubentry` operational attribute. Subschema entries contain `objectClasses`, `attributeTypes`, `matchingRules`, and related attributes. Clients MUST retrieve subschema using a base-object search with filter `(objectClass=subschema)`.
- **Operational Attributes**: Server-maintained attributes — `creatorsName`, `createTimestamp`, `modifiersName`, `modifyTimestamp`, `subschemaSubentry` — are not returned in search results unless explicitly requested.
- **Root DSE**: The entry at the zero-length DN provides server-specific metadata: `namingContexts`, `supportedLDAPVersion`, `supportedSASLMechanisms`, `supportedControl`, `supportedExtension`, `altServer`.

## Protocol Operations

All protocol exchanges are wrapped in the `LDAPMessage` envelope:

```
LDAPMessage ::= SEQUENCE {
    messageID   MessageID,
    protocolOp  CHOICE { ... },
    controls    [0] Controls OPTIONAL
}
MessageID ::= INTEGER (0 .. maxInt)   -- maxInt = 2147483647
```

RFC 2251 defines ten operations:

| Operation | Purpose |
|---|---|
| **Bind** | Authenticate via simple (cleartext) or [[reflect/concepts/sasl|SASL]] mechanism |
| **Unbind** | Terminate the session (no response defined) |
| **Search** | Query entries matching a filter within base/one/sub scope |
| **Modify** | Add/delete/replace attribute values atomically |
| **Add** | Create a new entry |
| **Delete** | Remove a leaf entry |
| **ModifyDN** | Rename or move an entry (change RDN and/or parent) |
| **Compare** | Test whether an entry has a specific attribute value |
| **Abandon** | Request abandonment of an in-progress operation |
| **Extended** | New operations identified by OID (open-ended extensibility, new in LDAPv3) |

### Bind and Authentication

The Bind request carries a `version`, a `name` (LDAPDN), and an `AuthenticationChoice` — either `simple` (cleartext password, strongly discouraged without lower-layer confidentiality) or `sasl` (mechanism name and optional credentials). SASL mechanisms may require multiple Bind request/response round trips; clients MUST NOT send other requests while a multi-stage Bind is in progress. If a SASL integrity or confidentiality layer is negotiated, it takes effect after the final BindResponse with `success`. The Bind response's `serverSaslCreds` field allows the server to authenticate itself to the client.

### Search

Search parameters: `baseObject` (DN), `scope` (base/singleLevel/wholeSubtree), `derefAliases`, `sizeLimit`, `timeLimit`, `typesOnly`, a `filter`, and an `attributes` list. Filters support: and/or/not, equality, substring, presence, approximate, greaterOrEqual, lessOrEqual, and `extensibleMatch` (new in LDAPv3 — allows matching against all attributes in an entry or against DN attributes). Filters evaluate using the three-valued logic of X.511: TRUE, FALSE, or Undefined. Entries evaluating to TRUE are returned. The `attributes` list accepts `*` or an empty list for all user attributes, or `1.1` to suppress all attributes.

### Controls

The [[reflect/concepts/ldap-controls|controls]] mechanism — introduced for the first time in RFC 2251 as the primary LDAPv3 extensibility mechanism — attaches extension data to any operation:

```
Control ::= SEQUENCE {
    controlType   LDAPOID,
    criticality   BOOLEAN DEFAULT FALSE,
    controlValue  OCTET STRING OPTIONAL
}
```

A critical control (`criticality = TRUE`) forces the server to reject the operation with `unavailableCriticalExtension` if unrecognized. A non-critical control is silently ignored. RFC 2251 does not define any specific controls — it only specifies the mechanism, leaving all controls to companion documents.

### Result Codes

`LDAPResult` carries `resultCode`, `matchedDN`, `errorMessage`, and optional `referral`. RFC 2251 defines the full result code enumeration including codes new in LDAPv3: `referral` (10), `adminLimitExceeded` (11), `unavailableCriticalExtension` (12), `confidentialityRequired` (13), `saslBindInProgress` (14), and `affectsMultipleDSAs` (71). RFC 4511 later renamed `errorMessage` to `diagnosticMessage` and allowed it on non-error results.

### Referrals

When a server cannot service a request but knows of other servers that may, it returns a `referral` result code with one or more LDAP URLs. For Search operations, `SearchResultReference` messages provide mid-stream referrals to subtrees held by other servers. Clients following referrals must prevent loops and MUST support at least ten layers of referral depth.

## Protocol Encoding

LDAP protocol elements are encoded using a restricted subset of ASN.1 Basic Encoding Rules (BER):
- Only definite-length encoding
- OCTET STRING in primitive form only
- Boolean TRUE is always `0xFF`
- Default values are absent from the encoding

Encoded `LDAPMessage` PDUs are mapped directly onto TCP (default port 389). The complete ASN.1 module definition is provided in Appendix A of the RFC.

## Key Differences from RFC 4511

[[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]] (2006) introduced several clarifications over RFC 2251:
- `present` filter now evaluates to Undefined (was FALSE) for unknown attribute descriptions — fixing counter-intuitive NOT-filter behavior
- `errorMessage` renamed `diagnosticMessage`; now allowed on non-error results
- Removed the Binary transfer option (interoperability problems)
- Added `IntermediateResponse` message type for streaming Extended operation responses
- Non-zero requirement for request `messageIDs`; zero reserved for unsolicited notifications
- Explicit ASN.1 extensibility markers (`...`) throughout

A detailed mapping of the 1997-to-2006 decomposition is in [[reflect/synthesis/ldap-technical-specification-architecture|LDAPv3 Specification Architecture]].
