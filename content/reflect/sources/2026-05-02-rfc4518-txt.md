RFC 4518, "Lightweight Directory Access Protocol (LDAP): Internationalized String Preparation," was authored by [[reflect/entities/kurt-zeilenga|Kurt Zeilenga]] of the [[reflect/entities/openldap-foundation|OpenLDAP Foundation]] and published in June 2006 as a Standards Track RFC and integral part of the [[reflect/sources/2026-05-02-rfc4510-txt|RFC 4510 series]]. It defines string preparation algorithms for all character-based [[reflect/concepts/ldap-matching-rules|matching rules]] in [[reflect/concepts/ldap|LDAP]], providing the normative foundation consumed by [[reflect/sources/2026-05-02-rfc4517-txt|RFC 4517]]'s matching rule definitions.

## Problem Statement

Prior LDAP specifications defined character string matching rules loosely — for example, `caseIgnoreMatch` was "case-insensitive comparison where insignificant spaces are ignored." For `PrintableString` values this was adequate: there is one space character and case mapping is bijective. For Unicode strings (used in `universalString`, `bmpString`, and `UTF8String` types), the same imprecision led to interoperability failures:

- Implementations differed on which code points constituted "space" (U+0020 only? any Zs character? any WS character?)
- Case-folding direction (lowercase → uppercase vs. uppercase → lowercase) produced different results
- Combining marks were handled inconsistently

When directory string values are used in certificate chain validation, such divergences become security vulnerabilities — an attacker can exploit differences in string preparation between implementations. RFC 4518 addresses these problems with a single normative six-step algorithm.

## Relationship to stringprep (RFC 3454)

RFC 4518 is a refinement of the IETF's stringprep framework (RFC 3454, December 2002). stringprep itself was not published until 2002 — explaining why RFC 4518 has no direct 1997 predecessor among the original LDAPv3 RFCs. RFC 4518 extends the stringprep Unicode preparation steps with two LDAP-specific bookend steps:

- **Before** stringprep Unicode steps: transcode the string to Unicode from LDAP-specific string types
- **After** stringprep Unicode steps: apply insignificant-character handling specific to each matching rule type

## The Six-Step Algorithm

The [[reflect/concepts/ldap-string-preparation|LDAP String Preparation]] concept page documents each step in detail. In summary:

1. **Transcode** — Convert non-Unicode types (`PrintableString`, `TeletexString`) to Unicode; `UniversalString`, `UTF8String`, and `bmpString` require no transcoding. TeletexString has no standard Unicode mapping and is NOT RECOMMENDED.
2. **Map** — Remap specific code points to nothing (soft hyphens, variation selectors, control characters, zero-width space) or to SPACE (whitespace-class separators). For case-ignore and numeric rules: apply case folding per RFC 3454 §B.2.
3. **Normalize** — Apply Unicode Normalization Form KC (NFKC): compatibility decomposition followed by canonical composition.
4. **Prohibit** — Reject strings containing unassigned code points, deprecated characters, private use code points, non-character code points, surrogates, or U+FFFD.
5. **Check bidi** — Bidirectional characters are ignored (structural placeholder; no normative effect in RFC 4518).
6. **Insignificant Character Handling** — Rule-specific treatment: space normalization for case-ignore/exact rules (with special handling for substring assertion components); all-space removal for numericString; space-and-hyphen removal for telephoneNumber.

## Insignificant Character Handling Detail

Step 6 is the most rule-specific. For case-ignore and exact matching applied to non-substring values, the algorithm normalizes space sequences to a canonical form: exactly one leading space, exactly one trailing space, inner space runs → exactly two spaces. This ensures `"foo bar"` and `"foo  bar"` compare equal, but preserves the structure needed for substring matching.

Substring assertion components (initial, any, final) receive boundary-sensitive treatment that is more complex than simple strip-and-collapse. The rationale (explained in RFC 4518 Appendix B) is that simplified space handling would cause `(CN=foo\20*\20bar)` to incorrectly match `"foobar"`, and would break sub-partitioning: if a prepared substring matches a partition of an attribute value, asserting that same substring in subdivided form should also match.

## Security Note

RFC 4518 incorporates RFC 3454's security considerations by reference. The core security observation is that imprecise string preparation in security-sensitive contexts — especially certificate chain validation that queries directory values — creates exploitable divergences between implementations. The normative algorithm eliminates this class of vulnerability for LDAP character-string matching.
