RFC 1487 (Standards Track, July 1993) is the inaugural Lightweight Directory Access Protocol specification — the first RFC to define LDAP as a protocol. Authored by [[reflect/entities/wengyik-yeong|Wengyik Yeong]] (Performance Systems International), [[reflect/entities/tim-howes|Tim Howes]] (University of Michigan), and [[reflect/entities/steve-kille|Steve Kille]] (ISODE Consortium), it establishes the core [[reflect/concepts/ldap|LDAP]] model: a lightweight TCP-based alternative to X.500's Directory Access Protocol (DAP) that eliminates the OSI session/presentation overhead. RFC 1487 was superseded in March 1995 by [[reflect/sources/2026-05-02-rfc1777-txt|RFC 1777]] (LDAPv2), which refined the protocol while preserving its fundamental design.

## Historical Context

By 1993, X.500 directory technology was generating significant Internet interest, but the full Directory Access Protocol required an OSI session/presentation stack — a high "cost of entry." Two prior approaches had reduced this cost but were tied to specific implementations:

- **Directory Assistance Service** (RFC 1202, M. Rose, Performance Systems International, February 1991) — a gateway approach, but implementation-specific
- **DIXIE** (RFC 1249, Howes, Smith, Beecher, University of Michigan, August 1991) — also successful, but implementation-specific

RFC 1487 departs from these predecessors by consciously avoiding dependence on any particular implementation, defining a protocol that any LDAP client can speak to any LDAP server.

## Protocol Version Numbering

A historical curiosity: the BindRequest in RFC 1487 states "This document describes version 2 of the LDAP protocol," suggesting an earlier experimental version 1 preceded the public RFC. Despite specifying protocol version 2 internally, RFC 1487 is conventionally called "LDAPv1" — the first LDAP specification to receive an RFC number. [[reflect/sources/2026-05-02-rfc1777-txt|RFC 1777]] (March 1995) then became "LDAPv2."

## Protocol Model

RFC 1487 establishes the client-server model that all subsequent LDAP versions preserved:

- Clients transmit protocol requests; servers perform directory operations and return responses
- Operations are asynchronous — requests and responses are correlated by `messageID` and may be in-flight simultaneously
- **Servers handle referrals internally** — no provision for returning referrals to clients; servers locate the appropriate DSA and return only final results
- Minimizing client complexity is an explicit design goal, to facilitate widespread deployment

## Transport

LDAP runs directly over TCP with servers listening on **port 389**. `LDAPMessage` PDUs map directly onto the TCP bytestream, bypassing OSI session and presentation layers entirely. An alternative Connection Oriented Transport Service (COTS) mapping onto T-Data is also defined.

## Protocol Operations

All exchanges use the `LDAPMessage` envelope containing a `messageID` and a `protocolOp` choice. RFC 1487 defines nine operations:

| Operation | Purpose |
|---|---|
| **Bind** | Initiate session and authenticate (simple cleartext or Kerberos v4) |
| **Unbind** | Terminate session (no response defined) |
| **Search** | Query entries matching a filter with base/one/subtree scope |
| **Modify** | Add/delete/replace attribute values atomically |
| **Add** | Create a new entry |
| **Delete** | Remove a leaf entry |
| **ModifyRDN** | Change the last RDN component of an entry's name |
| **Compare** | Test whether an entry has a specific attribute value |
| **Abandon** | Request cancellation of an in-progress operation |

This nine-operation set is the same core set that [[reflect/sources/2026-05-02-rfc1777-txt|RFC 1777]] (LDAPv2) inherited essentially unchanged. LDAPv3 ([[reflect/sources/2026-05-02-rfc4511-txt|RFC 4511]]) added a tenth — Extended — as the open-ended extensibility mechanism.

### Bind and Authentication

The BindRequest supports three authentication choices:

- **`simple`** — cleartext password; also used for anonymous binds (zero-length name and password)
- **`krbv42LDAP`** — Kerberos version 4 ticket for the LDAP server (suggested service name: `ldapserver`)
- **`krbv42DSA`** — Kerberos version 4 ticket for the backing X.500 DSA (suggested service name: `x500dsa`)

Two Kerberos options exist because the LDAP server and its backing DSA may be distinct systems. The security section states plainly: "This version of the protocol provides facilities only for simple authentication using a cleartext password, and for kerberos version 4 authentication." No [[reflect/concepts/sasl|SASL]] framework, no [[reflect/concepts/ldap-tls|TLS]], no transport-level security of any kind — all LDAPv3 additions. The security trajectory from RFC 1487 through the 1997 gap to RFC 4513's resolution is traced in [[reflect/synthesis/ldap-authentication-security-architecture|LDAP Security Architecture]].

### Search Filters

The `Filter` type supports `and`, `or`, `not`, `equalityMatch`, `substrings`, `greaterOrEqual`, `lessOrEqual`, `present`, and `approxMatch`. This filter vocabulary is preserved verbatim through LDAPv2 and LDAPv3. The wire encoding uses ASN.1 BER; the human-readable [[reflect/concepts/ldap-search-filters|filter string syntax]] was defined in separate companion documents — RFC 1960 for LDAPv2, then [[reflect/sources/2026-05-02-rfc2254-txt|RFC 2254]] and later [[reflect/sources/2026-05-02-rfc4515-txt|RFC 4515]] for LDAPv3.

### Distinguished Names

[[reflect/concepts/distinguished-name|Distinguished Names]] are represented as `IA5String` values (ASCII only) encoded according to a companion specification (RFC 1485, Kille, July 1993). `LDAPDN` and `RelativeLDAPDN` are both defined as `IA5String` — reflecting the pre-Unicode assumption of the era. LDAPv3 later shifted to UTF-8 string representations via RFC 2253 (and its 2006 successor RFC 4514).

## BER Encoding

Protocol elements use ASN.1 Basic Encoding Rules (BER) with two restrictions to reduce decoder complexity:
1. Only definite-length encoding
2. Bitstrings and octet strings in primitive form only

## Result Codes

The `LDAPResult` type carries `resultCode`, `matchedDN`, and `errorMessage`. RFC 1487's 35 result codes (`success` through `other(80)`) are the same set as RFC 1777. LDAPv3 later added codes for referrals, SASL bind-in-progress, and other features absent from v1/v2.

## Editorial Note: ModifyRDN Naming Inconsistency

RFC 1487 contains an internal inconsistency: Section 4 defines the operation as `ModifyRDNRequest`/`ModifyRDNResponse`, while Appendix A (the complete ASN.1 definition) labels the same APPLICATION 12/13 tags as `modifyDNRequest`/`modifyDNResponse`. [[reflect/sources/2026-05-02-rfc1777-txt|RFC 1777]] resolved this inconsistency, standardizing on `ModifyRDNRequest`. Additionally, Appendix A's ASN.1 comments note "old RDN always deleted," documenting what LDAPv3's `ModifyDN` would later make configurable via a `deleteoldrdn` BOOLEAN field.
